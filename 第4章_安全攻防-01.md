# 第四章 | 安全攻防

> 安全是一场不对称战争——攻击者只需找到一个漏洞，防御者必须堵住所有。这一章从哈希到证书链、从 SQL 注入到 CSRF、从风控到零信任，覆盖最常见的攻击向量和防御纵深。

## Q46 · 哈希、加密、签名——三件事本质不同

### 你有没有思考过这么一个问题？

"哈希"能单向不可逆，"加密"能双向可解密，"签名"能证明"确实是我发的"。三件事新人常搞混。MD5、SHA-256（哈希）vs AES（加密）vs RSA 签名——分别解决什么问题？

### 底层发生了什么？

- **哈希（Hash）**：任意长度输入 → 固定长度输出。**单向不可逆。** 用途：密码存储（存 hash 不存明文）、文件完整性校验、Git commit ID。
- **加密（Encryption）**：明文 + 密钥 → 密文；密文 + 密钥 → 明文。**双向可解密。** 对称（AES）快，非对称（RSA）慢但解决密钥分发。
- **签名（Signature）**：用私钥对哈希值签名，公钥验证。**证明"确实是私钥持有者发的，内容未被篡改"。**

### 大白话解释

- 哈希 = 香肠机。猪肉进去，香肠出来。给你香肠——变不回猪肉（单向）。
- 加密 = 密码锁。你锁住箱子（加密），有密码的人打开（解密）。
- 签名 = 你的亲笔签名。别人可以验证这是你的笔迹（公钥验签），但只有你能写出这个笔迹（私钥签名）。

### 核心启示

> 哈希 ≠ 加密。哈希验证"数据没被改过"，加密保证"数据别人看不到"，签名证明"数据确实是我发的"。**三者解决三个完全不同的问题，但经常被同时使用。**

## Q47 · 设计一个密码存储系统——注册、登录、改密码

### 你有没有思考过这么一个问题？

注册时用户填了密码"123456"。你怎么存？**绝对不能存明文。** 但存哈希就够了吗？不够——因为"123456"的 SHA-256 值所有人都知道。

### 底层发生了什么？

**盐值（Salt）：** 每个用户分配一个随机盐值。存储的不是 `SHA256(password)`，而是 `SHA256(password + salt)`。即使两个用户密码相同，由于盐不同，存储的哈希完全不同。

**计算成本因子：** SHA-256 太快——GPU 每秒算数十亿次，暴力破解成本极低。**bcrypt/scrypt/Argon2——故意慢的哈希算法**——内置成本因子，让每次哈希计算消耗大量 CPU 和内存，暴力破解从"每秒十亿次"变成"每秒几次"。

**完整流程：** 注册 → 生成随机盐 → `hash = bcrypt(password, salt, cost=12)` → 存 `{salt, hash}`。登录 → 取出 salt → 同样算法计算 → 对比哈希。

### 大白话解释

你不存"原始钥匙"——只存钥匙的"模具凹痕"。用户说"这是我的钥匙"，你用模具压一下看凹痕是否吻合。盐 = 在钥匙上涂一层随机颜色——两把完全相同的钥匙涂不同颜色 → 凹痕完全不同。

### 核心启示

> 密码安全的三个原则：**不加盐的哈希 = 明文（彩虹表秒破）；用快哈希 = 送攻击者福利（GPU 暴力破解）；bcrypt/Argon2 故意慢 = 暴力破解从"秒级"变"年级"。**

## Q48 · HTTPS证书链——从根CA到你的浏览器，信任怎么传递？

### 你有没有思考过这么一个问题？

浏览器怎么知道 `baidu.com` 真的是百度的网站，不是黑客伪造的？不是"看长相"——是靠**证书链的逐级签名验证。**

### 底层发生了什么？

浏览器内置几十个**根 CA 证书**（公钥）。网站的证书由某个根 CA 签名 → 浏览器用根 CA 公钥验证签名 → 有效 → 信任。

证书链：**根 CA → 中间 CA → 网站证书。** 逐级签名，逐级验证。

**Certificate Transparency（CT）：** 所有新签发的证书必须记录在公开日志中。如果黑客攻破了某个 CA 偷偷给自己签了一个 `google.com` 的证书——CT 日志会被发现，浏览器拒绝。

### 大白话解释

HTTPS 证书链 = 身份证体系。你信公安局签发的身份证——不是因为你认识每个警察，而是因为公安局的公信力是全国默认信任的（根 CA）。中间 CA = 区公安局——由总局授权签发。

### 核心启示

> HTTPS 的安全不基于"加密算法不可破"——基于**根 CA 的公钥——是你选择信任的起点。** 信任链崩塌的唯一方式：根 CA 被攻破或被污染。**信任永远是一个锚点的问题——你信谁，谁就可以签名。**

## Q49 · SQL注入——为什么参数化查询是唯一正确解？

### 你有没有思考过这么一个问题？

`SELECT * FROM users WHERE name='$input'`——如果 input 是 `' OR '1'='1`，这条 SQL 变成了 `WHERE name='' OR '1'='1'`——返回所有用户。

**本质：用户输入被当成 SQL 代码执行了——输入和代码的边界被模糊了。**

### 底层发生了什么？

参数化查询（Prepared Statement）将 SQL 模板和数据分开传递：

```sql
-- 模板
SELECT * FROM users WHERE name=?
-- 数据: "admin' OR '1'='1"
```

数据库驱动在模板编译后只接受参数值填充——参数永远被当做纯数据（即使包含 `' OR '1'='1` 也只是 name 字段的值，不是 SQL 指令）。

ORM 底层也是参数化——但**拼接 SQL 字符串**（`"WHERE name='" + input + "'"`）绕过了保护。

### 大白话解释

你去图书馆借书。管理员问"你要借什么书？"你说"《三体》"——管理员查 `WHERE title='《三体》'`。但如果你说"《三体》' OR '1'='1"——查询变成返回所有书。参数化查询 = 管理员说"书名是______"——你填空——无论填什么都是书名。

### 核心启示

> SQL 注入的本质是**输入和代码的边界被模糊。** 参数化查询把边界重新划清——输入永远是数据，不是代码。**所有注入类攻击（SQL注入、XSS、命令注入）都遵循同一模式：输入逃逸了它的数据身份，变成了可执行代码。**

## Q50 · XSS攻击——反射型、存储型、DOM型

### 你有没有思考过这么一个问题？

你在留言板写了一句话。攻击者写了一个脚本标签。所有看了这条留言的人，浏览器执行了这个脚本——Cookie 被偷走了。

### 底层发生了什么？

**反射型 XSS：** 恶意脚本在 URL 参数中 → 服务端原样返回在页面 → 浏览器渲染执行。需要诱导用户点击精心构造的链接。

**存储型 XSS：** 恶意脚本存进数据库（留言板、用户昵称）→ 其他用户访问触发。危害最大——被动触发，影响所有访客。

**防御：** 输出编码——`<` 变成 `&lt;`。CSP（内容安全策略）——HTTP 头限制页面只能加载白名单域名脚本。HttpOnly Cookie——禁止 JS 访问 Cookie，即使 XSS 成功也偷不走。

### 大白话解释

你写"今天天气真好"——正常内容。攻击者写"<偷走你的密码>"——所有看到这条留言的人密码被偷。防御 = 保安把攻击者的话中的"<"和">"换成无害的显示字符。

### 核心启示

> XSS 是**"让别人的浏览器执行你的代码"。** 防御的第一原则：**永不相信用户输入。** 所有用户可控的数据在插入 HTML 前必须转义——这是输出端的事，不是输入端的事。

## Q51 · CSRF——你登录了网银，点了一个链接，钱被转走了

### 你有没有思考过这么一个问题？

你登录了网银（Cookie 还在浏览器里）。然后你点开了一个"萌猫图片"链接。这个链接里有个隐形表单——自动向网银发起转账请求。网银看到请求带着你的合法 Cookie——**以为是你本人在操作。** 钱被转走了。

### 底层发生了什么？

CSRF 利用：浏览器自动携带 Cookie 的特点。攻击者从第三方站点发请求到目标站点——目标站点收到请求时 Cookie 被自动带上——服务器以为是已登录用户的操作。

**防御：** CSRF Token——服务端生成随机 Token 嵌入表单和 Cookie，提交时对比。SameSite Cookie——`SameSite=Strict/Lax` → 跨站请求不携带 Cookie。Referer/Origin 头检查。

### 大白话解释

你坐在银行窗口前，身份已验证（登录 Cookie 在）。有人趁你不注意，从背后抓住你的手按了"转账"按钮。银行看到是你的手（Cookie）、听到你说"转账"（请求）——照做了。SameSite Cookie = 银行规定"转账必须你本人亲自来窗口说"——背后抓你的手的不算。

### 核心启示

> CSRF 的巧妙在于**攻击者不需要偷走你的凭证——他利用你已有的凭证替他做事。** SameSite=Lax 是当前最简洁有效的方案——**它告诉浏览器"跨站请求别带我的 Cookie"。**

## Q52 · DDoS攻击为什么"防不住"？

### 你有没有思考过这么一个问题？

你花 100 块租一小时僵尸网络，可以发动几 Gbps 的 DDoS 攻击。防御方需要花 100 万买清洗设备。**这种不对称的根本原因是：攻击的边际成本极低，防御的边际成本极高。**

### 底层发生了什么？

**三种攻击类型：** 流量洪水（UDP Flood）、协议攻击（SYN Flood——只发 SYN 不完成握手）、应用层攻击（HTTP Flood——大量"正常"请求消耗 CPU）。

**防御手段：** 流量清洗、Anycast 分散（攻击流量分到全球多节点）、CDN 吸收（静态内容由 CDN 扛）。但 Tbps 级巨型 DDoS——任何单点都扛不住——最终靠上游 ISP 在源头封堵。

### 大白话解释

你开了家小店，门面只能进 10 个人。攻击者雇了 10000 人什么也不买只是站在你门口——真正的客人挤不进来。这不是"技术漏洞"——你的门就这么大，物理极限。你雇保安赶人，后来又来了 20000 人。攻防成本完全不对称。

### 核心启示

> DDoS 的根本困境不是技术性的——是**经济性的。** 攻击者花 100 块，防御方花 100 万。**防御不是在技术层面赢——是在成本层面让对方觉得"不划算"。**

## Q53 · 设计一个风控系统——规则引擎和机器学习各自处理什么？

### 你有没有思考过这么一个问题？

你需要拦截：黄牛脚本抢票、羊毛党批量注册、刷单冲销量。有些攻击模式明确（同 IP 一秒 100 次），有些隐藏（正常购买量但行为轨迹异常）。

### 底层发生了什么？

**规则引擎：** 布尔表达式组合——"同 IP 一分钟 > 100 次 → 拦截"。微秒级，确定性高，但**规则靠人维护，跟不上新攻击手法。**

**ML 模型：** 多维特征打分——设备指纹、鼠标轨迹、页面停留时间、购买历史——分数超阈值拦截。优势：发现规则覆盖不到的异常模式。局限：可解释性差——"为什么拦这个？说不清。"

**工程实践：** 规则做前置快速过滤 → ML 做深度判别 → **人审做兜底（高价值决策人工复核）。**

### 大白话解释

规则引擎 = 门卫手里的清单"穿拖鞋不让进"。明确、快，但换双布鞋就绕过了。ML 模型 = 门卫的经验直觉——说不清原因，但看多了知道"这人眼神不对可能有鬼"。好的风控 = 规则处理已知威胁，模型发现未知威胁，两者配合。

### 核心启示

> 风控的永恒矛盾：**误杀（拦了真实用户）vs 漏杀（放了黄牛）。** 规则太严 → 误杀高 → 用户投诉。规则太松 → 漏杀高 → 黄牛泛滥。**不是在"更严"和"更松"之间选——是在"不同场景接受不同的误杀率"。** 支付：零容忍。领优惠券：容忍 5% 误杀。

## Q54 · OAuth2.0的四种授权模式——隐式模式为什么被淘汰了？

### 你有没有思考过这么一个问题？

你用某 App"用微信登录" —— App 把你跳到微信页面，确认授权后回到 App，你已登录。为什么微信不是直接把你的密码告诉 App？

### 底层发生了什么？

**授权码模式（最安全）：** 用户授权 → 授权服务器返回 Authorization Code → App 后端用 Code + ClientSecret 换 Access Token。**Code 即使被截获也没用——没有 ClientSecret 换不到 Token。**

**PKCE 扩展：** 用于没有 ClientSecret 的公共客户端（SPA）。Client 生成随机 code_verifier → SHA256 哈希 → 换 Token 时必须提供原始值 → 防止授权码被拦截。

**隐式模式（已淘汰）：** Token 直接在 URL 里返回——任何人看到 URL 就能截获 Token。

### 大白话解释

OAuth2.0 = 你去朋友的健身房，不用办卡——扫朋友分享的二维码，系统问"你是老王的客人吗？"确认 → 给你临时访客卡。健身房不告诉你老王的密码——只确认"老王说你可以进来"。

### 核心启示

> OAuth2.0 不解决"你是谁"——解决的是**"你允许谁以什么权限访问你的什么资源，而不暴露你的密码。"** 密码永远不出边界，权限可以随时收回。

## Q55 · 设计一个SSO单点登录——登录了淘宝，自动登录天猫

### 你有没有思考过这么一个问题？

不同域名——`taobao.com` 和 `tmall.com`——Cookie 不能跨域共享。你怎么实现"登录一个，其他自动登录"？

### 底层发生了什么？

**CAS 协议：** 用户访问 taobao.com → 没登录 → 重定向到认证中心 → 登录 → 认证中心返回 ticket → taobao.com 拿 ticket 去认证中心验证 → 确认身份 → 创建本地会话。

用户再访问 tmall.com → 没本地会话 → 重定向到认证中心 → **认证中心有用户的全局会话** → 不需要重新登录 → 直接发 ticket → tmall.com 验证 → 自动登录。

**OIDC（基于 OAuth2.0 的身份认证层）：** 比 CAS 更现代——基于 JWT ID Token，支持更多场景。

### 大白话解释

SSO = 你进了一个大型购物中心（认证中心）。在中心门口刷身份证——领到一张通行卡。进淘宝店——刷卡确认身份。进天猫店——同一张卡——不需要再掏一次身份证。

### 核心启示

> SSO 的核心是**"认证"和"应用"分离。** 每个应用不需要自己管理用户密码——认证中心统一管理。**认证是横切关注点——应该集中处理，而不是每个应用重复实现。**
