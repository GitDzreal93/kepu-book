2021年，产品经理给了份PRD："用户能搜索商品"。我花两天实现了——一个搜索框、输入关键词、返回匹配商品列表。Demo演示时，产品经理说："不对，搜索结果要按相关度排序、要支持按价格筛选、要自动纠错（搜'苹锅'能出'苹果'）、要显示搜索历史、要热门搜索推荐……"

我说你PRD里没写啊。他说"搜索功能不都这样吗？我以为你知道。"

**需求文档写不清楚的根源：写的人脑子里有大量"不言自明"的假设，而读的人脑子里有不同的假设。两份假设碰撞的地方，就是Bug的源头。**

## 核心结论

1. **隐性知识是需求偏差的根源**："我们知道的远比我们能说出来的多"
2. **敏捷的答案是"用频繁交付暴露假设"**，比一次性穷举所有场景成本更低
3. **但敏捷不是"不需要文档"**：接口契约、设计决策、状态机必须记录
4. **好的需求 = 用户故事 + 验收标准 + 交互原型**，不是长篇大论的"需求规格说明书"

## 深度拆解

### 为什么"写清楚"如此之难？

Michael Polanyi在1958年提出了"隐性知识"概念，我们知道的远比我们能说出来的多。你骑自行车时知道怎么保持平衡，但让你把保持平衡的物理公式写出来，写不出来。你知道，但你说不出来。

产品经理用了十年淘宝，他对"搜索商品"的理解包含了十年的使用经验：自动纠错、相关度排序、筛选面板、历史记录、热门推荐，这些东西他"觉得"是搜索功能的标配，所以没写。而一个刚从Java转过来的开发，搜索就是 `SELECT * FROM products WHERE name LIKE '%keyword%'`。

这个认知差距不是态度问题，而是知识结构差异。产品经理无法"意识到自己知道但没说出来的东西"，因为那就是他的常识。

### 敏捷的答案：用交付暴露假设

传统软件开发的做法是：写一份尽可能详细的需求文档 → 评审 → 冻结 → 按文档开发。问题是：
- 文档永远不可能穷举所有场景（上面搜索的例子）
- 评审时发现的问题只是冰山一角
- 冻结后业务变化了，文档反而成了障碍

敏捷的做法：MVP（最小可行产品）→ 上线 → 看到用户真实行为 → 发现缺失的场景 → 迭代。你花三天实现一个基础搜索，上线后看到用户搜"苹锅"搜不到结果→加模糊匹配。看到用户反复搜同一个词→加热门搜索。这些场景不是产品经理在PRD里漏掉的，是用户行为暴露出来的。

**快速交付的价值不在于"做得快"，而在于"暴露虚假假设"。**

### 但敏捷不是"不需要文档"

有些东西必须文档化，不能靠迭代暴露：

- **接口契约**：API的请求/响应格式、错误码、幂等性保证。调用方不能"等你迭代完再改"。
- **状态机**：订单的状态流转（待支付 → 已支付 → 已发货 → 已完成）。不能上线后才发现"已取消的订单不应该能变成已发货"，这是逻辑漏洞。
- **设计决策**：为什么选了方案A没选方案B。三个月后新的同事接手，你告诉他"当时ABC三个方案我们选了A"但没告诉他选A的原因→他可能觉得B更好，推倒重来。

## 实战要点

### 好需求的三个要素

1. **用户故事**：作为[角色]，我想要[功能]，以便[价值]。"作为买家，我想要按价格筛选搜索结果，以便快速找到预算内的商品"。包含谁、做什么、为什么，比"搜索支持筛选"多了"为什么"。
2. **验收标准**：Given/When/Then格式。"Given 搜索'手机'的结果有100条，When 用户选择价格1000-2000元筛选，Then 只显示价格在范围内的手机"。这给了开发和测试明确的"什么叫完成"。
3. **交互原型**：一张线框图或Figma原型比1000字的描述更直观。"搜索框在页面顶部、筛选在左侧、结果在中间"，文字描述的歧义远大于图片。

### 臻叔踩坑笔记

1. **"需求文档=法律合同"心态**：产品经理写需求像写合同——生怕漏掉什么被开发钻空子；开发看需求像看合同——"你写什么我做什么，你没写我不做"。正确的态度：需求文档是沟通工具，不是法律文件。
2. **需求和实现混在一起**："用户点击搜索→前端调用/search接口→后端查ES→返回JSON"，这是实现方案不是需求。需求只描述"用户想要什么"，不描述"怎么实现"。
3. **"敏捷=不写文档"**：敏捷的原意是"可工作的软件高于详尽的文档"，不是"高于"是"zero"。接口契约、状态机、关键决策必须文档化。
4. **需求变更的恐惧**：开发怕需求变，产品怕需求不变。好的实践：用Feature Flag控制需求的上线节奏而不是阻止变更。

### 一句话总结

> 需求文档写不清楚不是谁的错，是人类沟通的根本困境。你能做的是：尽快交付一个能跑的东西，让真实的交互暴露你和你产品经理各自脑中的假设有哪些不一样。
