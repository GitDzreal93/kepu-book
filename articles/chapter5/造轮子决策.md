# 大厂为什么要自己造轮子？——用开源是省钱，自研是买命

![配图](../../images-new/chapter5/build-vs-buy.png)


2021年，某互联网公司的技术选型会上吵了三个小时。讨论的命题是："消息队列用RabbitMQ还是RocketMQ？"

一方说："RocketMQ是阿里开源的，双十一验证过，别再造了。"另一方说："日均消息量已经到50亿了，RabbitMQ集群模式下跨机房的延迟受不了。"最后CTO拍板："自研。基于Raft协议写一个轻量级MQ，只做需要的特性，不做通用。"

三个月后这个自研MQ上线了。代码量只有RocketMQ的十分之一，但在该团队的特定场景下跑得比RocketMQ还快——因为不需要事务消息、不需要定时消息、不需要消息轨迹，只要"可靠投递+高吞吐"。这些东西砍掉后，实现极简，性能反而更好。

**"造轮子"的本质命题不是"你做不做得到"——是"你做这件事的收益是否大于不做的代价"。**

## 核心结论

1. **自研的合理理由只有三个**：差异化竞争优势、规模突破现有方案极限、不可妥协的合规要求
2. **用开源是默认选择**：社区维护、文档齐全、踩坑经验丰富——除非你有一个无法拒绝的自研理由
3. **大厂开源自己的轮子是另一套逻辑**：降低维护成本（外部贡献者）、建立生态标准、吸引人才
4. **自研的真正成本不是开发成本——是长期的维护成本**

## 深度拆解

### 什么时候不该自研？

一个简单的决策框架：**如果你的需求跟大多数公司的需求一样，就不要自研。**

- 你需要一个Web框架 → 用Spring/Express/Django。每年有几百万人用，你遇到的Bug别人早踩过了。
- 你需要一个数据库 → 用MySQL/PostgreSQL。它们被验证了几十年，比你三个月写出来的可靠一万倍。
- 你需要一个缓存 → 用Redis。单线程为什么比你多线程快？因为它的作者花了十年优化数据结构，你的"优化"大概率是退化。

原则是：**非核心业务 + 成熟开源方案 = 用开源。** 除非你做的事情跟所有人都不一样——规模不一样、场景不一样、要求不一样。

### 什么时候该自研？

**1. 差异化竞争**

微信的消息同步算法、抖音的推荐引擎、谷歌的搜索引擎——这些是公司的核心竞争力。用开源做这些就等于告诉竞争对手"我们跟你们用的是一样的东西，区别只是品牌"。核心竞争力必须自研——不是为了"效率"，是为了"控制权"。你能今天优化A、明天改B，不需要等开源社区release。

**2. 规模突破极限**

12306的库存扣减——任何现成的分布式事务方案在那个并发量下都扛不住。淘宝双十一的流量调度——没有现成的开源系统能处理那个规模的流量。当你的规模超过了所有现有方案的极限，自研不是选择——是唯一的出路。

**3. 合规与安全**

银行核心系统不能用开源数据库？不是技术原因——是监管要求代码必须自主可控，出了问题能自己追责。军工、政府、金融行业的某些场景——用开源本身就是合规风险。

### 大厂为什么要开源自己的轮子？

Google开源了Kubernetes、Facebook开源了React、阿里开源了RocketMQ。这些是他们的轮子——为什么给别人用？

- **降低维护成本**：你一个人维护一个项目，Bug修得慢。全球一千个公司用你的项目、贡献补丁——维护成本大幅下降。
- **建立生态标准**：Kubernetes成为容器编排的事实标准后，所有云厂商都去兼容K8s API——Google的GKE天然是"正宗K8s"，竞争壁垒从此建立。
- **吸引人才**：最好的React开发者可能最终加入Facebook，因为他已经对React源码烂熟。

他们开源不是因为"无私"，是因为开源是更聪明的利己。

## 实战要点

### 自研的成本公式

```
自研总成本 = 开发成本 + 维护成本（每年） × 系统存活年数 + 学习成本 × 团队人数 + 迁移成本
```

很多团队只考虑了"开发成本"（三个月写完），但写完之后：
- 每个新入职的人都要学你这个轮子（没有教程、没有社区、只有内部Wiki）
- 每次主版本升级你都要自己处理兼容性（没有社区帮你踩坑）
- 你花在维护这个轮子上的时间，本来可以用来做业务

如果你的团队只有10个人——维护5个自研基础组件的代价是：每个人花30%的时间在"非业务"的事情上。

### 臻叔踩坑笔记

1. **"开源的那个不够好"**：不够好≠必须自研。可以给开源项目提PR、fork一份自己维护。完全从零写——你连"不够好"的标准都达不到。
2. **低估隐藏成本**：只算了开发成本，没算Bug修复、文档编写、新人培训、版本升级、安全补丁。
3. **"这个很简单，两天写完"**：原型两天能写。生产级别——考虑边界条件、错误处理、监控、灰度、回滚、文档——是原型的20倍工作量。
4. **人才流失风险**：自研轮子的核心开发者离职→接手的人需要花三个月理解→期间轮子出了Bug没人能修。

### 一句话总结

> 用开源是"花别人的时间解决自己的问题"；自研是"花自己的时间解决只有自己才有的问题"。前者是默认策略，后者需要有无法拒绝的理由。
