2020年，我们有一张用户行为日志表，每个月新增5亿行，一年就60亿行。起初在单库MySQL上跑得挺好，到了20亿行后，索引大到内存放不下，`COUNT(*)`跑一次要40秒，`SELECT ... ORDER BY create_time LIMIT 100`直接在磁盘上扫描百万行。

更头疼的是，业务代码里到处都有JOIN、子查询、GROUP BY。如果直接拆表，这些SQL全部要重写。当时我花了三周时间，把分库分表方案从"怎么拆"到"怎么迁移"到"怎么让业务代码几乎无感"完整走了一遍。

这篇文章就是那次分库分表经验的系统化复盘，关注点不在"分库分表是什么"，而在"选择分片键的权衡"和"分表后那些SQL怎么改写"。

## 核心结论

1. **垂直拆分解决"表太宽"**——把一张大表按业务模块或冷热程度拆成多张小表，每张表列更少、访问更集中。适合字段多且冷热不均的场景。
2. **水平拆分解决"行太多"**——把一张大表按行拆分到多个库或表中。核心在分片键的选择，它决定了80%的查询能否定位到单一分片。
3. **分片键的选择没有银弹**——按用户ID分片对"查用户订单"极高效，但"按手机号查用户"需要全分片扫描。你只能让大部分查询走单分片，剩下的用冗余或异步补偿。
4. **拆表后的SQL改写**是最大的工程量——禁止跨分片JOIN、全局唯一ID改造、聚合查询（COUNT/SUM/AVG）变成应用层聚合、分页从`LIMIT offset, limit`变成更复杂的全局游标分页。
5. **数据迁移是分库分表最容易被低估的难题**——不是"建好新表，导数据，切流量"那么简单。在线业务的迁移必须双写（同时写新旧表）+ 数据校验 + 灰度切读 + 最终清理。

## 深度拆解

### 一、垂直拆分：列太多怎么拆

```
拆分前（user表，50个字段）：
id | name | phone | email | avatar | bio | addr_province | addr_city | 
addr_detail | login_count | last_login_ip | last_login_time | 
created_at | updated_at | settings_json | ...

拆分后：
user_basic（高频查，15个字段）：
id | name | phone | email | avatar | created_at | updated_at

user_profile（低频查，20个字段）：
user_id | bio | addr_province | addr_city | addr_detail | settings_json

user_security（敏感字段，5个字段）：
user_id | password_hash | salt | last_login_ip | last_login_time
```

**拆分原则：**

1. **按访问频率**：高频字段（昵称、头像）放主表，低频字段（签名、地址）放扩展表。
2. **按业务边界**：认证信息（密码、salt）独立成表，访问权限更严格。
3. **按字段大小**：大字段（TEXT、BLOB）独立出去，避免拖慢主表的行扫描。

**代价**：原来一条`SELECT * FROM user WHERE id = 1`变成两条（或JOIN）。但对大多数查询来说，需要的只是主表字段，扩展表只有在"查看个人主页"等少数场景才需要。总体来说利大于弊。

### 二、水平拆分：分片键是核心

水平拆分决定了一切——同一个分片键的查询能精确定位到单个分片，不同分片键的查询要在所有分片执行后汇总。

**常见分片策略：**

| 策略 | 适用场景 | 优势 | 劣势 |
|------|----------|------|------|
| 哈希取模（`hash(user_id) % 16`） | 用户表、订单表 | 数据均匀分布 | 扩缩容需要重新分布全部数据 |
| 一致性哈希 | 缓存、需要弹性扩缩容 | 增删节点只影响少部分数据 | 实现稍复杂 |
| 按时间范围（`2024-01`, `2024-02`...） | 日志、时序数据 | 自然按时间归档、范围查询高效 | 热点不均匀（最新分片压力大） |
| 按地域（`cn`, `us`, `eu`） | 有明显地域属性的业务 | 数据就近访问、合规友好 | 地域间数据量不均 |
| 号段映射（`user_id 1-1000万→分片1`） | 已有自增ID、需要确定性路由 | 无需哈希计算 | 分片数据量不均 |

**选择分片键的四条原则：**

1. **让最多查询能带上分片键**。如果你90%的查询都是`WHERE user_id = ?`，那就按user_id分片。
2. **数据分布均匀**。避免热点分片（如按时间分片时最新分片被打爆）。
3. **避免跨分片事务**。分布式事务是灾难，尽量让一个业务操作的所有数据在同一个分片。
4. **考虑未来扩容**。你需要预估2-3年后的数据量，避免频繁扩缩。

### 三、拆表后SQL怎么改写

**禁止跨分片JOIN：**

```sql
-- 原来：查用户的订单
SELECT u.name, o.amount 
FROM user u 
JOIN orders o ON u.id = o.user_id 
WHERE u.id = 1;

-- 拆表后：分两步
-- 1. 先查用户
SELECT name FROM user WHERE id = 1;  -- 路由到用户分片
-- 2. 再查订单
SELECT amount FROM orders WHERE user_id = 1;  -- 路由到订单分片
```

**关键风险点：**

- **双写一致性**：旧表写入成功、新表写入失败时，需要补偿。通常做法是旧表写入成功才写新表，新表失败记录到重试队列，异步补偿。
- **切读过程中读到不一致数据**：灰度期间部分用户读新表、部分读旧表。如果旧表有数据但新表还没同步过去——读新表的用户看不到。这是暂时的，等双写追平即可。
- **切写的时间点**：在100%读新表+双写稳定运行一段时间后（如一周），再切全量写新表。

## 实战要点

**臻叔踩坑笔记：**

1. **不要一上来就拆64个分片**。"预留未来增长"听起来合理，但64个分片意味着64个数据库连接、64次聚合查询、64份运维成本。2-3年内的数据量不需要这么多分片。从4或8个开始，真需要扩的时候再扩。

2. **全局唯一ID的生成不要依赖数据库自增**。拆表后每个分片的自增ID会重复。提前规划全局ID方案（Snowflake/号段/UUID），不要在拆完才发现"订单ID重复了"。

3. **分片键的选择要考虑"最坏情况查询"**，而不只是"最常见查询"。常见查询只占80%，但最坏情况查询（如运营后台按手机号搜用户）如果需要全分片扫描，用户体验极差。解决方案：对非分片键查询建立反向索引表（`phone_index: phone → user_id`），先在索引表查出user_id，再路由到分片。

4. **分布式事务不是必须的，但"最终一致性"方案要提前设计**。跨分片的转账操作，如果用2PC，性能会很差。大部分场景可以用"本地事务 + 消息队列 + 补偿"实现最终一致性——但需要在拆表前就想清楚哪些操作涉及跨分片，提前设计补偿逻辑。

5. **分库分表后，MySQL的一些功能不能用了**。LOAD DATA INFILE（跨分片）、存储过程（分片间无法通信）、触发器、外键——这些东西在分库分表环境下要么不可用，要么需要大量改造。迁移前盘点一下哪些功能在用，提前找替代方案。

**一句话总结：**

> 分库分表不是"把一个表切成几个表"的技术操作，而是"从单机思维到分布式思维"的范式转换。原来一条SQL能搞定的，现在变成"分片路由 → 各分片独立执行 → 应用层聚合"的三步曲。分片键的选择决定了80%的查询复杂度，而剩下的20%非分片键查询，才是真正考验方案成熟度的地方。
