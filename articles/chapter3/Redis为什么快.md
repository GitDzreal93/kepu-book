# Redis为什么快——不是因为"纯内存"那么简单

![配图](../../images-new/chapter3/redis-fast.png)


一次技术分享会上，有人问我"Redis为什么快"，我随口答了句"因为数据在内存里"。旁边一个老工程师笑了："Memcached也在内存里，Node.js单线程也在内存里——你觉得它们和Redis一样快吗？"

这个问题让我沉默了足足五秒。回来后我重新翻了Redis的源码和设计笔记，才发现"内存"只是表面原因。Redis快的原因是一套精心设计的**减法哲学**：不是靠"加更多线程、加更多缓存、加更复杂的算法"，而是靠"砍掉一切不必要的开销"。

这篇文章就把Redis快在哪些层面、每一层贡献了多少、以及Redis 6.0/7.0为什么还是加入了多线程，完整拆解一遍。

## 核心结论

1. **内存存储贡献了基础速度**（10-100ns vs 磁盘1-10ms），但这只是门槛——Memcached也在这个起跑线上，但它没有Redis快。
2. **单线程事件循环消除了锁竞争和上下文切换**——Redis每秒处理10万次请求，零次线程切换、零次锁等待。但前提是每个命令执行时间极短（微秒级），慢命令会阻塞所有请求。
3. **epoll IO多路复用**让单线程同时监听数千个连接——有数据的连接才处理，空闲连接不消耗CPU。这和Node.js的Event Loop是同一思路。
4. **手工优化的数据结构**（SDS、ziplist、intset、skiplist）针对每种数据模式的读写模式做了极致优化——不是用通用的list/map，而是根据实际数据分布选最优结构。
5. **Redis 6.0/7.0加入多线程不是推翻单线程模型**——多线程只负责网络IO的读写（收发数据包），命令执行仍然是单线程。核心单线程模型没有改变。

## 深度拆解

### 第一层：内存——1000倍的起步优势

| 操作 | 内存（Redis） | 磁盘（MySQL） | 差距 |
|------|-------------|-------------|------|
| 随机读 | ~100ns | ~10ms（机械盘寻道） | 100,000倍 |
| 顺序读 | ~100ns | ~0.1ms（顺序读） | 1,000倍 |
| 随机写 | ~100ns | ~10ms | 100,000倍 |

内存和磁盘的差距是1000到10万倍。但这里有个关键点：**第一层优势是相对于MySQL的，不是相对于Memcached的**。Memcached也在内存里，Redis相对Memcached的优势在后面的层次。

### 第二层：单线程——反直觉的"少就是多"

Redis 6.0之前，**整个命令处理流程在一个线程里完成**：读取客户端请求 → 解析命令 → 执行命令 → 返回结果。

**为什么单线程反而更快？**

多线程程序的开销：
1. **上下文切换**：CPU从一个线程切到另一个，保存/恢复寄存器和栈（约1-2微秒/次）。每秒10万次请求如果涉及大量线程切换，开销不可忽视。
2. **锁竞争**：多线程同时读写同一个数据结构 → 需要加锁。锁的CAS操作、等待队列、内核态切换，都会吃掉性能。
3. **伪共享（False Sharing）**：两个线程修改不同变量但它们在同一个CPU缓存行（64字节）中 → CPU缓存一致性协议强制同步缓存行 → 性能骤降。

Redis选择单线程，一次性避免了所有这些问题。每个命令之间没有任何干扰，执行的原子性由单线程天然保证。

**但单线程的前提条件：**

每个命令必须极快。`GET`（O(1)）、`SET`（O(1)）、`LPUSH`（O(1)）这些是微秒级的。但如果有人执行了`KEYS *`（遍历所有key，O(N)），单线程会阻塞所有其他请求——这就是Redis强调"避免慢命令"的原因。

### 第三层：IO多路复用——一个线程监听了3000个连接

Redis使用epoll（Linux）/ kqueue（macOS）/ IOCP（Windows）做IO多路复用：

关键点：
- 3000个客户端连接，但只有100个在活跃发送请求 → epoll只处理这100个，其他2900个不消耗CPU。
- `epoll_wait`在没有事件时阻塞（不占CPU），有事件时立即返回。
- 和传统的"一个连接一个线程"模型（如Tomcat BIO）相比，省了2900个线程的栈空间、上下文切换和调度开销。

**epoll、select、poll的对比：**

| | select | poll | epoll |
|---|---|---|---|
| 数据结构 | 固定大小位图(fd_set) | 动态数组 | 红黑树+就绪链表 |
| 最大fd数 | 1024（FD_SETSIZE） | 无限制 | 无限制 |
| 事件通知 | 每次遍历全部fd | 每次遍历全部fd | 只返回就绪fd |
| 时间复杂度 | O(n) | O(n) | O(1) |
| 内核态→用户态拷贝 | 拷贝整个fd_set | 拷贝整个pollfd数组 | 只拷贝就绪事件 |

### 第四层：手工优化的数据结构

Redis不是简单地用C的`list`、`map`，而是为每种数据模式手工打造了最优结构。

**SDS（Simple Dynamic String）：**

C语言的`char*`存储字符串有两个致命问题：获取长度需要O(n)遍历、二进制不安全（遇到`\0`截断）。Redis的SDS额外存储了长度和剩余空间：

```c
struct sdshdr {
    int len;    // 已使用长度（O(1)获取）
    int free;   // 剩余可用空间
    char buf[]; // 实际数据（二进制安全）
};
```

`STRLEN`命令O(1)、追加字符串不需要每次realloc（预分配空间）、支持二进制数据。

**ziplist（压缩列表）：**

当哈希或列表的元素很少且每个元素很短时，Redis用ziplist代替hashtable或linkedlist：

```
ziplist内存布局：
[zlbytes][zltail][zllen][entry1][entry2][entry3]...[zlend]

每个entry：[prevlen][encoding][data]
```

ziplist把多个元素紧凑地存在一块连续内存中——没有指针开销、没有内存碎片、CPU缓存友好。缺点是插入删除需要移动数据（对于小量数据可以接受）。

**intset（整数集合）：**

当集合中全是整数且数量不多时（<512个），用有序整数数组而非hashtable：

```
[encoding][length][int1][int2][int3]...
```

查找用二分（O(log n)），对几百个元素非常快，且内存效率极高。

**skiplist（跳表）：**

有序集合（ZSet）的核心数据结构。跳表的查询、插入、删除平均O(log n)，且实现比红黑树简单得多：

跳表支持范围查询（`ZRANGEBYSCORE`），这是红黑树不擅长的。

### 五、Redis 6.0/7.0的多线程：不是推翻，是补充

Redis 6.0引入了多线程IO：

为什么一定是**网络IO用多线程、命令执行保持单线程**？

因为瓶颈分析发现：在千万QPS级别，网络报文的读取和发送（`read()`/`write()`系统调用）占了可观时间，而命令执行的CPU时间反而占比不大。把IO并行化能获得整体吞吐提升，同时保持命令执行的简单性和原子性。

默认情况下多线程IO是关闭的——需要在配置文件中设`io-threads 4`才能启用。大多数场景下单线程已经足够快，不需要开多线程IO。

## 实战要点

**臻叔踩坑笔记：**

1. **不要用KEYS命令**。`KEYS *`会遍历所有key，如果key数量百万级，它会阻塞整个Redis数秒。取而代之用`SCAN`（游标迭代，不会阻塞）。生产环境建议直接禁用`KEYS`命令（`rename-command KEYS ""`）。

2. **不要在一个key里存巨大的list/hash/set**。一个大key的删除（`DEL bigkey`）是O(N)操作，会阻塞主线程。Redis 4.0引入了`UNLINK`（异步删除，后台线程慢慢删），但最好的办法是不产生大key——定期检查`redis-cli --bigkeys`。

3. **Lua脚本要保持短小**。Lua脚本执行期间Redis是阻塞的（整个脚本原子执行）。如果脚本里遍历一个大集合——对不起，所有其他请求排队等着。单个Lua脚本执行时间控制在毫秒级。

4. **AOF和RDB的持久化配置要合理**。RDB的`bgsave`会fork子进程，fork时主线程短暂阻塞。如果Redis内存100GB，fork可能需要几百毫秒到1秒。AOF的`everysec`模式（每秒刷盘）是生产环境推荐配置。`always`模式（每次写入刷盘）性能极差，不适合高吞吐场景。

5. **不要迷信"单线程所以CPU没用"**。虽然命令执行是单线程的，但持久化（RDB/AOF）、集群通信、异步删除、以及6.0+的多线程IO——这些都会用到多核。在高并发场景，Redis的CPU使用率可以到200-300%（多核）。一个8核机器跑单实例是浪费，可以用多个Redis实例分摊。

**一句话总结：**

> Redis的快不是单一技术的胜利，而是"内存定基底、单线程去竞争、epoll管连接、手工数据结构做精细优化、多线程IO补充网络瓶颈"的五层叠加——每一层单独看都不稀奇，但组合在一起就是每秒10万QPS的工程奇迹。
