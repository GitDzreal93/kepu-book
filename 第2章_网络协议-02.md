## Q23 · NAT——家里所有设备共用一个公网IP，外部回包怎么精准找到你？

### 你有没有思考过这么一个问题？

你家 Wi-Fi 下连了手机、电脑、平板、智能电视——十几台设备。但你的宽带只有**一个公网 IP**。百度服务器回包时，怎么知道这个包该发给你的手机而不是你的电脑？

### 底层发生了什么？

NAT 维护一张**映射表**：

```
内网IP:端口       →   公网IP:端口
192.168.1.2:54321  →  123.456.789.0:40001
192.168.1.3:54321  →  123.456.789.0:40002
```

你的手机（192.168.1.2）访问百度时，NAT 路由器把源地址替换为公网 IP + 新端口号 40001，记录映射。百度回包时发给 40001，NAT 查表→转发给 192.168.1.2。

**NAT 的几个变体：**

- **Cone NAT**：映射固定。任何外网 IP 给这个端口发包都会转发给你。P2P 穿透相对容易。
- **Symmetric NAT**：不同的外网目标地址映射不同端口。更安全但 P2P 穿透极难。**STUN/TURN/ICE 协议就是为穿透 NAT 设计的。**

P2P 应用（视频通话、BT 下载）需要双方直连，而不是经过服务器中转。STUN 让双方知道各自 NAT 映射的公网地址，TURN 是中继服务器（兜底——NAT 太严打不通时走中继）。

### 大白话解释

NAT = 公寓门卫。一栋公寓只有一个对外门牌号（公网 IP），但住了 100 户。快递员送到门牌号，门卫根据登记表（映射表）转交给正确住户。Cone NAT = 固定分配房间。Symmetric NAT = 每来个新快递公司重新分配——更安全但朋友找不到你。

### 核心启示

> NAT 是 IPv4 地址短缺的"创可贴"——它让几十亿设备共用几亿公网 IP，但打破了互联网"端到端"的原始设计。**P2P 的复杂性——STUN/TURN/ICE——本质上都在弥补 NAT 造成的"找不到你"问题。**

## Q24 · 设计一个CDN——你的视频需要全球毫秒级加载

### 你有没有思考过这么一个问题？

你在北京看一个美国 YouTube 视频。视频文件存在美国的服务器上。如果每次直接从美国拉，延迟几百毫秒，4K 视频要缓冲半天。但你实际体验是秒开。**因为你实际上看的是离你最近的 CDN 节点上的副本。**

### 底层发生了什么？

CDN 三层架构：

- **边缘节点（Edge）**：离用户最近，几毫秒可达。缓存最热门内容。
- **区域节点（Regional PoP）**：中间层，缓存次热门内容。边缘没命中回这里拉。
- **源站（Origin）**：全量数据存放处。只有区域也没命中时才回源。

**DNS 智能调度是核心。** 用户请求 `video.example.com` → DNS 解析 → 不是返回一个固定 IP，而是**根据用户 IP 的地理位置和实时负载，返回最近的边缘节点 IP。**

缓存策略：热门视频在边缘长期缓存（LRU），冷门视频回源拉取。**预热机制**：新视频发布前，CDN 提前把内容推到各边缘节点——等用户来访问时直接从边缘拿。

### 大白话解释

CDN = 全国连锁图书馆分馆。每本畅销书（热门视频）在离你最近的街角分馆都有复印本。你要借——直接去街角拿。你要借一本冷门绝版书——分馆没有，分馆帮你从总馆调。

### 核心启示

> CDN 的本质不是"让网络变快"——是**用存储空间换网络延迟。** 把数据提前放到离用户最近的地方，把"远距离传输"变成"本地读取"。**所有缓存系统都在重复这个模式：空间换时间。**

## Q25 · 负载均衡——轮询、最少连接、一致性哈希，什么时候用什么？

### 你有没有思考过这么一个问题？

你有 3 台后端服务器。1000 个用户请求来了，怎么分给这 3 台？"轮着来"听起来公平——但如果有些请求处理 10ms、有些要 10 秒呢？

### 底层发生了什么？

- **轮询（Round Robin）**：依次分发。简单但对"胖瘦"请求不均——A 分到三个轻请求（30ms），B 分到一个重请求（10s），B 忙死 A 闲着。
- **最少连接（Least Connections）**：分给当前活跃连接数最少的后端。对长连接场景更均衡。但连接数 ≠ 负载（一个 WebSocket 连接可能只是挂着）。
- **IP 哈希 / 一致性哈希**：对客户端 IP 或 Session ID 做哈希。**同一客户端始终路由到同一后端**——解决 Sticky Session 问题，但后端扩缩时哈希环重分配。

四层负载均衡（L4，只看 IP + 端口）vs 七层（L7，可以看 HTTP Header/Cookie 做路由）。L7 更灵活——比如把 `/api/payment` 路由到高性能服务器组，把 `/static` 路由到普通组——但性能开销更高。

### 大白话解释

轮询 = 轮流喊号。最少连接 = VIP 客户优先分到不忙的窗口。IP 哈希 = 按身份证尾号固定分到某窗口——让同一客户永远走同一窗口（需要 Session 时好用）。

### 核心启示

> 没有一个负载均衡算法是"普适最优"的。轮询简单但无视请求差异，最少连接接近公平但长连接下不准，哈希固定映射但扩缩时数据重分布。**选型 = 你的请求模式是什么 + 你能不能容忍 Session 漂移。**

## Q26 · 一条微信消息从北京到纽约——延迟主要花在哪？

### 你有没有思考过这么一个问题？

光速是 30 万公里/秒，北京到纽约约 1.5 万公里——光飞行时间应该只有 **50 毫秒**。但你的微信消息端到端延迟通常是 150-300 毫秒。另外 100-250 毫秒去哪了？

### 底层发生了什么？

端到端延迟 = **传播延迟 + 传输延迟 + 处理延迟 + 排队延迟。**

- **传播延迟**：光在光纤中跑的时间。光纤折射率约 1.5，光实际速度 ≈ 20 万公里/秒。北京到纽约 ≈ 75ms（不是 50ms）。
- **传输延迟**：数据从网卡"推"到链路上的时间。带宽越高越快。100KB 消息在 10Mbps 链路上需要 80ms，在 1Gbps 链路上只需 0.8ms。
- **处理延迟**：每个路由器检查包头、查路由表、转发。一跳通常 < 1ms，但经过 15-20 跳就累计了。
- **排队延迟**：路由器队列里有其他包在等。高峰期排队延迟是**最大的变量**——可能从 0 飙到 100ms+。

微信消息额外延迟：应用层处理（序列化、加密、写数据库）通常 10-50ms。

### 大白话解释

你寄一封信从北京到纽约。光飞行（75ms）= 飞机在天上的时间。但送信员骑车去机场（传输延迟）+ 安检排队（排队延迟）+ 每个中转站分拣（处理延迟）——加起来才是总时间。**飞机不是最慢的，地面折腾才是。**

### 核心启示

> 网络延迟的大头往往不是"距离远"——而是**"中间环节多"。** 跨国链路的跳数多、海底光缆带宽有限导致的排队、最后一公里的 Wi-Fi 干扰——是延迟的三大元凶。**减少延迟 = 减少中间环节，而不是"让光跑更快"。**

## Q27 · IPv4 vs IPv6——为什么喊了二十年还没全面切换？

### 你有没有思考过这么一个问题？

IPv4 地址早在 2011 年就分配完了。IPv6 地址多到能给地球上每一粒沙子分配一个。**为什么二十多年了还没全面切换？**

### 底层发生了什么？

IPv6 不兼容 IPv4——完全不同的地址格式和包头结构。迁移需要：

- **双栈**：设备同时跑 IPv4 和 IPv6。过渡方案，管理复杂。
- **隧道（6to4/Teredo）**：IPv6 包封装在 IPv4 中传输。额外开销，MTU 问题。
- **NAT64 + DNS64**：IPv6-only 设备通过翻译网关访问 IPv4 资源。

**核心阻力：IPv4 + NAT 已经勉强够用（NAT44/CGNAT），迁移的边际收益对很多企业不明确。** 移动网络和物联网是 IPv6 增长的主要驱动力——手机运营商需要海量地址。

### 大白话解释

全球交通靠右行驶（IPv4），某天宣布改靠左（IPv6）。新路靠左，老路靠右。你开靠左的车上了老路——不兼容——只能换车或修互通立交桥（双栈/隧道/NAT64）。所有车和路标都要改，工程量是"所有联网设备 + 所有网络中间设备"。

### 核心启示

> 技术升级的最大阻力往往不是"技术不好"——是**"旧系统还在'勉强能用'，迁移成本看不到回报"。** IPv4 + NAT 凑合了二十年，IPv6 的价值一直在增长但回报曲线还没跨过成本线。**所有"下一代"技术的命运都遵循这个曲线。**

## Q28 · Wi-Fi满格却上不了网——二层连接 vs 三层路由

### 你有没有思考过这么一个问题？

Wi-Fi 图标满格。但你打不开网页。重启路由器——好了。为什么"信号满格"不等于"能上网"？

### 底层发生了什么？

**Wi-Fi 信号满格（RSSI 强）只说明你跟路由器之间的无线链路质量好——这是二层（数据链路层）的事。**

但上网需要**三层（网络层）连通**：路由器必须通过 DHCP 给你分配 IP、子网掩码、默认网关、DNS；路由器的 WAN 口（上行链路）必须正常工作，能把你的包路由到互联网。

**"满格不通" = 二层 OK，三层出了问题。** 常见原因：路由器 WAN 口物理断开、PPPoE 拨号失败、运营商 DNS 故障、IP 地址冲突。

### 大白话解释

你和邻居老王拿着对讲机聊得热火朝天（**Wi-Fi 信号满格**）。但老王家的电话线断了（**路由器 WAN 口没通**）——他能跟你说"我也上不了网"，但对讲机解决不了电话线的问题。

### 核心启示

> 网络问题的排查永远从分层开始：**一层一层往上查，不要跳。** 满格不通 → 检查是否能 ping 通路由器（二层 OK 吗？）→ 检查路由器能否上网（WAN OK 吗？）→ 检查 DNS 解析（能拿到 IP 吗？）。**分层思维不仅是设计网络的工具，也是排查网络问题的工具。**

## Q29 · 设计一个P2P文件分发系统——像BitTorrent那样，人越多越快

### 你有没有思考过这么一个问题？

传统下载——所有人从同一个服务器下载。人越多越慢（服务器带宽被瓜分）。

BitTorrent——**人越多越快。** 为什么？

### 底层发生了什么？

BT 将文件切成多个 **Piece（通常 256KB-1MB）**。种子文件包含 Tracker 地址和所有 Piece 的 SHA-1 哈希。

- 下载者从 Tracker 或 DHT（分布式哈希表）获取其他 Peer 列表
- 向多个 Peer 并行请求不同 Piece
- 下载完一个 Piece，校验哈希，成功后宣告自己拥有此 Piece——**变成上传者**
- 其他 Peer 可以向你请求这个 Piece

**三个关键设计：**

- **Tit-for-Tat**：优先给上传速度快的 Peer 发数据——**鼓励贡献。** 只下载不上传？你的下载速度越来越慢。
- **Rarest First**：优先下载最少人拥有的 Piece——**保证稀缺数据分布。** 防止某个 Piece 只有一个人有、这个人下线后所有人都卡住。
- **DHT**：无 Tracker 模式——Peer 之间自己组成分布式网络，互相知道谁有数据。

### 大白话解释

BT = 你从仓库领了一批货（下载一个 Piece），然后你也变成了小仓库——后来的人可以直接从你这里拿货。人越多仓库越多——领货反而更分散更快。但如果你光拿货不开仓库（不上传），大家对你就"断供"。

### 核心启示

> P2P 的智慧在于**把"消费者"变成"供给者"。** 传统 C/S 架构是单向流——服务器的带宽是瓶颈。P2P 把每个下载者变成上传者，带宽随人数增长。**所有"共享经济"模型都在底层遵循这个逻辑：利用节点自身的资源，而非依赖中心节点的无限扩展。**
