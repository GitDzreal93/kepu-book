## Q56 · 验证码军备竞赛——为什么验证码越来越复杂？

![配图](images/chapter4/q56-captcha.png)

### 你有没有思考过这么一个问题？

早期的验证码是扭曲数字，现在已经进化成滑动拼图、点选文字、甚至"我不是机器人"的复选框。为什么验证码越来越复杂？因为攻击者也在不断升级：OCR 识别、机器学习模型、打码平台、人工众包破解。验证码是人和机器之间的一场军备竞赛。

### 底层发生了什么？

验证码（CAPTCHA，Completely Automated Public Turing test to tell Computers and Humans Apart）的设计目标是：**区分人类和机器**。如果攻击者能用自动化程序批量通过验证，注册、登录、投票、抢票等业务就会被滥用。

**验证码的发展**

1. **文本验证码**：扭曲数字/字母。早期有效，但很快被 OCR 破解。
2. **图像验证码**：识别图中的文字、物体、场景。需要更复杂的 CV 模型才能破解。
3. **滑动验证码**：拖动滑块拼图。结合鼠标轨迹、点击位置、完成时间做行为分析。
4. **行为验证**： Google's reCAPTCHA v3 不需要用户输入，而是在后台分析用户行为（鼠标移动、Cookie、浏览历史），给出"是人"的概率分数。
5. **无感验证**：结合设备指纹、环境信息、行为生物特征，在不知不觉中完成验证。

**验证码背后的风控**

现代验证码不只看"答案对不对"，还看"你怎么答的"：

- 鼠标移动是否自然？机器通常直线移动，人类有抖动和加速曲线。
- 点击间隔是否均匀？机器点击时间太规律。
- 是否用 headless 浏览器？无头浏览器往往缺少某些浏览器特性。
- 设备指纹是否异常？同一设备反复注册账号。

**验证码不是万能的**

- 打码平台：把验证码分发给真人解答，再返回结果。成本低廉，但确实是真人在答。
- 攻击者成本模型：验证码提高了攻击成本，但无法完全阻止有资源的大规模攻击。

### 大白话解释

验证码就像夜店的门禁：最开始看身份证（文本验证码），后来发现假证太多，改看人脸（图像验证码），再后来要看你会不会走路像真人（滑动验证码）。但门禁再严，也挡不住有人雇真人排队进场。

### 核心启示

验证码的本质是**提高自动化攻击的成本**。它不是绝对安全，而是安全体系中的一层。好的验证码系统要结合行为分析、设备指纹、频率限制、风控策略，形成多层防御。不要把所有安全希望寄托在验证码上。

## Q57 · 侧信道攻击——为什么 CPU 功耗和运算时间也能泄露密码？

![配图](images/chapter4/q57-side-channel.png)

### 你有没有思考过这么一个问题？

你设计了一套加密算法，数学上无懈可击。但攻击者不直接破解算法，而是观察你的 CPU 在运算时消耗的电力、散热风扇的声音、电磁辐射，甚至运算时间的长短，来推断你处理的数据。这就是侧信道攻击（Side-Channel Attack）。

### 底层发生了什么？

侧信道攻击利用的是**信息泄露的非预期渠道**。它不是攻击算法本身，而是攻击算法在物理实现上留下的痕迹。

**常见类型**

1. **时序攻击（Timing Attack）**

   某些操作的执行时间取决于输入数据。比如 RSA 的平方-乘模幂运算中，如果当前 bit 为 1，会多一次乘法操作。乘法比平方慢，攻击者测量成千上万次运算的耗时，可以逐 bit 还原私钥。

   防御：**恒定时间算法（Constant-Time Algorithm）**，让无论输入是什么，执行的操作序列和时间都完全相同。

2. **功耗分析（Power Analysis）**

   芯片执行不同指令时功耗不同。攻击者用高精度仪器测量功耗曲线，结合算法结构推断密钥。差分功耗分析（DPA）非常强大。

   防御：功耗平衡、随机延迟、噪声注入。

3. **电磁分析（EMA）**

   CPU 运算会产生电磁辐射，不同操作辐射模式不同。攻击者可以在不远处用天线截获信号。

4. **缓存侧信道**

   Spectre 和 Meltdown（2018 年公开）利用 CPU 的推测执行和缓存。攻击者通过测量内存访问延迟，推断出被推测执行访问过的数据，包括内核内存。这直接动摇了用户态和内核态隔离的根基。

   防御：微码更新、操作系统补丁、浏览器站点隔离、禁用高精度计时器。

5. **声学/热量攻击**

   风扇声音、CPU 温度变化也能泄露信息。虽然精度较低，但理论上可行。

### 大白话解释

你练开保险箱，每次转动密码盘，锁芯会发出"咔嗒"声。不同数字对应的咔嗒声微妙不同，转对的时候声音更清脆。你不需要撬锁，只需要"听"。侧信道攻击就是听 CPU 的咔嗒声、看运算的时间、感受电磁的泄露信号。

### 核心启示

侧信道攻击提醒我们：**安全不能只考虑协议和算法，还要考虑实现和物理环境**。一个数学上完美的加密系统，如果实现上有功耗泄露或时序差异，仍然会被攻破。高安全场景（密码学、支付、硬件安全模块）必须进行侧信道分析和防护。

## Q58 · 接口防刷——为什么 API 不能随便让人无限调用？

![配图](images/chapter4/q58-api-rate-limit.png)

### 你有没有思考过这么一个问题？

你做了一个公开 API，比如查天气。一开始谁都能用，结果有人写了个脚本每秒请求 10000 次，把你的服务器打挂，还把数据爬走卖给别人。接口防刷就是防止 API 被滥用的一系列机制。

### 底层发生了什么？

接口防刷的核心是**限制调用频率和识别调用者身份**。

**1. 限流算法**

- **固定窗口**：每秒钟允许多少请求。简单但窗口边界可能突刺。
- **滑动窗口**：统计最近 N 秒内的请求数，更平滑。
- **令牌桶（Token Bucket）**：系统以固定速率往桶里放令牌，请求消耗令牌。适合允许突发流量。
- **漏桶（Leaky Bucket）**：请求先进入桶，以固定速率流出。适合严格限制流出速率。

**2. 身份识别**

没有身份识别的限流容易被绕过：攻击者换 IP、换 User-Agent。常见做法：

- **API Key**：分配给每个开发者，按 Key 限流。
- **OAuth Token**：按用户或应用限流。
- **设备指纹 / 用户 ID**：结合账号和设备识别。
- **验证码**：当频率超过阈值时弹出验证码。

**3. 业务层防刷**

- 重要接口要求登录或付费。
- 对批量爬取行为做行为分析，如请求顺序过于规律、访问深度异常。
- 返回数据做脱敏，避免泄露敏感信息。
- 设置 IP 黑名单、User-Agent 黑名单。

**4. 成本不对称**

防刷的最高境界是**让攻击成本远高于收益**。比如每次 API 调用需要消耗积分、需要付费、需要完成人机验证。当攻击者发现成本高于爬取数据的价值，就会放弃。

### 大白话解释

API 就像一家公共图书馆。一开始谁都能免费借书，结果有人用推车把书全搬走。于是图书馆规定：每人每次最多借 5 本（限流），需要办借书证（身份识别），热门书需要预约（验证码/付费）。

### 核心启示

接口防刷不是简单加一道限流，而是**多层策略组合**：频率、身份、业务、成本。设计 API 时，要提前考虑被滥用的可能性，并为不同级别的调用者设置不同的配额和访问策略。

## Q59 · SIM 卡交换攻击——为什么骗子拿到你的手机号就能转走你的钱？

![配图](images/chapter4/q59-sim-swap.png)

### 你有没有思考过这么一个问题？

你的银行账户很安全，密码复杂、短信验证码只能发到你的手机。但骗子冒充你，打电话给运营商说"我手机丢了，请补张卡"，运营商把手机号换到骗子的新 SIM 卡上。于是银行发的验证码都到了骗子手里。这就是 SIM Swap 攻击。

### 底层发生了什么？

SIM Swap（SIM 卡交换）是一种社会工程学攻击。攻击者不攻击技术系统，而是攻击"运营商的客服流程"：

1. 骗子收集你的个人信息：姓名、身份证号、手机号、甚至住址。
2. 骗子联系运营商客服，声称手机丢失或 SIM 卡损坏，要求补办。
3. 如果运营商验证不严，骗子成功把你的手机号转移到新的 SIM 卡。
4. 骗子用这个手机号接收银行验证码、登录你的各种账号、重置密码、转账。

**防御方法**

- **不要在短信里收验证码做重要操作**：银行和重要账号尽量用 App 内推送、硬件安全密钥、TOTP（Time-based One-Time Password，基于时间的一次性密码）等更安全的二次认证。
- **为手机号设置 PIN 码**：SIM 卡 PIN 可以在手机重启时阻止 SIM 被直接使用。
- **运营商侧加强身份验证**：用户要求换卡时必须严格验证身份。
- **账号增加非短信的二次认证**：如 Google Authenticator、YubiKey。
- **关注异常短信**：如果手机突然没信号，可能是 SIM 被交换，应立即联系运营商。

### 大白话解释

SIM Swap 就像有人冒充你去物业说"我钥匙丢了，请给我换把锁"。物业没认真核实，就真给他换了。结果那人拿新钥匙进了你家，还把你的快递（验证码）全收了。

### 核心启示

SIM Swap 暴露了一个根本问题：**短信验证码不是强认证**。它依赖运营商，而运营商的流程可能被社会工程学攻破。关键账号应该使用 TOTP 或硬件密钥，而不是只依赖短信。安全设计要假设任何单一因素都可能失效，使用多因素认证（MFA）。

## Q60 · 撞库攻击——为什么你的密码在其他网站泄露后，这里的账号也不安全了？

![配图](images/chapter4/q60-credential-stuffing.png)

### 你有没有思考过这么一个问题？

你注册了 30 个网站，很多都用同一个密码。某天 A 网站数据库泄露，攻击者拿到你的邮箱和密码后，去 B 网站、C 网站、D 网站尝试登录。如果 BCD 网站你也用了同样密码，攻击者就成功了。这就是撞库（Credential Stuffing）。

### 底层发生了什么？

撞库利用的是**人类喜欢重复用密码的弱点**。它不是暴力破解，而是"拿着已泄露的账号密码，批量尝试登录其他平台"。成功率往往不低，因为大量用户确实在多个网站用相同密码。

攻击流程：

1. 收集泄露的账号密码数据库（如 "combos"）。
2. 用自动化工具或脚本，针对目标网站批量登录。
3. 成功的账号被收集起来，用于进一步诈骗、盗刷、发垃圾信息。

**防御方法**

- **用户教育**：提醒用户不同网站用不同密码，使用密码管理器。
- **检测异常登录**：同一 IP 短时间大量尝试不同账号、登录地点异常、设备指纹变化。
- **强制 MFA**：即使密码泄露，没有二次认证也进不来。
- **IP 限流**：单个 IP 登录失败次数过多就封禁或加验证码。
- **密码泄露检测**：一些服务（如 Have I Been Pwned）会检查用户密码是否在已知泄露数据库中。
- **暗网监控**：企业监控自己账号是否在暗网被贩卖。

### 大白话解释

撞库就像有人捡到了一把钥匙，他不知道开哪扇门，于是去小区里挨家挨户试。如果很多人都用同一把钥匙（同一个密码），他很快就能打开好几户。

### 核心启示

撞库攻击的源头是**密码复用**。任何系统都无法保证用户密码不泄露，但可以通过 MFA、异常检测、登录限流来降低风险。作为用户，最重要的是用密码管理器生成唯一强密码；作为平台，必须假设用户密码可能在别处泄露，设计相应的防御机制。
