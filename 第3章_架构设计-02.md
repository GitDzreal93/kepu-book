## Q38 · 分库分表——一张表到了10亿行，怎么拆？

### 你有没有思考过这么一个问题？

用户表到了 10 亿行，单库扛不住了。你说"分库分表！"——但按什么字段拆？拆完怎么跨库 join？怎么保证分布式事务？怎么无感迁移？

### 底层发生了什么？

**分片键选择：** 按 user_id 哈希取模 → 用户相关的查询都落在同一分片。按时间分片 → 查询热点集中在最新分片。

**跨库 join 的代价：** 分库后，SQL join 不能跨库——只能**应用层拼接**（查分片 A 拿一部分数据、查分片 B 拿另一部分、应用层合并）。或者用"异构索引"——冗余一份按另一维度分片的数据。

**分布式事务：** 分库后本地事务不跨分片。跨分片的操作需要 2PC、TCC 或 Saga 保证一致性——代价极大。**所以分片键的选择决定了"多少操作变跨分片"。**

### 大白话解释

分库分表 = 一个仓库满了，你得把货分到多个仓库。按"用户 ID 尾号"分——同一个用户的货都在一个仓库，查起来方便。但如果需要"所有昨天的订单"——每个仓库都要去一趟。**分片键决定了你的"方便方向"。**

### 核心启示

> 分库分表不是"拆了就快"——是**把"单库的容量瓶颈"换成了"跨库查询的复杂度"。** 选分片键 = 选"什么查询变慢"。**所有分片设计的核心是：让 80% 的查询落在单个分片上。**

## Q39 · 设计一个分布式ID生成器——不用数据库自增、不用Redis

### 你有没有思考过这么一个问题？

几百台机器，各自独立生成全局唯一 ID。不能靠数据库自增（单点故障），不能靠 Redis（单点）。**怎么让几百台机器不互相商量，生成永不重复的 ID？**

### 底层发生了什么？

**Snowflake 算法：** 64 位——1 位未用 + 41 位毫秒时间戳 + 10 位机器 ID + 12 位序列号。

各机器独立生成，无需协调。每毫秒每台机器能生成 4096 个 ID。时间戳保证趋势递增（对数据库索引友好）。

**时钟回拨是最大挑战。** NTP 同步导致时间倒退 → 可能产生重复 ID。解法：等时间追上（阻塞等）或使用扩展位。

**号段模式（美团 Leaf）：** 从数据库批量取一段号段（如 1-1000）缓存在本地，用完再取下一段。减少 DB 交互——但号段不连续。

### 大白话解释

Snowflake = 每个人自己生成身份证号：前几位是省市区（机器 ID）+ 出生年月日（时间戳）+ 一个自增序号。不需要问中央机构——自己就能算出全国唯一的号。

### 核心启示

> 分布式 ID 的核心矛盾：**确定性（唯一）vs 去中心化（不依赖单点）。** Snowflake 在确定性上做了去中心化改造——各机器独立生成，靠时间戳和机器 ID 的组合保证唯一。**时钟回拨是这个方案唯一的阿喀琉斯之踵。**

## Q40 · 微服务 vs 单体——10人团队20个微服务，是不是在找罪受？

### 你有没有思考过这么一个问题？

微服务很火——拆！你们团队 10 个人，维护了 20 个微服务。每次改一个小需求要改 4 个服务。部署一次要跑 20 条流水线。排查一个问题要翻 5 个服务的日志。

**微服务没有消除复杂性——把"代码的复杂度"转移成了"运维的复杂度"。**

### 底层发生了什么？

单体的优势：单进程，函数调用（无需序列化/网络），本地事务（无分布式事务），部署简单。劣势：耦合——修改 A 可能影响 B。

微服务的代价：网络延迟累加（一个请求经过 10 个微服务 → 端到端 ≥ 10×每跳延迟），分布式追踪困难，运维成本线性增长。

**判断标准：** 不是"单体 vs 微服务"——是"如果把这个模块拆成独立部署，代价和收益分别是什么？" 拆的收益 > 拆的代价 → 拆。10 人团队维护 20 个微服务 → 拆的代价远超收益。

### 大白话解释

单体 = 一套大公寓，所有人住一起——共享水电（共享内存），沟通方便（函数调用）。但如果要改格局——牵连所有人。微服务 = 拆成 N 个小房间——各管各的，但房间间沟通需要"发消息"（网络通信），且需要额外管委会（服务治理）。**10 个人住 20 个小房间——管委会比住户还多。**

### 核心启示

> 微服务是组织架构的映射，不是技术选型。**康威定律：系统设计必然反映组织沟通结构。** 如果你的团队本身是 10 人紧密协作——你的系统应该是模块化单体，不是 20 个微服务。**先拆团队，再拆服务。**

## Q41 · 设计一个支付系统——下单、支付、回调、退款、对账

### 你有没有思考过这么一个问题？

用户下单付了钱，支付回调延迟了怎么办？用户付了两次怎么保证不扣两次钱？对账发现差了一分钱怎么修？

### 底层发生了什么？

**支付流程：** 用户下单 → 生成支付订单 → 调用支付渠道 → 用户付款 → 支付渠道异步回调 → 更新订单状态。

**回调延迟：** 回调可能延迟数小时甚至丢失。解法：**主动查询 + 回调双重确认。** 定时任务查询未支付订单，主动向支付渠道确认状态。

**幂等：** 用户点了两次支付按钮 → 生成两个支付订单？不——**订单号 + 支付流水号联合唯一索引。** 重复请求被数据库唯一约束拦截 → 返回"已支付"。

**对账：** 你的账单 vs 支付渠道账单。T+1 对账发现差异 → 长款（你多收了）/ 短款（你少收了）→ 人工或自动调账。

### 大白话解释

支付系统 = 一个必须有记账本、有备用方案、能对账的收银台。用户说"我付了"——你不能只信回调（可能丢），还得主动去问支付渠道"他真的付了吗？"付了两次？记账本上有唯一订单号——第二次自动说"已付，别重复"。月底对账差了一分钱——记账本和支付渠道对一下，发现谁记错了，修正。

### 核心启示

> 支付系统的核心不是"接收付款"——是**"在任何异常情况下，钱不会多扣、不会少扣、不会丢。"** 主动查询兜底回调、唯一索引兜底幂等、对账兜底长款短款。**关键路径上永远有备用方案，没有备用方案的那一步就是你的单点风险。**

## Q42 · 设计一个配置中心——动态配置推送、灰度、回滚

### 你有没有思考过这么一个问题？

线上服务有一个功能开关。你想关掉它——不改代码、不重启、不停服。怎么做到？

静态配置文件改不了——改完要重启。重启 = 服务不可用窗口。配置中心解决了这个问题。

### 底层发生了什么？

**配置推送：** 配置变更 → 配置中心（Apollo/Nacos）→ 长轮询或实时推送通知所有订阅客户端 → 客户端刷新本地缓存。

**灰度发布：** 不是全量推送——先 1% 的节点收到新配置，观察稳定后逐步放量。出问题？立即回滚到历史版本。

**版本管理：** 每次变更留记录——谁、什么时间、改了什么。一键回滚到任意历史版本。

配置 = 代码的一部分，需要 Code Review 和发布审批。

### 大白话解释

配置中心 = 餐厅的电子菜单屏幕。改价格只改一次——所有厨师（服务节点）看同一块屏幕自动更新。如果新价格导致客人投诉（异常）——一键切回旧菜单（回滚）。

### 核心启示

> 配置中心的本质是**把"配置变更"从"部署流程"中解耦出来。** 部署是重操作——重启、滚动更新、等待就绪。配置变更是轻操作——一个 HTTP 请求级别。**"变"和"部署"分离——是持续交付的核心。**

## Q43 · 分布式事务——2PC、TCC、Saga，什么时候用什么？

### 你有没有思考过这么一个问题？

订单服务扣了库存，要通知积分服务加积分。库存扣成功了，积分服务挂了——库存应该回滚。但这个"回滚"——在分布式系统里叫**分布式事务**，比你想象中难得多。

### 底层发生了什么？

**2PC（两阶段提交）：** 协调者 → Prepare："你们能提交吗？" → 全答"能"→ Commit。全答"能"之后协调者挂了——参与者锁住的资源不知道该提交还是回滚——**事务悬挂。**

**TCC（Try-Confirm-Cancel）：** 业务层面。Try（预留资源——冻结 100 元）→ Confirm（正式扣款）→ Cancel（释放冻结的 100 元）。比 2PC 灵活——但**业务代码必须显式实现补偿逻辑。**

**Saga：** 长事务拆成多个本地事务序列。每个本地事务有对应的**补偿事务**。任何步骤失败 → 逆序执行补偿。最终一致性——中间态可见。适合长流程（如电商下单 10 个步骤）。

### 大白话解释

2PC = 全部说"能"才一起动——一个人说"不能"全部暂停。TCC = 每个人先"预留资源"（冻结钱），全预留成功再"确认扣除"。Saga = 每个人做完自己的事，标记好"如果失败了，这样补救"。失败时按倒序一个个补救。

### 核心启示

> 分布式事务没有银弹。2PC 简单但锁太久，TCC 灵活但代码复杂，Saga 长流程但中间态可见。**选择 = 你能接受多久的不一致窗口 + 你愿意写多少补偿代码。**

## Q44 · 设计一个API网关——认证、限流、路由、协议转换

### 你有没有思考过这么一个问题？

前端请求先经过网关。网关做什么？不是"转发"——是**认证、限流、路由、协议转换。** 一个请求经过网关的完整生命周期是什么？

### 底层发生了什么？

**认证鉴权：** 统一入口验证 JWT/OAuth Token——避免每个微服务自己验证。

**流量控制：** 限流（令牌桶/漏桶）——防止某个用户或 IP 打爆后端。熔断——下游连续失败 → 自动切断一段时间。

**路由转发：** 根据路径/Header 路由到不同微服务。内部可以做协议转换——外部 RESTful → 内部 gRPC。

**可观测性：** 统一打日志、埋 Metrics、注入 TraceID——全链路追踪的关键节点。

### 大白话解释

API 网关 = 公司大门保安。访客进门：查证件（认证）、查有没有预约（授权）、记录来访（日志）、人太多排队（限流）、问"你找哪个部门"指路（路由）。

### 核心启示

> 网关不是反向代理——是**"横切关注点"的集中处理层。** 认证、限流、日志、追踪——每个微服务都需要但每个微服务都不应该自己实现的东西——交给网关统一做。**集中横切关注点 = 让微服务只关心业务逻辑。**

## Q45 · 游戏App的热更新——怎么不重新下载整个App就更新内容？

### 你有没有思考过这么一个问题？

你打开王者荣耀——"正在更新资源包，48MB"。没有让你去 App Store 重新下载 20GB 的完整包。**游戏怎么做到只更新"改过的部分"？**

### 底层发生了什么？

热更新分两种：

**资源更新（最常见）：** 替换图片/音频/配置等非代码资源。下载差分包（bsdiff 计算新旧版本差异 → 极小 patch）→ 替换本地资源目录 → 重启生效。

**代码更新：** iOS 被 App Store 严格限制（禁止动态加载可执行代码）。只能用解释型脚本——**Lua/JS 嵌入**更新逻辑。Android 可用 ClassLoader 动态加载 dex/jar。

**patch 容错：** patch 应用失败 → 完整包兜底 → 回滚机制。

### 大白话解释

你要修一栋楼——不能拆了重建（重新下载 App），只能趁住户不在时换掉坏墙砖（替换资源文件）。如果改动太大（加整层楼）——那确实只能重建。脚本方案 = 修改物业运营手册（Lua 更新逻辑）——不用动楼的结构。

### 核心启示

> 热更新的核心是**把"改了什么"和"没改什么"分离开。** 差分包（bsdiff）只传输变化的部分——把"完整替换"变成"增量修补"。**所有的增量更新系统——Git Diff、Docker 层缓存、OTA 差分包——都在重复这个模式。**
