## Q41 · 设计一个幂等的支付接口——用户点了两次支付，钱会被扣两次吗？

![配图](images/chapter3/q41-idempotent-payment.png)



### 你有没有思考过这么一个问题？

你点外卖，提交订单时不小心点了两次"确认支付"，结果收到两条扣款短信。这就是支付接口没有做好**幂等性（Idempotency）**。幂等性是指：同一个操作执行一次和执行多次，结果相同。

面试会问：支付接口怎么设计幂等？幂等键（Idempotency Key）是什么？如何防止重复支付？网络超时导致的不确定状态怎么处理？

### 底层发生了什么？

支付接口幂等需要解决三类问题：重复请求、超时重试、回调乱序。

**1. 幂等键（Idempotency Key）**

前端在发起支付请求时，生成一个全局唯一 key（如 UUID + 用户 ID + 订单号），放在请求头里。后端用这个 key 作为业务去重标识：

- 如果 key 已经处理过，直接返回上次结果，不重复扣款。
- 如果 key 不存在，执行扣款，并把结果记录到幂等表或 Redis 中。

幂等键要有过期时间，通常 24 小时，避免 key 无限增长。过期后，用户如果要重新支付，需要生成新的幂等键。

**2. 重复请求**

同一个按钮连续点击两次：前端可以做按钮置灰，但后端不能依赖前端。后端收到两个相同幂等 key 的请求，只有第一个真正执行，第二个返回"处理中"或"已处理"。

**3. 超时重试**

客户端发起支付后，网络超时了。客户端不知道服务器是否收到请求，通常会重试。如果服务器没有幂等控制，重试就会导致重复扣款。

幂等 key 就是解决这个问题的：服务器用 key 判断"这个支付请求我已经处理过了，不用再执行"。

**4. 回调乱序**

支付成功后，第三方支付（如支付宝、微信支付）会回调商户服务器。回调可能重复，也可能乱序。商户侧要根据回调里的订单号 + 支付状态做幂等：

- 如果订单已经是"已支付"，后续回调直接返回成功。
- 如果订单状态更"旧"，忽略。

**5. 对账补偿**

即使幂等设计再好，也可能出现银行和商户系统状态不一致。所以需要**日终对账**：把本地订单状态和第三方支付流水逐笔核对，发现差异后人工或自动补偿。

### 大白话解释

幂等就像你去餐厅买单。你给服务员一张一百块，服务员把钱放进收银台。如果因为网络不好，你以为没付成功，又点了一次支付，服务员看到收银台里已经有一张你的钱了，就不会重复收。

幂等键就是这张钱的编号：你每次付钱都有一个唯一的钞票编号，服务员看到编号就知道"这笔钱我收过了"。

### 核心启示

支付接口幂等的本质是**用唯一标识建立"请求-结果"的映射表**。任何写操作（扣款、下单、转账、库存扣减）在分布式系统中都要考虑幂等，因为网络超时和重试是不可避免的。设计幂等时，不仅要防重复请求，还要防重复回调、乱序到达、状态不一致。

## Q42 · 配置中心——为什么改一个配置就要重新发版？

![配图](images/chapter3/q42-config-center.png)



### 你有没有思考过这么一个问题？

你公司系统的某个参数写死在代码里，比如"每页显示 10 条"。运营说活动页要改成 20 条，结果你只能改代码、重新打包、发布上线。如果系统有几百台机器，这个流程可能要好几个小时。配置中心就是为了解决这个问题而生的。

面试会问：配置中心和注册中心有什么区别？配置变更怎么推送到所有机器？Apollo、Nacos、Spring Cloud Config 各有什么特点？配置变更如何保证一致性？

### 底层发生了什么？

配置中心（Configuration Center）是把应用配置从代码里剥离出来，集中管理、动态下发。它的核心能力：

1. **集中管理**：所有配置在一个地方修改，不用每台机器去改。
2. **动态推送**：配置修改后，实时推送到客户端，不需要重启应用。
3. **版本控制**：记录配置的历史版本，可以回滚。
4. **权限控制**：谁能看什么配置、谁能改什么配置，做权限隔离。

**配置中心的典型架构**

- **服务端（Config Server）**：存储配置，提供 API 查询，负责推送变更。
- **客户端 SDK**：内嵌在应用里，连接服务端，监听配置变化。
- **存储层**：通常用 MySQL 或 etcd 存储配置，用长连接或轮询机制推送变更。

**配置推送机制**

- **长连接推送**：服务端和客户端保持长连接，配置变更时主动推送。延迟低，但连接维护成本高。
- **定时轮询**：客户端每隔几秒查询一次。实现简单，但延迟较大。
- **混合模式**：长连接 + 兜底轮询。Apollo 采用这种方式。

**配置类型**

- **静态配置**：启动时加载，运行中不变。如数据库连接池大小。
- **动态配置**：运行中可变更。如开关、阈值、Feature Flag（特性开关）、页面文案。
- **敏感配置**：密码、密钥等，需要加密存储和传输。

**配置中心和注册中心的区别**

- **配置中心**：管"配置参数"。
- **注册中心**：管"服务地址"。如 Eureka、Nacos、Consul、ZooKeeper。服务启动后把地址注册上去，调用方从注册中心发现服务实例。

两者有时会共用组件（如 Nacos 同时支持配置管理和服务发现），但解决的问题不同。

### 大白话解释

配置中心就像剧院的灯光控制台。以前每个灯有自己的开关在墙上，想调整全场灯光要跑遍整个剧院。配置中心把所有开关集中到一个控制台，灯光师一键调整，所有灯立刻响应。

### 核心启示

配置中心把"代码"和"配置"分离，让系统在不发版的情况下就能响应业务变化。它是**持续交付和快速试错**的基础设施。设计系统时，要把可能变化的参数（开关、阈值、文案、策略）抽出来，不要写死在代码里。

## Q43 · 分布式事务——A 给 B 转账，如何同时保证两个数据库一致？

![配图](images/chapter3/q43-distributed-transaction.png)



### 你有没有思考过这么一个问题？

你从一个银行转账到另一个银行。转出银行扣了钱，但转入银行因为网络故障没加上钱。这 100 块去哪了？分布式事务要解决的，就是这种跨多个数据库或服务的操作**要么都成功，要么都回滚**的问题。

面试会问：两阶段提交（2PC）是什么？TCC、Saga、本地消息表、最大努力通知各是什么？CAP 定理下分布式事务的取舍？

### 底层发生了什么？

单机数据库用 ACID 事务保证一致性。但分布式环境下，不同数据库之间无法直接共享事务，需要额外协议。

**两阶段提交（2PC, Two-Phase Commit）**

2PC 有一个协调者（Coordinator）和多个参与者（Participant）。

- **阶段一（Prepare）**：协调者问所有参与者："你能执行这个操作吗？" 参与者执行本地事务但不提交，回复 Yes 或 No。
- **阶段二（Commit/Rollback）**：如果所有参与者都回复 Yes，协调者发 Commit 指令；如果有任何参与者回复 No 或超时，发 Rollback 指令。

2PC 的问题是：

- 协调者单点故障：如果协调者挂了，参与者可能一直等待。
- 同步阻塞：参与者要锁定资源直到收到最终指令，性能差。
- 不能 100% 保证一致性：如果协调者 Commit 指令发出后部分参与者没收到，会出现不一致。

**TCC（Try-Confirm-Cancel）**

TCC 把业务拆成三个阶段：

- **Try**：预留资源，比如冻结用户余额。
- **Confirm**：真正执行业务，比如扣减冻结金额。
- **Cancel**：回滚，释放预留资源。

TCC 性能比 2PC 好，但业务代码需要为每个操作写三套逻辑，侵入性强。

**Saga 模式**

把长事务拆成多个本地事务，每个本地事务有对应的补偿操作。如果中间某步失败，就按相反顺序执行补偿，把已执行的步骤回滚。适合业务流程长、可以最终不一致的场景，如机票酒店预订。

**本地消息表 / 最大努力通知**

- **本地消息表**：业务操作和消息写入同一个本地事务，保证消息不丢失。然后异步发送消息，由消费者处理。
- **最大努力通知**：服务 A 尽最大努力通知服务 B，B 接收后处理。如果通知失败，有补偿机制（如日终对账）。适合对一致性要求不高的场景，如积分、通知。

### 大白话解释

两阶段提交就像你和几个朋友凑钱买礼物。你先问每个人："钱准备好了吗？"大家都说"准备好了"。然后你说："买！" 大家一起掏钱。如果有人说没准备，你就说："算了，都别买。"

问题是：如果说完"买"之后你手机没电了，有人没听见，他就没掏钱，结果大家凑不齐。这就是协调者故障导致的不一致。

Saga 模式则像一场旅行：先订机票，再订酒店，再订车。如果订车失败，就取消酒店和机票，按相反顺序撤销。虽然有短暂不一致，但最终能回滚到干净状态。

### 核心启示

分布式事务没有完美解决方案。2PC 追求强一致性但性能差；TCC 性能好但业务侵入大；Saga 和消息表追求最终一致性，适合大多数互联网业务。根据业务对一致性的要求选择方案，不要追求"绝对一致"而牺牲系统可用性。

## Q44 · API 网关——为什么微服务架构里需要一个"大门"？

![配图](images/chapter3/q44-api-gateway.png)



### 你有没有思考过这么一个问题？

微服务架构里，客户端可能要调用用户服务、订单服务、库存服务、支付服务……每个服务都有自己的地址和认证方式。如果客户端直接和这些服务打交道，会很混乱。API 网关就是在客户端和后端服务之间加一道"大门"。

面试会问：API 网关和反向代理有什么区别？网关做什么？网关的瓶颈在哪里？

### 底层发生了什么？

API 网关（API Gateway）是系统的统一入口。客户端只访问网关，由网关根据请求路由到后端服务。

**网关的核心职责**

1. **路由**：根据 URL、Header、Method 把请求转发到对应服务。如 `/api/users` 到用户服务，`/api/orders` 到订单服务。
2. **认证鉴权**：统一校验 JWT、OAuth Token、API Key，判断用户是否有权限。
3. **限流熔断**：防止流量把后端服务打挂。超过阈值就拒绝或降级。
4. **协议转换**：外部用 HTTP/REST，内部用 gRPC、Thrift，网关负责转换。
5. **负载均衡**：后端服务多个实例时，网关做负载均衡。
6. **日志监控**：统一记录请求日志、响应时间、错误率，便于观测。
7. **缓存**：对部分响应做缓存，减少后端压力。

**API 网关 vs 反向代理**

反向代理（如 Nginx）主要是转发请求，做负载均衡和静态缓存。API 网关在此基础上增加了更多应用层能力：认证、路由、限流、协议转换、API 生命周期管理。

**网关的性能和可用性**

网关是所有流量的入口，一旦挂掉，整个系统不可用。所以网关本身需要：

- 高可用部署：多实例 + 前置负载均衡。
- 无状态设计：不把会话状态存在网关上，方便横向扩展。
- 轻量逻辑：网关只负责通用横切逻辑，业务逻辑下沉到微服务。

常见的网关产品：Kong、Nginx + Lua/OpenResty、Spring Cloud Gateway、Envoy、Zuul。

### 大白话解释

API 网关就像一栋大楼的物业前台。快递员（客户端）不用直接敲每家每户的门，而是把包裹送到前台，前台根据地址分发到各住户（后端服务）。前台还负责检查访客身份（认证）、记录进出（日志）、高峰期限制外来人员进入（限流）。

### 核心启示

API 网关是**微服务架构的横切关注点聚合层**。它把认证、限流、路由、监控等通用能力从各个服务里抽出来，避免重复实现。但网关不能太重，否则会成为新的单点瓶颈。设计网关时，要牢记"入口薄、服务厚"的原则。

## Q45 · 热更新——为什么游戏 App 不用重新下载安装包就能更新？

![配图](images/chapter3/q45-hot-update.png)



### 你有没有思考过这么一个问题？

你玩手游，每次大版本更新都要去应用商店下载几百 MB 的安装包。但小版本更新，比如修 bug 或调数值，游戏经常提示"资源更新中"，几分钟就完事了。这就是热更新（Hot Update）：在不重新安装 App 的情况下，替换部分资源或代码。

面试会问：热更新和 OTA 升级有什么区别？热更新怎么保证代码安全？Unity / 微信小程序 / React Native 的热更新机制各有什么不同？

### 底层发生了什么？

热更新通常更新的是**资源文件和脚本**，而不是底层二进制。它的核心流程是：

1. **版本检查**：App 启动时向服务器请求当前版本号和资源清单。
2. **差异对比**：App 把本地资源清单和服务器清单对比，找出新增、修改、删除的文件。
3. **下载补丁**：只下载有变化的文件，通常用差分包进一步减少体积。
4. **替换资源**：把新资源放到 App 沙盒的特定目录，覆盖旧资源或并行加载新资源。
5. **重启生效**：有些热更新需要重启 App 或重新加载脚本才能生效。

**热更新 vs OTA 升级**

- 热更新：更新资源、脚本，不需要重新安装，用户无感知或感知很小。速度快，但只能改表层。
- OTA 升级：更新整个 App 二进制，需要重新安装。适合底层引擎、框架、大版本升级。

**安全问题**

热更新最大的风险是：**如果更新包被中间人截获或篡改，攻击者可以直接在设备上运行恶意代码。** 所以必须：

- 更新包用私钥签名，客户端用公钥验证。
- 更新包下载用 HTTPS。
- 对敏感更新（如支付逻辑）限制热更新，必须走应用商店审核。

iOS 平台对热更新有严格限制：苹果不允许 App 下载可执行代码替换自身。所以 iOS 游戏通常只更新资源（图片、配置、音频），不能更新脚本引擎。Android 和微信小程序的限制相对宽松。

**不同技术栈的热更新**

- **Unity / Unreal**：通常用 AssetBundle 或 Addressable 资源系统更新资源，脚本更新受限。
- **React Native**：可以更新 JavaScript Bundle，但原生模块不能改。
- **微信小程序**：由微信框架统一提供热更新能力，开发者无需关心。

### 大白话解释

热更新就像你给房子换壁纸和家具，不用推倒重建。OTA 升级就像重新盖一栋房子。换壁纸快，但改变不了房子的结构；重建慢，但能把结构也改了。

但换壁纸也有风险：如果有人偷偷给你换了有毒油漆（恶意更新包），你就要遭殃。所以每次换壁纸前要检查油漆的品牌认证（签名验证）。

### 核心启示

热更新是**提升迭代速度和用户体验的重要手段**，但它和安全性、平台政策是冲突的。设计热更新方案时，必须明确：哪些内容可以热更新，哪些必须走完整发版；更新包如何签名和校验；失败时如何回滚。不要为了追求快，绕过安全底线。
