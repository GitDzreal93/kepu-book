# 第五章 | 工程实践

> 软件工程不是"写代码"——是管理复杂度。从 CI/CD 到测试金字塔、从 Git 到技术债、从微服务到可观测性。

---

## Q66 · 设计CI/CD流水线——代码提交后自动到上线

代码 push → 编译 → 静态检查 → 单元测试 → 集成测试 → 构建镜像 → 部署测试环境 → 部署预发布 → 人工审批 → 灰度发布 → 全量上线。

**核心价值：** 消除"在我机器上能跑"（Docker 保证环境一致）+ 失败快速发现（提交几分钟暴露问题）+ 发布从"大事件"变成"日常操作"。

**大白话：** 做菜流水线。切好菜（提交）→ 自动检测农药残留（编译+检查）→ 自动试吃（测试）→ 自动装盘打包（构建）→ 自动端到测试间（部署）→ 老板尝过点头（审批）→ 端到所有客人桌上（发布）。

**核心启示：** CI/CD 不是"自动化工具"——是**让"发布"从恐惧变成日常。** 当你一天可以发布十次，每次只改一点点时，排查范围从"三个月改了 500 个文件"缩小到"最近一次改了 3 个文件"。

## Q67 · 测试金字塔——为什么单元测试多而快，E2E少而慢？

**单元测试**（底层 70-80%）：测试单个函数/方法，毫秒级。**集成测试**（中层 15-20%）：测试模块间交互，秒级。**E2E**（顶层 5-10%）：模拟真实用户操作，分钟级，脆弱且维护成本高。

**反金字塔**（大量 E2E + 少量单元）= CI 跑 2 小时 → 开发跳过测试直接提交 → 线上事故。

**大白话：** 盖房子——单元测试 = 检查每块砖是否合格；集成测试 = 检查砖之间水泥是否牢固；E2E = 检查整栋房子所有系统协同工作。大部分时间花在检查砖头（单元），少量时间测整栋（E2E）。

**核心启示：** 测试金字塔不是教条——是**经济学。** 单元测试便宜且定位精准 → 堆量。E2E 昂贵且脆弱 → 只覆盖核心路径。**把测试成本放在"最可能出问题且修起来最便宜"的层。**

## Q68 · Git Rebase vs Merge——提交历史的两种哲学

**Merge：** 创建 Merge Commit 连接两个分支——保留真实并行开发的时间线，DAG（有向无环图）。事实真相。**Rebase：** 把 feature 分支的 commits 搬到目标分支顶端——历史变一条直线。好看但不真实，改写了 SHA。

**黄金法则：永远不要 rebase 已 push 的公共分支。** 因为重写历史 SHA 后，其他基于旧 commit 的人会冲突到崩溃。

**大白话：** Merge = "A 和 B 各自干了什么，最后结果合在一起"——保留真实时间线。Rebase = "把 B 做的事嫁接到 A 最新版上，假装 B 是从 A 最新版开始的"——好看但是假历史。

**核心启示：** 选 Merge 还是 Rebase = **你更在乎"历史的真相"还是"历史的可读性"。** 个人 feature 分支 → Rebase 整洁。公共 main 分支 → Merge 保留真相。

## Q69 · 蓝绿部署、金丝雀发布、灰度发布

**蓝绿：** 两套完整环境切换，秒级回滚。成本：双倍基础设施。**金丝雀：** 1%→5%→25%→100% 逐步放量，异常自动回滚。暴露面小。**灰度（Feature Flag）：** 代码已部署但功能用开关控制，按用户分组开放。

三者可以组合：蓝绿切换环境 + 金丝雀逐步放量 + Feature Flag 控制功能。

**大白话：** 蓝绿 = 有两套宴会厅——这周用蓝色，下周绿色。金丝雀 = 新厅先放 5% 客人试吃——不好只影响 5%。Feature Flag = 菜单上一道菜写着"仅 VIP 可见"——已做好，看人上菜。

**核心启示：** 发布策略的核心不是"怎样上线"——是**"出问题了怎样以最小损失回退"。** 回滚速度比上线速度更重要。

## Q70 · 技术债——什么债值得借，什么债绝对不能借？

技术债不总是坏的——就像企业借贷，合理杠杆加速发展。但要区分：

**值得借：** 非核心业务、明确后期会重构、验证 MVP 的临时方案。**绝对不借：** 支付核心、安全相关、数据库 schema 设计——这些一旦错，修起来是灾难。

管理技术债 = 把债可视化（SonarQube 度量）+ 分配固定比例资源还债（如每 Sprint 20% 时间）。

**大白话：** 开餐厅——开业第一周菜单手写（技术债），你知道该做点餐系统，但现在做意味着推迟开业一个月。先开张，边赚边补。但食品安全（支付安全）绝对不能马虎。

**核心启示：** 技术债不是"烂代码"——是**"为了更快上线故意简化了设计，后面要重构"。** 区别是：你知道这是债，你有还债计划。烂代码是不知道它有问题。

## Q71 · 设计监控告警系统——百万台服务器，怎么不告警风暴？

Metrics（指标：CPU 90%）告诉你"有问题"，Tracing（链路追踪）告诉你"在哪段"，Logging（日志）告诉你"具体什么"。

**防告警风暴：** 一个交换机挂了触发 1000 条告警 → 合并成一条"交换机 X 挂了"。连续 N 次超时才告警（防止网络短暂抖动误报）。

**大白话：** 监控自己健康——不能只测"有没有呼吸"（活着），还要测心跳速率（Metrics）、记录每天吃了什么（Logging）、追踪"为什么周二下午头疼"（Tracing）。

**核心启示：** 可观测性 ≠ 监控。监控只能回答"已知的已知"（CPU > 90% 了吗？）。可观测性能回答"未知的未知"（为什么延迟突然高了？）。

## Q72 · 大厂为什么造轮子？

不是"有 Kafka 了为什么还自研消息队列"——是 Kafka 的存储模型在你的极端场景下变成了瓶颈。

**理性决策条件：** 差异化核心业务、规模超过现有方案极限、外部方案无法满足合规。**徒劳的条件：** 非核心、规模小、成熟方案够用。

**大白话：** 开餐馆——大众点评给你现成点餐系统，80% 吻合需求。但你想加拼桌功能——做不到。自研 = 完全定制但需投入两个工程师半年。大厂选自研 = 等不了开源社区响应自己的需求 + 系统差异大到通用轮子不再适用。

**核心启示：** 造轮子不是技术决策——是**"定制化收益"vs"维护成本"的经济决策。** 大厂开源自己的轮子（K8s/React）又是另一个逻辑——降低维护成本（外部贡献者）+ 建立生态标准。

## Q73 · 单体 vs 微服务——10人20个微服务是找罪受

微服务把"代码的复杂度"转移成了"运维的复杂度"。10 人团队维护 20 个微服务——每次改需求改 4 个服务，排查问题翻 5 个服务日志——**运维开销超过业务价值。**

**判断标准：** 拆的收益 > 拆的代价 → 拆。否则保持模块化单体（源码级模块隔离，代替网络级服务隔离）。

**大白话：** 微服务 = 拆成 N 个小房间各管各的，但房间间需要发消息（网络通信）+ 额外管委会（服务治理）。10个人住 20 个小房间 = 管委会比住户还多。

**核心启示：** 康威定律：**系统设计必然反映组织沟通结构。** 先拆团队，再拆服务。

## Q74 · 设计RESTful API——资源建模、分页、幂等

**资源建模：** `/users/123/orders` vs `/orders?userId=123` ——前者表达"用户 123 的订单"的从属关系。**分页：** Offset 分页在数据变化时有重复/遗漏风险；Cursor 分页（基于 ID）是数据量大的唯一解。**幂等：** 相同 POST 发两次，用 Idempotency-Key 防止创建两个订单。

**大白话：** 好的 API 像好的餐厅菜单——菜名清晰（资源命名）、分页合理（一页别放 100 道菜）、点了两次同一道菜不会上两份（幂等）。

**核心启示：** API 设计不是"选 GET 还是 POST"——是**让调用方不需要读文档就能猜出怎么用。** 一致性比灵活性更重要。

## Q75 · 软件基准测试为什么常不准？

JVM 预热：前几次跑得慢（JIT 还没编译）。GC 暂停：测量期间发生 GC 引入长暂停。死代码消除：JIT 测出结果被丢弃 → 直接删掉整段代码 → 测出 0 纳秒（其实什么都没做）。

**正确做法：** 先预热足够多轮不记录 → 跑 N 轮取平均 → 确保计算结果被"消耗"（防止死代码消除）→ 用 JMH 而非手写循环。

**大白话：** 测跑步运动员速度——他刚起床、穿拖鞋、没热身（JVM 预热）→ 你得出"速度一般"。第二天充分休息、热身、穿跑鞋 → 完全不一样。**你测的不是"能力"，是"状态"。**

**核心启示：** 看基准测试报告时先问三个问题：**你控制了 JIT 预热吗？控制了 GC 暂停吗？确认了死代码没被消除吗？**

## Q76 · 需求文档为什么永远写不清楚？

不是"没写清楚"——是你大脑里有大量**你以为对方也知道的前提假设。** 敏捷的答案：不要试图一次性写清楚——用**频繁交付暴露假设。**

但敏捷 ≠ 没文档——接口契约、设计决策、状态机必须记录。

**大白话：** 你跟你妈说"买个西瓜回来"。她买回来——你说"不对，我要无籽、冰镇、切好的。"你没说清楚 ≠ 你没想清楚，而是"你以为对方知道的背景信息对方不一定知道"。

**核心启示：** 需求沟通的核心难题是**隐性知识——领域专家脑中有大量"不言自明"的假设。** 迭代式暴露假设比一次性文档穷举所有场景成本更低。

## Q77 · 可观测性三支柱——Metrics、Tracing、Logging

**Metrics：** 聚合数值（QPS、P99延迟）→ 宏观趋势和阈值告警。**Tracing：** 一次请求在分布式系统中的完整路径→ 定位瓶颈。**Logging：** 离散事件记录 → 事后排查和审计。

三者互补：Metrics 告诉你有问题 → Tracing 告诉你哪段 → Logging 告诉你具体什么。**缺一个都有盲区。**

**大白话：** 查案——Metrics = 警局统计看板（"本周入室盗窃 ↑15%"）→ 知道有问题。Tracing = 嫌疑人行动轨迹图 → 知道从哪进来的。Logging = 案发现场监控录像 → 知道具体怎么作案的。

**核心启示：** 可观测性不是买一套工具——是**让你的系统在设计时就考虑了"怎么被观测"。** 日志没有 TraceID = 查不了链路。Metrics 没有维度标签 = 只能看全局不能下钻。
