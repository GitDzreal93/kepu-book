# 第六章 | 分布式系统

> 当一台机器不够用时——CAP 的取舍、共识的协商、熔断的隔离、可观测性的全貌。分布式系统的根本挑战不是"让多台机器协同工作"，而是"在不可靠的网络上模拟可靠的系统"。

---

## Q78 · CAP定理——为什么银行选CP，朋友圈选AP？

CAP 不是说"三选二"——是**网络分区发生时，在 C（一致性）和 A（可用性）之间二选一。** P（分区容错）是分布式系统的前提——你无法假设网络永远可靠。

银行转账：宁可拒绝服务（牺牲 A），也不能余额不一致（保持 C）——选 CP。朋友圈：宁可短暂看到不一致（牺牲 C），也绝不能"加载失败"（保持 A）——选 AP。

**大白话：** 两本笔记本，两人各持一本。电话线断了（网络分区）→ A 收到一笔新订单但告诉不了 B。此时有人问 B"总数多少？"→ 为了保持数据一致，B 只能拒绝回答（牺牲 A）→ 或者 B 回答旧数据（牺牲 C）。**网络断的时候，你必须选一个。**

**核心启示：** 现代系统不在 C 和 A 之间二元选择——做细粒度取舍。银行核心交易选 CP、用户头像展示选 AP。

## Q79 · 设计分布式锁——Redis SETNX vs ZooKeeper临时节点

多个服务节点同时操作共享资源，只允许一个成功。

**Redis SETNX：** 设置锁 + 过期时间。死锁问题——业务没执行完锁过期了 → Redisson 看门狗自动续期。性能高但 CP 保证弱（Redis 主从切换可能丢锁）。

**ZooKeeper 临时顺序节点：** 所有请求者在同一路径下创建临时顺序节点 → 序号最小的获得锁 → 释放或断开连接自动删除。强一致（CP）但性能低于 Redis。

**大白话：** 公共厕所——Redis = 你进去锁门，门锁有个定时器（过期时间）。ZooKeeper = 门口取号排队，叫到你的号才进去。

**核心启示：** 选 Redis 还是 ZooKeeper = **你能容忍多大程度的"锁偶尔失效"。** Redis 快但可能丢锁（适合性能优先）；ZooKeeper 慢但绝对可靠（适合正确性优先）。

## Q80 · Raft共识算法——Leader挂了，新Leader怎么知道哪些日志已提交？

集群 5 节点。Leader 把一条日志复制到 3 个节点（多数）后挂了。新 Leader 选举出来后——**怎么知道这条日志该提交还是该丢弃？**

Raft 保证：新 Leader 的日志一定包含了所有已提交的日志。选举时，Candidate 必须获得多数票——多数节点的日志必然包含了所有已提交日志（因为已提交 = 已复制到多数节点）。

**日志恢复：** 新 Leader 把自己的日志强制复制给所有 Follower——Follower 的冲突日志被覆盖。一致性通过 **Term 号 + Log Index** 保证。

**大白话：** 议会。议长（Leader）提案，议员投票。议长挂了 → 新选举 → 得票过半者当新议长 → 新议长把自己手里所有提案（日志）同步给所有人。

**核心启示：** Raft 的核心不是"选举 Leader"——是**"确保任何时候，已提交的日志永远不会丢失"。** 这个保证来自多数派原则——任何两个多数派必有交集。

## Q81 · 容器 vs 虚拟机——不仅仅是"轻量vs重量"

**虚拟机：** Hypervisor 在硬件层虚拟——每 VM 运行完整 Guest OS（自己的内核、驱动）→ 强隔离，资源开销高（GB 级内存），启动分钟级。

**容器：** Linux Namespace + Cgroups——所有容器共享 Host OS 内核 → 轻量（MB 级额外内存），启停秒级，但隔离性弱（共享内核 → Host 内核漏洞影响所有容器）。

**互补而非替代：** 物理机上跑 VM 做强隔离（多租户云环境），VM 内部跑容器做高效编排。

**大白话：** VM = 同一块地上建 N 栋独立房子，每栋独立水电（完整 OS）。容器 = 同栋房子隔 N 个房间，共享水电（共享内核）。盖房间比盖房子快得多——代价是隔音差一点。

**核心启示：** 隔离层级决定了你能在什么粒度做"安全围栏"。**容器是"应用之间的隔离"，VM 是"租户之间的隔离"。**

## Q82 · 设计分布式消息系统——百万QPS写入、消息不丢、不乱序

Kafka 的答案：Producer 按 Key 哈希分区 → 分区内顺序写（append-only log）→ ISR（同步副本）确认 → Consumer 拉取自己管 offset。

**为什么 Kafka 不帮你管 offset？** 让你自己管——先处理后提交 = 至少一次（可能重复，但不会丢）。先提交后处理 = 至多一次（可能丢，不会重复）。**选你的可靠性模型。**

**大白话：** 邮局分拣流水线。信件到了就分拣按地址放（分区），按顺序处理（顺序写）。送信员自己记住送到哪封了（offset 管理）。

**核心启示：** 消息系统不能"替消费者做可靠性决策"——**只有消费者知道"什么时候算处理完"。** Kafka 提供工具，你定策略。

## Q83 · 设计海量日志实时搜索——Elasticsearch能做什么，不能做什么？

ES 的倒排索引——"关键词 → 包含它的文档列表"——O(1) 搜索。但写入峰值超过 indexing 吞吐时 → 背压 + 限流。时间序列数据 → ClickHouse（列存 + 压缩）比 ES 更高效。

**大白话：** ES = 图书馆卡片目录——按"作者""书名""关键词"建索引，"人工智能"→ 第 3、7、12 本书。但你今天入库 100 万本新书 → 建索引跟不上 → 新书暂时搜不到。

**核心启示：** 没有"最好的存储引擎"——只有**"在写入速度、查询延迟、存储成本三者之间的最优取舍"。**

## Q84 · 熔断器——一个服务崩了，怎么阻止全链路雪崩？

A → B → C。C 开始超时 → B 的线程池全被 C 占满 → B 也开始超时 → A 的线程池也被占满 → **全链路雪崩。**

熔断器：B 发现 C 连续失败 N 次 → 熔断 → 直接返回降级结果 → 保护 B 和 A 不被 C 拖下水。半开状态（Half-Open）的探针请求检测 C 是否恢复。

**大白话：** 多米诺骨牌。C 倒了 → B 等 C 等到自己的线程耗尽 → A 也倒了。熔断 = B 发现 C 倒了 → 主动切断对 C 的调用 → 对 A 说"暂时不可用"→ 保护自己和 A。

**核心启示：** 分布式系统中，**"快速失败"比"一直等待"更可靠。** 一个死的下游不要让所有上游陪葬。

## Q85 · 设计可横向扩展的实时计数器——微博热搜的阅读量

同一篇文章被千万人同时阅读——实时计数。Redis INCR 是单点瓶颈 → 分片计数器（按文章 + 时间窗口分片）→ 定期汇总。

**允许 1% 误差换 100 倍性能：** HyperLogLog 用极少量内存估算 UV 量级——误差可控。不是所有计数器都需要精确到个位数。

**大白话：** 不是每个人发"我看了"的消息——太吵。而是每个人在自己的记事本（分片计数器）上画"正"字，每 5 分钟汇总一次。热搜阅读量本来就不需要精确到个位——"1.2 亿"和"1.21 亿"差别不大。

**核心启示：** 大规模计数系统的本质是**用"近似准确"换"可以横向扩展"。** 不是所有场景都需要精确计数——**知道你可以牺牲什么，才知道你能扩展到哪里。**

## Q86 · K8s为什么能自愈——不是AI，是声明式API + 控制器循环

Deployment 期望 replicas=3。控制器持续循环：**Observe（观察实际）→ Diff（对比期望）→ Act（消除差异）。**

Pod 挂了 → 控制器发现 replicas=2（实际）≠ 3（期望）→ 创建新 Pod。这个循环永远在跑，不需要人工干预。

**大白话：** 管家。你说"我家要 3 个佣人"。管家每隔一秒巡视一圈——少了一个？立刻按模板雇一个。某个佣人站错位置了？纠正。你从来不需要告诉管家"怎么做"——只告诉**"我要什么状态"**。

**核心启示：** 声明式 API 的核心不是"自动化"——是**"你描述目标状态，系统自己找出到达路径"。** 从"命令"到"声明"的转变，是运维思想最深刻的变化。
