## Q70 · 技术债——什么债值得借，什么债绝对不能借？

![配图](images/chapter5/q70-tech-debt.png)

### 你有没有思考过这么一个问题？

你的团队为了赶一个上线 deadline，临时写了一段"能跑但设计很差"的代码——硬编码了配置参数、没有异常处理、重复代码复制了三遍。上线后老板很开心，你心里默念"以后再重构"。

三个月后，这段代码被三个人改过，每次改都引入新 bug。你想重构了，但老板说"新需求排满了，没有时间"。又过了半年，系统越来越难维护，加一个简单功能需要两天，而新来的同事看不懂这段代码。

你有没有想过：当初"借"的这笔技术债，值不值得？或者说——是不是有些技术债从一开始就不该借？

### 底层发生了什么？

**技术债（Technical Debt）** 是 Ward Cunningham 在 1992 年提出的概念。类比金融债务：你为了更快拿到钱（更快上线）而借了一笔贷款（写了简化的代码），但你必须支付利息（维护成本增加），最终还要还本金（重构）。

**技术债的四种类型：**

1. **有意为之的债（Intentional Debt）**：你知道当前设计不是最优的，但为了赶时间故意简化。你明确知道"这里应该用策略模式但暂时用了 if-else"，并且有还债计划。这是"良性债"。

2. **无意中产生的债（Unintentional Debt）**：你不知道当前设计有问题——因为经验不足或对需求理解不深。这不是"借债"，是"不小心欠的债"，通常通过 Code Review 和技术评审发现。

3. **不可避免的债（Inevitable Debt）**：随着业务发展，原本合理的设计变得不合理了。比如系统最初支持 1000 用户，现在要支持 100 万用户——当初的架构成了技术债。这不是谁的错，是"业务变了"。

4. **技术演进债（Tech Evolution Debt）**：框架、语言、工具升级了，旧版本不再维护。比如 Python 2 到 Python 3 的迁移——不迁移就是债。

**什么债值得借？**

- **验证 MVP（Minimum Viable Product，最小可行产品）**：不确定产品方向是否正确时，先用最快速度验证。如果方向不对，这段代码就会被扔掉，不值得投入做完美设计。
- **非核心业务**：一个临时活动页面、一次性数据导入脚本——用完就扔，不需要长期维护。
- **明确知道后期会重构**：比如用 SQLite 做开发环境数据库，上线后切 PostgreSQL。这是有计划的过渡方案。

**什么债绝对不能借？**

- **支付核心逻辑**：支付金额计算、对账逻辑——这里的 bug 直接导致资金损失。一个"临时"的金额计算方式可能导致对不上账、重复扣款。修复成本是天文数字——你需要追溯每一笔交易。
- **安全相关**：密码存储、权限校验、数据加密——这些地方"偷懒"等于给攻击者开后门。而且安全漏洞的影响是持续的——数据泄露后影响持续数年。
- **数据库 Schema 设计**：表结构设计不好，后续所有业务代码都要适配这个烂结构。修改 Schema 需要数据迁移，风险高、耗时长。这是"基础债"——它会让上层所有代码都变差。
- **API 契约设计**：一旦 API 发布了，调用方就依赖了当前的接口。后续修改 API 需要版本兼容，成本极高。烂 API 设计会持续折磨所有调用方。

**技术债的管理——把债可视化**

技术债和金融债一样——最大的风险不是"债多"，而是"不知道欠了多少债"。

- **度量工具**：SonarQube、Code Climate 可以量化代码复杂度、重复率、技术债规模。每个技术债项对应一个预估的"还债时间"。
- **分配还债资源**：每个 Sprint（迭代周期）分配固定比例（如 20%）的时间专门处理技术债。这就像每月还房贷——不能等到"有空了再还"，因为永远不会有空。
- **技术债登记簿**：维护一个文档或 Issue 列表，记录每笔技术债：是什么、为什么借、预计什么时候还、如果不还会怎样。这让团队对技术债有共识。

**技术债 vs 烂代码——关键区别**

| | 技术债 | 烂代码 |
|---|---|---|
| 是否有意识 | 有——知道是简化方案 | 无——不知道有问题 |
| 是否有计划 | 有——有还债计划 | 无——没人管 |
| 是否可管理 | 是——有度量和登记 | 否——散落各处 |

### 大白话解释

开一家餐厅——开业第一周，菜单手写在纸上（技术债），你知道该做点餐系统，但现在做意味着推迟开业一个月。先开张，边赚边补。这是"良性债"——你有意识、有计划。

但食品安全（支付安全）绝对不能马虎——不能说"先用过期食材开业，赚到钱再买好的"。这个债不能借，因为一旦出事，餐厅直接关门。

最大的风险是：你以为只借了一笔债，实际上不知不觉欠了五十笔。就像刷信用卡——每笔都不多，但月底一看账单吓一跳。技术债度量工具就是你的"信用卡账单"——让你看清楚到底欠了多少。

### 核心启示

技术债不是"烂代码"——是**"为了更快上线故意简化了设计，后面要重构"。** 区别在于：你知道这是债，你有还债计划。烂代码是不知道它有问题。管理技术债的核心不是"消灭所有债"——而是"把债可视化、有计划地还、知道哪些债绝对不能借"。

## Q71 · 设计监控告警系统——百万台服务器，怎么不告警风暴？

![配图](images/chapter5/q71-monitoring-alert.png)

### 你有没有思考过这么一个问题？

你有 1000 台服务器，每台装了 20 个监控项（CPU、内存、磁盘、网络、应用指标……）。某天，一个核心交换机故障了——所有经过它的服务器瞬间网络不通。你的告警系统在 30 秒内发出了 20000 条告警短信。

你的手机震了 5 分钟没停。你打开告警平台一看——20000 条告警，全是"服务器不可达"。你根本看不出是交换机的问题——你以为 20000 台服务器同时出了问题。真正有价值的告警（"交换机 X 端口 down"）被淹没在 20000 条噪声里。

你有没有想过：为什么不能只发一条告警——"交换机 X 故障，影响 N 台服务器"？或者说，告警系统应该怎么做才能让人"看一眼就知道发生了什么、影响多大"？

### 底层发生了什么？

**监控告警系统的三层架构：**

1. **数据采集层**：每台服务器运行 Agent（如 Prometheus Node Exporter、Telegraf），采集指标（CPU、内存、磁盘 IO、网络流量）并上报。采集频率从 10 秒到 1 分钟不等。
2. **数据存储层**：时序数据库（Time Series Database，TSDB）存储指标数据。如 Prometheus、InfluxDB。时序数据库优化了"按时间范围查询大量时间序列"的场景。
3. **告警引擎层**：基于规则和阈值判断是否触发告警。如 Alertmanager、Grafana Alerting。

**告警风暴（Alert Storm）的成因：**

告警风暴是指短时间内大量告警同时触发。常见原因：

- **根因传播**：一个底层组件（交换机、DNS 服务器、数据库主节点）故障，导致依赖它的所有服务报错——每个服务都触发告警。1000 台服务器依赖同一个交换机，交换机挂了 = 1000 条告警。
- **阈值过于敏感**：CPU > 80% 就告警，但批量任务时 CPU 短暂飙升是正常的。频繁误报让团队"告警疲劳"——真正出问题时反而被忽略。
- **没有告警抑制**：告警 A 和告警 B 是同一个根因，但系统不知道它们相关，分别发了两条。

**防止告警风暴的核心策略：**

1. **告警聚合（Alert Grouping / Deduplication）**：相同根因的告警合并成一条。Alertmanager 支持按标签（label）聚合——所有 `instance=server-{1..1000}` 的告警合并为一条"1000 台服务器受影响"。
2. **告警抑制（Inhibition）**：当高级别告警触发时，抑制低级别告警。交换机 down 了 → 抑制所有"服务器不可达"的告警，只保留交换机本身的告警。配置规则：`if alertname=SwitchDown then inhibit alertname=ServerUnreachable`。
3. **延迟告警（Pending Duration）**：告警条件满足后不立刻发，等 2-3 分钟。如果 2 分钟内恢复正常，就不发了。这能过滤掉短暂的抖动（网络抖动、GC 暂停导致的瞬时延迟飙升）。
4. **分级告警**：
   - P0（电话告警）：核心服务完全不可用——直接打电话叫人
   - P1（短信告警）：核心服务降级——发短信
   - P2（IM 告警）：非核心服务异常——发飞书/钉钉消息
   - P3（邮件告警）：容量预警——发邮件，不紧急
5. **告警路由（Routing）**：不同告警发给不同人。数据库告警发给 DBA，网络告警发给网络组。不要把所有告警发给所有人。

**告警质量指标：**

- **信噪比（Signal-to-Noise Ratio）**：有效告警 ÷ 总告警。低于 30% 说明大量告警是噪声。
- **MTTR（Mean Time To Repair，平均修复时间）**：从告警触发到问题解决的时间。衡量告警是否帮到了快速定位。
- **告警确认率**：被人工确认"是真问题"的告警比例。如果只有 10% 是真问题，说明阈值需要调。

### 大白话解释

想象一个城市的 110 报警中心：

如果某条街变压器爆炸导致停电——那条街的 500 户居民同时打电话报警。接线员接了 500 个电话，每个都在说"我家停电了"——但没人知道是变压器的问题。这就是告警风暴。

好的报警系统应该：500 户停电 → 系统自动判断"同一区域同时停电=大概率基础设施故障" → 合并成一条"XX 街区变压器故障，影响 500 户" → 派供电局去修。而不是让 500 户分别打电话。

### 核心启示

监控告警系统的核心不是"多发现问题"——是**"让人看一眼就知道发生了什么、影响多大、该谁处理"。** 告警风暴不是系统太灵敏——是告警系统没有做聚合和抑制。好的告警系统：每个告警都是 actionable（可操作的）——你知道该干什么；没有冗余——同一个问题不会被反复告。如果告警太多大家开始忽略——不如不发。

## Q72 · 大厂为什么造轮子？

![配图](images/chapter5/q72-build-vs-buy.png)

### 你有没有思考过这么一个问题？

市面上已经有 Kafka 了——成熟、稳定、有大量文档和社区支持。但某大厂还是花了一个团队两年时间自研了一套消息队列。你会不会觉得"有病？用现成的不香吗？"

但如果你知道他们的场景：每秒百万级消息写入、跨地域复制延迟必须小于 50ms、需要精确到消息级别的顺序保证——而 Kafka 的存储模型在这个规模下变成了瓶颈。Kafka 的分区（Partition）数量超过一定数量后，Controller 节点的元数据管理变成性能瓶颈。他们试过调优、试过 patch，但问题出在架构设计层面，不是调参数能解决的。

你有没有想过：在什么条件下"自研"比"用开源"更合理？这个决策的经济学是什么？

### 底层发生了什么？

"造轮子"（Reinventing the Wheel）——用自研系统替代成熟的开源方案。在工程界通常是贬义的，但在大厂特定场景下，它是理性决策。

**造轮子的理性条件——满足任一条：**

1. **差异化核心业务**：如果你是做消息队列的公司（如 RocketMQ 之于阿里），消息队列就是你的核心竞争力——自研是战略必需，不是技术选择。核心业务的系统控制权不能掌握在第三方手里。

2. **规模超过现有方案极限**：开源方案的通用设计在某一个维度上撞到了天花板。比如 Google 之所以自研 Borg（K8s 的前身）——因为他们的集群规模（万台机器）远超当时所有集群管理系统的设计极限。通用方案在极端规模下不再通用。

3. **外部方案无法满足合规要求**：数据必须留在自有机房、安全审计需要完全控制源码。某些行业的合规要求直接排除了使用外部 SaaS 服务的可能。

4. **特殊需求组合**：你的需求是多个不常见需求的组合——比如"低延迟 + 高吞吐 + 消息顺序保证 + 跨地域复制"。开源方案可能满足三个但第四个不行。打补丁的成本可能超过自研。

**造轮子的成本——不只是开发成本：**

- **开发成本**：一个团队 1-2 年的开发投入。但这是"一次性"的。
- **维护成本**：这是持续的、长期的、被严重低估的成本。自研系统需要持续修 bug、适配新硬件、处理边缘场景、升级依赖。开源方案有社区帮忙做这些——自研系统全部自己来。
- **学习成本**：新员工需要学习自研系统的使用方式。开源方案有文档、教程、社区问答；自研系统只有内部文档和"问那个写了这个系统的人"（如果她还没离职的话）。
- **技术锁定成本**：自研系统的 API 和开源方案不同，迁移到其他方案需要大量适配工作。

**不造轮子的理性条件——满足任一条：**

1. **非核心业务**：日志收集、监控告警、CI/CD——这些不是你的核心竞争力，用开源方案就够了。
2. **规模未到瓶颈**：你的 QPS 是 1 万，开源方案的设计上限是 100 万——没有理由自研。
3. **成熟方案满足需求**：如果你需要的只是一个"能跑的"消息队列，Kafka/RabbitMQ/RocketMQ 都行——自研的价值为零。

**大厂开源自己的轮子——另一个逻辑**

为什么 Google 开源 K8s、Facebook 开源 React、阿里开源 RocketMQ？

- **降低维护成本**：开源后，外部贡献者帮忙修 bug、加功能、写文档。维护成本从"一个团队"变成"一个社区"。
- **建立生态标准**：如果你的系统成为行业标准（如 K8s 成为容器编排标准），你就拥有了话语权。人才会主动学习你的技术，招聘成本降低。
- **吸引人才**：工程师愿意去"创造开源项目"的公司工作。开源是一种雇主品牌建设。

### 大白话解释

开一家餐馆——大众点评给你现成的点餐系统，80% 吻合需求。但你想加"拼桌功能"——做不到。自研 = 完全定制但需要投入两个工程师半年。自研的前提是：拼桌功能是你核心竞争力的一部分，或者大众点评的系统在你这个规模下扛不住了。

但如果你只是想加一个"积分功能"——用大众点评就够了，不值得为此自研整个系统。

大厂自研后开源就像：你做了一套超好的点餐系统，开源给所有餐馆用。好处：其他餐馆帮你发现 bug、帮你加功能、帮你写教程。你不需要养一个团队维护——社区帮你维护。而且你成了行业标准——以后所有餐馆都按你的标准来做。

### 核心启示

造轮子不是技术决策——是**"定制化收益"vs"维护成本"的经济决策。** 满足四个条件之一才值得自研：差异化核心、规模超极限、合规限制、特殊需求组合。否则用成熟开源方案，把精力花在业务逻辑上。造轮子前先问自己：这个轮子是"我的核心竞争力"还是"我重新发明了一个已经存在的轮子"？

## Q73 · 单体 vs 微服务——10人20个微服务是找罪受

![配图](images/chapter5/q73-monolith-vs-microservice.png)

### 你有没有思考过这么一个问题？

一个 10 人的技术团队，把系统拆成了 20 个微服务。每次改一个需求需要动 4 个服务。排查一个线上问题需要翻 5 个服务的日志。服务间网络通信的延迟占了整个请求时间的三分之一。新人入职两周还在学习"这个功能在哪个服务里"。

你有没有想过：他们为什么拆这么多？或者说——拆成微服务到底解决了什么问题，又带来了什么新问题？

### 底层发生了什么？

**单体架构（Monolith）**：所有代码在一个仓库、一个进程里运行。模块之间通过函数调用通信。

- **优点**：部署简单（一个包）、开发方便（一个 IDE 就能看所有代码）、调用快（函数调用，纳秒级）、排查简单（一个进程的日志）。
- **缺点**：代码膨胀后编译变慢（10 万行代码编译 5 分钟）、一个模块的 bug 可能拖垮整个进程（内存泄漏影响所有功能）、无法独立扩容（下单和搜索不能分别扩容）。

**微服务架构（Microservices）**：系统按业务边界拆成多个独立部署的服务。服务之间通过网络通信（HTTP、gRPC、消息队列）。

- **优点**：独立部署（改订单服务不需要重新发布用户服务）、独立扩容（秒杀时只扩容订单服务）、技术栈灵活（搜索用 Go、AI 用 Python）、故障隔离（一个服务挂了不影响其他服务）。
- **缺点**：运维复杂度爆炸（20 个服务 = 20 套部署 + 监控 + 告警 + 日志）、网络通信开销（每次调用都是网络请求）、分布式事务困难（跨服务的数据一致性）、调试困难（一个请求经过 5 个服务，日志分散在 5 个地方）。

**微服务把"代码的复杂度"转移成了"运维的复杂度"**——这是核心 insight。你在代码层面省的事，在运维层面加倍还了回来。

**什么时候该拆微服务？**

判断标准：**拆的收益 > 拆的代价**。

- **拆的收益**：独立扩容需求明确、团队规模足够大（每个服务至少 3-5 人维护）、不同模块的技术栈差异大。
- **拆的代价**：运维基础设施成本（服务发现、配置中心、链路追踪、容器编排）、团队沟通成本（接口定义、版本管理）、分布式系统的复杂性（网络分区、最终一致性）。

**康威定律（Conway's Law）**：系统设计必然反映组织的沟通结构。如果你的组织是 3 个团队，你拆 20 个服务——每次改动需要 3 个团队协调 20 个服务的部署，这比单体还痛苦。正确做法是先拆团队，再拆服务——一个团队对应一个或几个服务。

**模块化单体（Modular Monolith）——中间地带**

在团队规模不够大时，不需要拆成微服务。在单体内部做模块化：每个模块有明确的边界和接口，模块间通过定义好的接口通信（不是直接调函数），但没有网络开销。

- 源码级模块隔离（不同的 package / module）
- 接口契约通过 Interface 定义
- 需要拆微服务时，把模块抽出来变成独立服务——接口不变，只需要从函数调用变成网络调用

这是"留好了拆的路，但先不拆"的策略。Rails 社区的 Shopify 就是模块化单体的成功案例——他们在单体里跑了几千个工程师的工作。

**微服务的拆分原则：**

1. **按业务能力拆**，不按技术层拆。不要拆成"用户数据访问服务 + 订单数据访问服务"——应该拆成"用户服务（含数据访问）+ 订单服务（含数据访问）"。
2. **一个服务一个数据存储**。不要多个服务共享一个数据库——数据库耦合是最强的耦合。
3. **服务间通过 API 通信**，不共享数据库、不共享内存。
4. **先粗后细**。一开始拆成 3-5 个粗粒度服务，而不是 20 个细粒度服务。需要时再细拆。

### 大白话解释

微服务 = 把一栋大房子拆成 N 个小房间。每个房间各管各的（独立部署、独立扩容），但房间之间需要通过走廊发消息（网络通信），还需要额外的管委会（服务治理——服务发现、配置中心、链路追踪、容器编排）。

10 个人住 20 个小房间 = 管委会比住户还多。你花在管理房间上的时间比住在房间里做事情的时间还多。

模块化单体 = 还是住在一栋大房子里，但每个房间有门——需要拆的时候把门外的走廊接出去就行。先在一栋房子里住好，等真的住不下了再拆。

### 核心启示

康威定律说系统设计反映组织沟通结构——**先拆团队，再拆服务。** 10 人团队用模块化单体足够；50+ 人才有拆微服务的必要。拆微服务的判断标准不是"系统太大"——是"多个团队同时开发的冲突频率太高"。如果没有人因为改同一个文件而冲突，就不要拆。
