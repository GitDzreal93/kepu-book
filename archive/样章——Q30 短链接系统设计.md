# 设计一个短链接系统（像 t.cn）

## Q30 · 样章 — Version 1 — ready for review

### 你有没有真正想清楚过？

面试官说："设计一个短链接系统，像 t.cn 那样。用户给一个长 URL，你返回一个短链接。其他人点短链接，重定向回原 URL。"

你脑子里可能立刻跳出了几个关键词：哈希、Base62、Redis。你觉得这不难——把长 URL 哈希一下取前七位，存进数据库，重定向的时候查出来，301 跳转。五分钟就能写完 Demo。

然后面试官补充了三句话：

"每天生成 100 万条短链接。"  
"重定向延迟不超过 10 毫秒。"  
"短码全局唯一，永远不能冲突。"

你会发现你把"Demo"和"系统"搞混了。Demo 解决的是"能不能跑通"。系统要解决的是"跑通之后的每一天，会发生什么"。

### 底层发生了什么？

短链接系统表面上是"把长变短"。实际上，你要回答的是四个层层递进的问题。

**第一问：短码怎么生成？**

这是最核心的决策，而且三秒钟就能把面试官分成两类：觉得"哈希一下就行"的新手，和知道"哈希会碰撞"的老手。

方案 A：对长 URL 做 MD5（取前 7 位）。听起来完美——同一个长 URL 永远生成同一个短码，天然去重。但你算一下：7 位 Base62 字符能表示 62^7 ≈ 3.5 万亿种组合，似乎很大。然而 MD5 的 128 位被截断到 7 个字符（约 41 位），碰撞概率遵���生日悖论——当你生成了大约 2^20.5 ≈ 150 万条时，碰撞概率就超过 50%。你每天生 100 万条，撞是必然的。撞了怎么办？加一位变 8 位？那昨天生成的 7 位链接全废了。重新 MD5 加盐？那同一个长 URL 会生成不同短码——去重没了。

方案 B：用一个自增 ID（比如数据库的自增主键或 Redis 的 INCR），把数字转成 62 进制字符串。1 → "1"，10 → "a"，62 → "10"。这个方案完全没有碰撞——自增 ID 天然唯一。但它有两个新问题。第一，你可以遍历。如果你生成了一条短链接 id=1000，把 62 进制反转一下就能算出上一个短码是什么——所有短链接都被你"枚举"出来了，安全性为零。第二，自增 ID 是单点。如果 ID 生成器挂了，整个系统停摆。

方案 C：把方案 B 修一下——不让 ID 自增，而是随机分布但保证唯一。比如用 Snowflake（分布式 ID 生成算法，64 位中 41 位是毫秒时间戳）再把 64 位数字转成 Base62——得到一个 11 位的短码。或者更简单：预生成一个 ID 号段池——从数据库批量取 1000 个 ID 缓存在内存里，用完了再取下一批。这样即使数据库挂了也能撑一阵子。至于"可遍历"的问题，最简单的解法是：短码不是原始 ID 直接编码，而是对一个随机种子做 XOR 后再编码——外人看到的短码是"随机"的，内部可以还原回真实 ID。

**第二问：长链接和短链接的映射关系怎么存？**

注意，这是一个**双向映射**。顺着——"你给我短码，我找到长 URL"——这是重定向的核心路径，必须极快。逆着——"你给我长 URL，我查它有没有对应的短码"——这是去重的路径，可以稍慢一些。

顺着查是主路径：用 KV 存储。短码是 Key，长 URL 是 Value。Redis 一个 O(1) 的 GET 请求在 0.1ms 内完成。但你不能只靠 Redis——它贵，而且数据不持久化的话重启就没了。MySQL 是你的持久层——如果 Redis 没命中，再查 MySQL，查到后回写 Redis。

逆着查是去重路径：同一个长 URL 不应该生成两个不同的短码（浪费资源 + 统计困难）。在 MySQL 里对长 URL 建一个哈希索引（`long_url_hash CHAR(32)`），用 MD5 或 SHA-256 存长 URL 的摘要——比给一个可能几 KB 的 URL 直接建索引更高效。MySQL 的 InnoDB 单表在索引合理的情况下轻松扛住千万级数据。

**第三问：重定向怎么做到 <10ms？**

302 临时重定向。为什么是 302 而不是 301？因为 301 是"永久移动"——浏览器会记住这个映射，下次用户访问短链接时浏览器直接跳转，根本不经过你的服务器。听起来很好？但这对统计是灾难——你永远不知道这个链接被访问了多少次。302 是"临时跳转"，浏览器每次都会先找你确认，你可以在跳转前记录一次点击。

现在看延迟。一次 Redis GET ≈ 0.1ms。HTTP 响应 ≈ 1ms。网络往返 ≈ 取决于用户到服务器的距离。如果你的服务器在北京，用户在纽约——光飞行时间就要 40ms，你的 <10ms 直接破产。解法是 CDN：把重定向逻辑推到边缘节点。用户点短链接→请求到 CDN 边缘→边缘节点内部重定向→用户直接跳转。边缘到用户的延迟可以控制在 5ms 以内。CDN 的缓存策略：热门短链接的映射关系缓存在边缘节点，冷门的回源查你的服务器。

还有一个加速手段：布隆过滤器。在服务器入口放一个布隆——如果短码在布隆里说"一定不存在"（布隆的假阳性是"可能存在"但实际不存在，但它的正确判断是"绝对不存在"），直接返回 404，连 Redis 都不用查。用极小的内存挡住大量无效请求。

**第四问：怎么防滥用？**

如果任何人都能用你的 API 生成无限短链接——三件事会发生：一是你的存储被垃圾链接撑爆；二是有人把恶意链接藏在短链接里传播（钓鱼、木马）；三是有人遍历你的短码，找到敏感的未公开链接。

第一层：频率限制。同一 IP 每分钟最多生成 N 条。用 Redis 的滑动窗口计数器——对每个 IP，记录过去一分钟的请求次数，超过阈值返回 429。这个在 API 网关层做。

第二层：恶意链接检测。在生成短链接之前，对长 URL 做一次安全检查——调用 Google Safe Browsing API 或者自己维护的黑名单。这一步会加几十毫秒的延迟，但可以在用户提交请求后异步完成——先生成短链接，后台异步检查。如果发现恶意，直接禁掉这个短码。

第三层：防止枚举。短码不可预测（用随机分布的 ID + XOR 扰动）。即使攻击者知道你用了 Base62 编码，他也无法从一条短码推导出其他短码。

### 怎么讲给别人听？

如果你要跟产品经理解释、或者给你爸解释为什么"短链接系统"不只是一个"压缩工具"——

可以把它类比成一个**图书馆的索书号系统**。图书馆里的一本大部头参考书（长 URL）放在某个书架上，它的书名可能很长——《中国近代经济史 1840-1949 第三卷修订版》。索书号就是给它分配的唯一编号——比如"F129.5/003"。读者拿索书号（短码）去书架找书（重定向），一秒就能找到。

但这个系统的难点不是"把书名变成编号"——任何一个编号方案都能做。真正的难点是：①每天要进 100 万本新书，编号绝对不能重复；②读者拿着编号来"取书"，必须立刻拿到，不能排队等；③编号不能按顺序来——有人看到编号"002"就能猜到"003"对应哪本书（可能是一本未公开的机密报告），这是安全漏洞；④总有人试图在书上夹带小广告（恶意链接），图书管理员必须在书入库前检查。

所以短链接系统的本质是：**一个高并发、高可靠性、有安全防线的小型分布式系统——它刚好披了一件"压缩URL"的外衣。**

### 核心洞察

> 把这个系统拆到底，它真正教会我们三件事：
>
> **1. "看起来简单"的系统，约束一收紧就暴露复杂度。** 如果不考虑日生成量、延迟要求、唯一性，短链接是一个 Demo。加上这些约束，它在 ID 生成、缓存策略、安全防护三个维度都要做深度设计。面试官不是在看你能不能写出一个能跑的短链接，而是在看你面对这些约束时的取舍逻辑。
>
> **2. 选哈希还是选自增 ID，本质是"去中心化的不确定性"vs"集中式的确定性"的权衡。** 哈希没有中心点，但碰撞是概率事件；自增 ID 确定无碰撞，但引入了单点。一个好的方案会走向分布式 ID（Snowflake）或号段模式——在确定性上做去中心化的改造。
>
> **3. 缓存不是"加一层就快点"。** Redis 要不要存全量？全量太贵。要不要设 TTL？太短命中率低，太长数据更新不及时。CDN 边缘缓存什么策略？热门链接才缓存，冷门链接回源。每一步缓存决策的背后都是"成本和命中率的算术题"——没有银弹，只有具体场景下的算账。

---

**面试备用论点：**
- 短码生成最优解：分布式 ID（Snowflake/号段）→ Base62 编码 → XOR 扰动防枚举
- 存储：Redis（热数据 O(1) 读）+ MySQL（持久化 + 长 URL 哈希索引用去重）
- 重定向 <10ms：CDN 边缘节点 + 302 临时重定向 + 布隆过滤器拦截无效请求
- 防滥用：IP 频率限制（网关层）→ 异步恶意链接检测 → 不可预测短码防枚举
