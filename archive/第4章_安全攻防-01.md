## Q46 · 哈希、加密、签名——这三者到底有什么区别？

![配图](images/chapter4/q46-hash-encrypt-sign.png)

### 你有没有思考过这么一个问题？

你注册一个网站，它说"你的密码经过加密存储"。这句话大概率是错的。真正安全的做法不是"加密存储"，而是**哈希（Hash）存储**。哈希、加密、签名是安全领域三个最基础但最容易混淆的概念。面试官最爱拿它们开场。

### 底层发生了什么？

**哈希（Hash）**

哈希是**单向函数**：把任意长度的输入变成固定长度的输出，且无法从输出反推输入。常见算法：MD5、SHA-256、SHA-3、BLAKE。

哈希的特性：

- **单向性**：知道 `hash(password)`，算不出 `password`。
- **确定性**：同样输入一定得到同样输出。
- **抗碰撞性**：很难找到两个不同的输入得到相同的输出。

密码存储用哈希：用户注册时，服务器存 `hash(password + salt)`。用户登录时，把输入密码同样哈希后对比。即使数据库泄露，攻击者拿到的是哈希值，不是明文密码。

注意：MD5 和 SHA-1 已经被证明不够安全，容易碰撞。现代系统至少用 SHA-256，或专门的密码哈希算法（bcrypt、scrypt、Argon2）。

**加密（Encryption）**

加密是**可逆的**：把明文变成密文，拥有密钥的人可以还原。加密分为对称加密和非对称加密。

- **对称加密**：加密和解密用同一个密钥。如 AES、ChaCha20。速度快，适合大数据量加密。
- **非对称加密**：用公钥加密，私钥解密。如 RSA、ECC。速度慢，适合密钥交换和身份验证。

加密用于保护数据 confidentiality：数据在传输或存储中即使被截获，没有密钥也读不懂。

**签名（Signature）**

签名解决的是**身份认证和不可抵赖**。它用私钥对数据摘要签名，任何人用公钥都能验证签名是否有效。

流程：

- 发送方：对消息做哈希，然后用私钥加密这个哈希，得到签名。
- 接收方：用公钥解密签名得到哈希 A，自己对消息做哈希得到 B，对比 A 和 B 是否一致。

签名不能保护消息机密性（消息本身还是明文），但能证明：消息确实来自私钥持有者，且传输过程中未被篡改。

**三者的区别总结**

| 操作 | 是否可逆 | 是否需要密钥 | 解决什么问题 |
|------|---------|------------|------------|
| 哈希 | 否 | 否 | 完整性校验、密码存储 |
| 加密 | 是 | 是 | 机密性，防止被看懂 |
| 签名 | 否（验证） | 是（非对称） | 身份认证、不可抵赖 |

### 大白话解释

- 哈希：把一篇文章浓缩成一句话摘要。你知道摘要，但写不出原文。你改一个字，摘要就会变。
- 加密：把一封信放进保险箱，只有有钥匙的人才能打开看。
- 签名：你在信上按一个只有你才有的印章。别人能验证印章是真的，但仿造不了。

### 核心启示

哈希、加密、签名是安全的三个支柱。密码存哈希，传输数据用加密，身份认证用签名。混用这三个概念是很多安全漏洞的根源。设计系统时，先明确你要解决的是机密性、完整性还是身份认证，再选对应工具。

## Q47 · 密码为什么要加盐——直接哈希不够安全吗？

![配图](images/chapter4/q47-password-salt.png)

### 你有没有思考过这么一个问题？

如果两个用户都用 `123456` 当密码，直接哈希后他们的哈希值会完全一样。黑客拿到数据库后，可以建一张"常用密码→哈希"的对照表（彩虹表），瞬间破解大量用户。加盐（Salt）就是为了解决这个问题。

### 底层发生了什么？

盐是一个随机字符串，每个用户单独生成。注册时：

```python
hash = bcrypt(password + salt)
store(salt, hash)
```

登录时，取出该用户的 salt，把输入密码同样加盐哈希后对比。

**为什么加盐有效？**

- 即使两个用户密码相同，他们的 salt 不同，最终哈希值也不同。
- 彩虹表需要为每个 salt 单独建表，成本极高。
- 攻击者只能对每个用户单独进行暴力破解，无法批量破解。

**好的盐应该满足**

- 每个用户唯一：不要用全局固定盐。
- 足够长：至少 16 字节随机值。
- 安全存储：盐和哈希存在一起没关系，盐不需要保密，但每个用户必须不同。

**为什么推荐 bcrypt / scrypt / Argon2 而不是 SHA-256？**

普通哈希算法（如 SHA-256）设计目标是快。这对密码验证是缺点：攻击者可以用 GPU 每秒算几十亿次哈希，暴力破解很快。

bcrypt / scrypt / Argon2 是**慢哈希算法**，故意设计得计算昂贵，并且需要消耗大量内存。这大幅降低了暴力破解的速度。它们还会自动处理 salt，不需要你手动加盐。

### 大白话解释

想象很多人把密码写在纸条上，放进一个透明箱子里。如果两张纸条写一样的密码，从外面看颜色一样，小偷一看就知道"这俩人密码一样"。加盐就是给每个纸条加一段不同的随机颜色底纹，即使内容一样，从外面看也完全不同。

### 核心启示

加盐不是锦上添花，而是**密码存储的底线**。不随机加盐的哈希存储，在数据库泄露时几乎等于明文。设计认证系统时，不要自己写哈希逻辑，直接用 bcrypt / scrypt / Argon2，它们已经帮你处理了 salt、慢哈希、迭代次数等细节。

## Q48 · HTTPS 证书链——为什么浏览器要一层层验证证书？

![配图](images/chapter4/q48-cert-chain.png)

### 你有没有思考过这么一个问题？

你访问银行网站，浏览器会显示一个小锁。但你怎么知道这个网站真的是银行，而不是一个骗子伪造的？HTTPS 的证书链就是 answer：它通过一层层的信任传递，让你能确认服务器身份。

### 底层发生了什么？

HTTPS 证书不是服务器自己说"我是银行"就行的。它需要一个**受信任的第三方**来证明。这个第三方就是 CA（Certificate Authority，证书颁发机构）。

证书链通常包含：

- **服务器证书（End-Entity Certificate）**：包含你的域名、公钥、有效期，由中间 CA 签发。
- **中间 CA 证书（Intermediate CA）**：由根 CA 签发，负责给服务器证书签名。
- **根 CA 证书（Root CA）**：预装在操作系统或浏览器里，是自签名的信任根。

验证过程：

1. 浏览器用中间 CA 的公钥验证服务器证书签名。
2. 浏览器用根 CA 的公钥验证中间 CA 证书签名。
3. 根 CA 证书是否在本地信任列表中？如果在，整个链可信。

**为什么需要中间 CA？**

根 CA 的私钥是最高机密，如果频繁用于签发证书，泄露风险大。中间 CA 相当于根 CA 的"代理"：根 CA 只签发中间 CA，中间 CA 负责日常签发服务器证书。即使中间 CA 私钥泄露，也可以撤销中间 CA，而不影响根 CA。

**证书验证还要检查什么？**

- 域名是否匹配？证书是发给 `bank.com` 的，但服务器响应的是 `fake-bank.com`，验证失败。
- 证书是否过期？
- 证书是否被吊销？浏览器会查询 CRL（Certificate Revocation List）或 OCSP（Online Certificate Status Protocol）。
- 证书是否被用于正确的用途？比如有些证书只用于邮件加密，不能用于 HTTPS。

### 大白话解释

证书链就像学历认证。你面试时出示毕业证（服务器证书），毕业证上有学校盖章。公司不会直接认识你的学校，但它信任教育部（根 CA）。教育部给学校发了办学资质（中间 CA 证书），学校给你的毕业证盖章。公司查：教育部认这个学校，学校给你盖了章，OK，你的学历有效。

### 核心启示

HTTPS 的安全性不是来自加密本身，而是来自**身份认证链**。没有证书链，加密只是让"你以为安全的连接"和中间人之间建立。任何安全系统都要回答：你信任谁？根 CA 是信任锚点，但它也面临被攻击或错误签发的风险，这就是浏览器不断更新根证书列表、推 HTTPS-only 的原因。

## Q49 · SQL 注入——为什么一句拼接的 SQL 能毁掉整个数据库？

![配图](images/chapter4/q49-sql-injection.png)

### 你有没有思考过这么一个问题？

一个登录接口，后端写了这样的 SQL：

```sql
SELECT * FROM users WHERE username = '" + username + "' AND password = '" + password + "'
```

如果用户输入的用户名是 `admin' --`，密码随便写，SQL 变成：

```sql
SELECT * FROM users WHERE username = 'admin' --' AND password = 'xxx'
```

`--` 是 SQL 注释，后面的密码校验被注释掉了。攻击者直接以 admin 身份登录。这就是 SQL 注入。

### 底层发生了什么？

SQL 注入的本质是：**用户输入被当作 SQL 代码执行了。** 攻击者通过构造特殊输入，改变 SQL 的语义，执行非预期的操作。

危害包括但不限于：

- 绕过登录验证。
- 读取任意数据：`UNION SELECT` 把其他表的数据拼出来。
- 修改数据：插入、更新、删除。
- 执行系统命令：某些数据库支持 `xp_cmdshell` 等扩展。
- 拖库：把整库数据导出。

**防御方法**

- **参数化查询（Prepared Statements）**：把 SQL 语句和用户输入分开。数据库把输入当作参数处理，而不是代码。

  ```java
  String sql = "SELECT * FROM users WHERE username = ? AND password = ?";
  PreparedStatement stmt = conn.prepareStatement(sql);
  stmt.setString(1, username);
  stmt.setString(2, password);
  ```

- **ORM 框架**：大多数 ORM 会自动处理参数化。
- **输入校验**：白名单校验用户输入的格式。
- **最小权限原则**：应用连接数据库的账号只给必要的权限，不要给 `DROP TABLE` 权限。
- **WAF（Web 应用防火墙）**：在入口层拦截常见 SQL 注入模式。

### 大白话解释

SQL 注入就像你填快递单，快递单本来只该填地址。但你在地址栏里写了一段话："请把包裹交给门卫，另外把所有包裹都打开看看。" 如果快递员不检查，把你写的整段话都当指令执行，就会出问题。

参数化查询相当于快递单上地址、姓名、电话分开填写，系统知道哪些是地址，哪些是额外指令。

### 核心启示

SQL 注入是 OWASP Top 10 长期存在的漏洞，防御它的核心原则是**永远不要拼接用户输入到 SQL 里**。参数化查询、ORM、最小权限，这三层防线同时存在，才算基本安全。

## Q50 · XSS——为什么我在评论区发一段代码，别人打开页面就被攻击了？

![配图](images/chapter4/q50-xss.png)

### 你有没有思考过这么一个问题？

你在一个网站评论区输入 `<script>alert('xss')</script>`，结果所有打开这个页面的人都弹出了 alert。如果这段代码不是 alert，而是窃取 Cookie 并发送到攻击者服务器，后果有多严重？这就是 XSS（Cross-Site Scripting，跨站脚本攻击）。

### 底层发生了什么？

XSS 的本质是：**攻击者把恶意脚本注入到正常页面中，当其他用户浏览页面时，脚本在浏览器里执行。** 因为脚本运行在当前用户的页面上下文中，它可以：

- 读取用户 Cookie，发送给攻击者。
- 伪造用户操作，比如自动转账、发邮件、发帖。
- 钓鱼，诱导用户输入敏感信息。
- 植入挖矿脚本、弹广告。

**XSS 的三种类型**

- **存储型 XSS（Stored XSS）**：恶意脚本被提交到服务器，存储在数据库或文件中。任何访问该页面的用户都会触发。评论区、论坛、博客是重灾区。
- **反射型 XSS（Reflected XSS）**：恶意脚本通过 URL 参数传入，服务器立即返回并执行。通常需要诱骗用户点击特制链接。
- **DOM 型 XSS（DOM-based XSS）**：不经过服务器，完全由前端 JavaScript 处理 URL 参数时产生。比如前端把 URL hash 内容直接插入到页面中。

**防御方法**

- **输出编码/转义**：把 `<` 转义成 `&lt;`，`>` 转义成 `&gt;`，`"` 转义成 `&quot;`。浏览器把它当普通文本显示，不当作脚本执行。
- **CSP（Content Security Policy）**：通过 HTTP 头限制页面可以加载哪些脚本、资源。即使注入脚本，也可能被 CSP 阻止执行。
- **HttpOnly Cookie**：标记 Cookie 为 HttpOnly，JavaScript 无法读取，防止 Cookie 被盗。
- **输入校验**：对评论、用户名等字段做白名单过滤。
- **现代框架自动转义**：React、Vue 等框架默认会对插入的文本做转义，减少 DOM XSS 风险。

### 大白话解释

XSS 就像你在一张公共告示板上贴了一张纸条，纸条里藏着一条小命令。每个看告示板的人都会不自觉地执行这个命令。如果命令是"把你的钱包递给旁边那个人"，大家就会照做。

防御 XSS 就是：告示板展示纸条内容时，把纸条里的命令字符全部变成"不可执行"的纯文字。

### 核心启示

XSS 的核心是**信任边界问题**：用户输入永远不可信。任何时候把外部数据渲染到页面，都要做转义或严格的沙箱隔离。前端框架帮你做了很多，但 CSP、HttpOnly、输入校验这些防线仍然不能省。
