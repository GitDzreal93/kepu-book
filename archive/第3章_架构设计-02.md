## Q36 · 缓存——为什么有了数据库，还要用 Redis 缓存？

![配图](images/chapter3/q36-cache.png)



### 你有没有思考过这么一个问题？

你打开淘宝首页，"猜你喜欢"的商品列表、用户头像、销量数据——这些数据如果每次打开都现查数据库，数据库分分钟被拖死。但用户看的又大部分是重复数据。所以工程师在中间加了一层 Redis 缓存：先查缓存，缓存没有才查数据库。

这个思路听起来简单，但面试里会追出很多细节：缓存和数据库怎么保持一致？缓存穿透、击穿、雪崩是什么？Cache Aside、Read Through、Write Through 模式有什么区别？

### 底层发生了什么？

**为什么缓存比数据库快？**

数据库的数据存在磁盘上，查询需要经过 SQL 解析、索引查找、磁盘 I/O、数据返回。Redis 是内存数据库，读写都是内存操作，延迟在微秒级。而且 Redis 数据结构简单（字符串、哈希、列表、集合、有序集合），针对特定场景做了大量优化。

**缓存的经典问题**

1. **缓存穿透（Cache Penetration）**：请求一个缓存和数据库都不存在的数据，每次请求都打到数据库。比如 id=-1 的查询。防御方法：布隆过滤器、空值缓存、参数校验。

2. **缓存击穿（Cache Breakdown）**：一个热点 key 突然过期，大量请求同时打到数据库。防御方法：互斥锁（只有一个线程去重建缓存）、热点 key 永不过期、逻辑过期。

3. **缓存雪崩（Cache Avalanche）**：大量缓存同时过期，数据库压力瞬间暴增。防御方法：过期时间加随机值、多级缓存、限流降级。

**缓存更新策略**

- **Cache Aside（旁路缓存）**：读的时候先查缓存，没有就查数据库并写回缓存；写的时候先更新数据库，再删缓存。这是互联网最常用的模式。
- **Read Through / Write Through**：读写都走缓存，缓存负责和数据库同步。逻辑更统一，但缓存组件需要更复杂。
- **Write Behind（异步写）**：写操作只更新缓存，由缓存异步批量写数据库。性能最高，但一致性和数据丢失风险也高。

**缓存一致性难题**

Cache Aside 也有不一致窗口：线程 A 更新数据库 → 线程 B 读缓存命中旧数据 → 线程 A 删除缓存。虽然短暂不一致，但通常能接受。更严格的一致性需要分布式锁、版本号、TCC 等机制，代价更高。

### 大白话解释

缓存就像你书桌上的一摞常用资料。你写论文时，最常引用的几本书就放在手边（Redis），随手拿；不常用的放回书架（MySQL），用的时候再走几步去拿。

但如果书桌上某本书突然被收走了（缓存过期），而你正好需要它，你就会同时冲向书架，和一群人挤在一起——这就是缓存击穿。如果书桌上所有书同时被收走，书架就炸了——这就是缓存雪崩。

### 核心启示

缓存是架构中最常用的性能杠杆，但**缓存本质上是在牺牲一致性换取性能**。使用缓存时，必须同时设计防穿透、防击穿、防雪崩、数据更新策略。没有这些配套设计，缓存加的越多，系统越脆弱。

## Q37 · Redis 为什么单线程还那么快？

![配图](images/chapter3/q37-redis-single-thread.png)



### 你有没有思考过这么一个问题？

Redis 是单线程模型，但它每秒能处理 10 万次以上的读写请求。而很多多线程程序反而没它快。这是为什么？单线程不会成为瓶颈吗？

面试会问：Redis 单线程指的是什么？Redis 6.0 为什么引入多线程？Redis 的瓶颈在哪里？

### 底层发生了什么？

Redis 的"单线程"指的是**处理命令的主线程是单线程的**。一个客户端请求进来，从读取请求、解析命令、执行命令到返回结果，都在一个主线程里按顺序执行。这避免了多线程竞争、上下文切换、锁带来的开销。

Redis 为什么单线程还快？

1. **纯内存操作**：所有数据都在内存，读写速度比磁盘快几个数量级。
2. **IO 多路复用**：Redis 使用 epoll/kqueue/IOCP 等机制，用一个线程同时监听成千上万个连接。只有真正有数据的连接才会被处理。
3. **数据结构简单高效**：SDS（Simple Dynamic String）、跳表、整数数组、压缩列表等，都是为特定场景优化的。
4. **避免锁竞争**：单线程没有线程切换和锁开销，每个命令的执行时间可预测。

**Redis 的瓶颈不是 CPU，而是网络带宽和内存。** 对于单个 key 的复杂操作（如大列表排序、大键删除），会阻塞主线程，导致其他请求等待。所以 Redis 强调避免大 key 和慢查询。

**Redis 6.0 的多线程**

Redis 6.0 引入了**多线程 IO**：读取请求和发送响应可以并行处理，但执行命令仍然是单线程。这样可以在网络 IO 成为瓶颈时提升吞吐量，同时保持命令执行的原子性和简单性。

Redis 7.0 还引入了多线程功能模块（如 io-threads-do-reads），但核心执行逻辑仍保持单线程。

### 大白话解释

Redis 就像一个超级高效的便利店收银员。店里只有一个收银台（单线程），但这位收银员同时盯着所有排队通道（IO 多路复用），哪个通道有人要结账就处理哪个。因为所有商品都放在收银台旁边的架子上（内存），不用跑去仓库取，所以结账极快。

多线程好比开多个收银台，但顾客要在多个台之间换来换去，而且可能两个台同时去拿同一瓶饮料造成混乱。只要单个收银台已经够快，多开台未必更优。

### 核心启示

Redis 单线程的设计告诉我们：**性能瓶颈不一定靠加线程解决**。有时候减少竞争、简化模型、聚焦真正瓶颈（网络 IO、内存访问），比盲目上多线程更有效。评估性能时，先找到瓶颈在哪里，再决定优化方向。

## Q38 · 分库分表——为什么单表 1000 万行就要开始拆？

![配图](images/chapter3/q38-sharding.png)



### 你有没有思考过这么一个问题？

你的用户表一开始只有几万条，存在 MySQL 里运行良好。但用户增长到 1000 万、1 亿、10 亿时，查询越来越慢，索引也越来越大。终于有一天，单表数据量太大，数据库 CPU 和磁盘都扛不住了。怎么办？分库分表。

面试会问：分库和分表有什么区别？垂直分片 vs 水平分片？分片键怎么选？跨分片查询怎么办？

### 底层发生了什么？

**垂直分片（Vertical Sharding）**

按业务模块拆分。比如把用户表拆成 `user_basic`（用户ID、昵称、头像）和 `user_profile`（地址、签名、标签）。适合字段很多、冷热不均的表。核心表变小，访问更集中。

**水平分片（Horizontal Sharding）**

按行拆分。比如用户表有 1 亿行，按 user_id 取模分到 10 个库，每个库 1000 万行。这是最常见的"分库分表"。

**分片键（Sharding Key）选择**

分片键是水平分片的核心。常见选择：

- **按用户 ID 取模**：查询用户相关数据时直接定位到分片。但跨用户查询（如按手机号查）需要全分片扫描。
- **按时间范围分片**：适合日志、订单等有时序特征的数据。最近数据查询频繁，历史数据归档。
- **按地域分片**：适合有明显地域属性的业务，如外卖、网约车。
- **哈希分片**：打散数据，避免热点。但范围查询（BETWEEN）性能差。
- **一致性哈希**：分片数量变化时，只影响少部分数据映射。常用于缓存集群。

**分库分表带来的新问题**

- **跨分片 JOIN**：不同分片的数据不能直接 JOIN，需要在应用层组装，或改用宽表、数据冗余。
- **全局唯一 ID**：自增 ID 在多个分片上会重复。需要 Snowflake、号段分配、UUID 等全局 ID 方案。
- **跨分片事务**：ACID 事务在分库后很难保证。常用最终一致性（消息队列、Saga、TCC）替代。
- **分片扩容**：分片数量变化后，数据需要重新分布。这是一个痛苦的迁移过程，所以分片数量要预估未来增长。

### 大白话解释

分库分表就像图书馆书太多，一个书架放不下。垂直分片是：把小说放一个房间，科技书放另一个房间。水平分片是：小说太多，按作者姓氏首字母 A-J、K-T、U-Z 分到三个书架。

分片键就是"按什么规则找书"。按作者姓氏找书很方便，但如果顾客想"找出所有红色封面的书"，你就要跑遍所有书架——这就是跨分片查询的麻烦。

### 核心启示

分库分表是**数据规模增长后的必然选择**，但它不是银弹。它解决了单表容量和性能问题，却引入了分布式一致性、跨分片查询、扩容迁移等新问题。能不分就不分；如果必须分，要尽量让 80% 的查询通过分片键定位到单一分片，剩下的 20% 用聚合、冗余、异步同步来处理。

## Q39 · 全局唯一 ID——自增 ID 在分布式系统中为什么不灵了？

![配图](images/chapter3/q39-id-generation.png)



### 你有没有思考过这么一个问题？

单机 MySQL 里，主键用 `AUTO_INCREMENT` 非常方便：插入一条记录，ID 自动加 1。但系统拆成多个数据库后，每个库都自增，ID 就会重复。怎么生成全局唯一 ID？这是一个看似简单但很考验工程能力的问题。

面试会问：Snowflake 的 ID 结构是什么？Leaf 方案怎么解决时钟回拨问题？UUID 为什么不能做数据库主键？号段分配模式的优缺点？

### 底层发生了什么？

**方案一：UUID**

UUID（Universally Unique Identifier，通用唯一识别码）是 128 位随机数，理论上不重复。但它有以下问题：

- 太长（36 字符），存储空间大，索引效率低。
- 完全无序，插入数据库时会导致页分裂，性能差。
- 可读性差，不适合日志排查和人工识别。

所以一般不用 UUID 做数据库主键，但可以用它做业务上的唯一标识（如订单号之外的追踪号）。

**方案二：Snowflake**

Twitter 开源的 Snowflake 算法生成 64 位 Long 型 ID：

- 1 bit：符号位，固定为 0。
- 41 bits：时间戳（毫秒级），可以使用约 69 年。
- 10 bits：机器 ID，支持 1024 个节点。
- 12 bits：序列号，每毫秒每个节点最多生成 4096 个 ID。

Snowflake 的优点：趋势递增（按时间排序）、高性能、可排序。缺点：依赖机器时钟，如果时钟回拨（NTP 同步或手动调整），可能生成重复 ID。

**方案三：号段分配（Leaf Segment）**

美团 Leaf 方案的核心是：数据库维护一个 biz_tag 和 max_id，每次从数据库批量申请一个号段（如 1000 个 ID），存在内存中，用完再申请。这样数据库访问次数大大减少。

优点是 ID 单调递增，对数据库友好；缺点是依赖数据库，DB 故障时可能短时间内无法生成 ID。双缓冲机制可以缓解：一个号段快用完时异步申请下一个号段。

**方案四：Snowflake + 号段混合（Leaf Snowflake）**

Leaf 还改造了 Snowflake：用 ZooKeeper 分配 workId，同时用数据库定时上报时间戳，解决时钟回拨问题。它结合了 Snowflake 的高性能和号段分配的稳定性。

**选型建议**

- 高并发、可排序、不太在意趋势递增：Snowflake。
- 强递增、数据库友好、可接受少量中心依赖：号段分配。
- 要同时兼顾两者：Leaf Snowflake 或类似方案。

### 大白话解释

单机自增 ID 就像一个自动编号器，每次有人来了就按一次按钮。但如果有多个入口，每个入口都有一个编号器，两个入口可能同时按出"第 5 号"。

Snowflake 方案就像给每个入口分配一个独一无二的窗口编号，并且编号器上还刻着时间。每个人拿到的号码不仅是唯一的，而且大致按时间顺序排列。

号段分配则是：不是每个人来的时候都申请一个号码，而是每个窗口一次性领 1000 个号码在手里，慢慢发，发完再去领。这样减少了排队领号的次数。

### 核心启示

全局唯一 ID 的核心矛盾是**唯一性、递增性、性能、去中心化**之间的取舍。自增 ID 最容易但无法分布式；UUID 最去中心化但无序列号；Snowflake 和号段分配是工程实践中最常用的折中。设计 ID 时，要先明确你最关心什么，而不是一上来就用 UUID 或 Snowflake。

## Q40 · 微服务——为什么大公司要把系统拆成一堆小服务？

![配图](images/chapter3/q40-microservices.png)



### 你有没有思考过这么一个问题？

小公司的系统可能就是一个大单体应用：用户模块、订单模块、库存模块全都在一个代码仓库里。但大公司会把它们拆成独立的微服务：用户服务、订单服务、库存服务、物流服务……各管一摊。为什么要这么做？微服务真的是"银弹"吗？

面试会问：单体和微服务的优缺点？服务拆分的粒度怎么把握？微服务带来哪些问题？

### 底层发生了什么？

微服务架构（Microservices Architecture）是把一个大型应用拆分成多个小而自治的服务，每个服务独立开发、独立部署、独立扩展，通过 API 或消息队列通信。

**为什么拆分？**

1. **独立扩展**：订单模块流量大，只扩展订单服务；用户模块流量小，不用扩展。资源利用率更高。
2. **独立技术栈**：不同服务可以用不同的语言、数据库、框架。搜索服务用 Go，推荐服务用 Python，交易服务用 Java。
3. **团队自治**：康威定律（Conway's Law）说，系统设计会反映组织结构。大公司的团队可以各自负责一个服务，独立迭代。
4. **故障隔离**：一个服务出问题，不会拖垮整个系统。比如推荐服务挂了，商品详情页还能看。

**微服务的代价**

- **分布式复杂性**：服务调用链变长，网络延迟、超时、重试、熔断、限流都需要考虑。
- **数据一致性**：每个服务有自己的数据库，跨服务事务要用 Saga、TCC 等最终一致性方案。
- **运维复杂度**：服务数量多，部署、监控、日志、追踪、配置都更复杂。
- **接口契约**：服务间 API 变更需要兼容，版本管理变得重要。

**拆分粒度**

拆得太粗，失去微服务意义；拆得太细，管理成本爆炸。常见原则：

- **按业务能力拆分**：订单、库存、支付、物流，每个对应一个业务领域。
- **DDD（领域驱动设计）Bounded Context**：以业务边界为服务边界。
- **两披萨团队原则**：一个服务应该由不超过两个披萨能吃饱的团队维护。

### 大白话解释

单体应用就像一家全能型餐馆：厨师既要做川菜、又要做日料、还要做甜品。菜单越来越大，厨房越来越乱，一个人请假整个餐厅可能受影响。

微服务就像美食广场：川菜馆、日料店、甜品店各自独立。想吃川菜就去川菜馆，川菜馆排队不会影响甜品店。但美食广场需要统一的水电、物业管理、招牌导视（服务发现、网关、监控）。

### 核心启示

微服务不是目的，而是**解决大型组织复杂性和规模化问题的手段**。小公司用单体更省成本；大公司拆微服务是为了让团队独立演进。不要盲目追微服务，要问自己：你的团队规模、业务复杂度、运维能力，是否真的需要它？
