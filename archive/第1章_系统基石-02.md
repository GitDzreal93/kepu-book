## Q6 · 一个数组访问越界——Java抛异常、Go panic、C可能静默破坏内存

![配图](images/chapter1/q06-array-bounds.png)





### 你有没有思考过这么一个问题？

你写 Java 时数组越界了——JVM 抛出一个清晰干净的 `ArrayIndexOutOfBoundsException`，附带完整堆栈跟踪。

你写 Python 时越界了——`IndexError: list index out of range`。

Go 直接 `panic: runtime error: index out of range`，程序退出。

然后老前辈说："C 语言里，数组越界不一定会报错。有时它崩、有时不崩还'正常'跑、有时它默默改了别人变量的值然后你排查一整天。"

**这不是"C 语言不好"。** 这是不同语言对内存暴露的层次不同。

### 底层发生了什么？

数组在内存中的布局，决定了越界的后果。

---

**Java 的数组：有元数据，强制检查。**

每个 Java 数组对象在堆内存中的布局是：

```
[对象头（Mark Word + Klass 指针）] [数组长度（4字节 int）] [arr[0] ... arr[n-1]]
```

当 JVM 执行 `arr[30]` 时，它先从对象头拿到数组长度 10，然后判断 30 >= 10 → 抛异常。

这个越界检查是在 JVM 内部做的——它是字节码指令 `aaload` / `iastore` 的内置逻辑。**这个检查是强制的**，JVM 标准规定必须做，你不能绕过。

---

**Go 的数组：无运行时元数据，编译器插检查。**

`[10]int` 在编译后的二进制里就是 10 个连续的 8 字节整数，共 80 字节。数组长度 10 只在编译时存在，**运行时不存在**。

但 Go 的编译器在每次数组访问前**插入了边界检查指令**：

```
CMP index, len(arr)   // 比较索引和长度
JAE panic             // 如果索引 >= 长度，跳转到 panic
```

在某些场景（比如你写了一个循环，编译器能证明 `index < len`），Go 做 **BCE（边界检查消除）** 优化，把这行 CMP 去掉。

---

**C 语言的数组：什么都没有。**

`int arr[10]` 在栈上分配 10 个连续的 4 字节 int，共 40 字节。数组名 `arr` 只是一个**指针**——指向首元素地址。

**没有任何地方存储"这个数组有多长"。**

`arr[30]` 被编译器翻译成：取 `arr` 的地址 + 30 × sizeof(int) = `arr` 地址 + 120 字节，然后读那个地址的内容。

至于那个地址里是什么——它可能是同一栈帧里另一个变量的值、可能是返回地址、可能是未映射的内存（一读就段错误）。**没有报错不是因为它"被允许"，是因为没有人检查。**

### 大白话解释

- **Java** = 有严格海关登记的国家。过海关必须出示护照，超了通行范围一定被拦。代价：每次出入境都要过海关。
- **Go** = 有简化电子登记的国家。海关把规则写在了所有人扫码的卡里（编译器插边界检查）。有些场景确定安全就免扫（BCE）。
- **C** = 完全开放的边境。没有检查站。你踩油门冲向铁丝网——穿过田野可能安全（破坏了别人农田），穿过地雷区（未映射内存）挂了。**C 的哲学是"我们相信你不会越界"——代价是你必须自己负责。**

### 核心启示

> 数组不是一个数据容器——它是一个"被包裹了不同层级安全协议的内存块"。包裹层级越高，越安全但越慢；包裹层级越低，越快但越危险。**选择哪种语言 = 选择"花多少运行时开销换取多大程度的'不可能犯这类错'的保证"。**

## Q7 · 0.1+0.2≠0.3——设计金融系统时，浮点数该不该碰？

![配图](images/chapter1/q07-float-precision.png)





### 你有没有思考过这么一个问题？

打开任何语言的 REPL，敲 `0.1 + 0.2`：

- Java 返回 `0.30000000000000004`
- Python 返回 `0.30000000000000004`
- JavaScript 也一样

这不是 bug。这是 **IEEE 754 浮点数标准**的特性，而且是"不是 bug 的特性"。

但如果你在写一个金融交易系统，`0.1 + 0.2` 的不精确会让你**损失真金白银**。

### 底层发生了什么？

浮点数的真相：**二进制无法精确表示大多数十进制小数。**

10 进制中，1/3 = 0.333333... 是无限循环。同理，在 2 进制中，1/10 = 0.0001100110011... 是**无限循环**。

当我们用 `double`（IEEE 754 双精度：1 位符号 + 11 位指数 + 52 位尾数）存储 0.1 时，它被"裁剪"了——存储的是它能表示的**最近似值**：

```
实际存储的 0.1 ≈ 0.1000000000000000055511151231257827021181583...
```

0.2 同理——也是近似值。两个近似值相加 = 结果的近似值 ≠ 0.3 的近似值。

---

**金融系统的方案抉择：**

| 方案 | 代表 | 优点 | 代价 |
|------|------|------|------|
| 精确十进制 | `BigDecimal` | 不损失精度 | 比 double 慢一个数量级 |
| 整数"分"方案 | `long cents = 1099` (10.99元) | 快，无误差 | 乘除法有截断差异（税率计算） |
| 定点数库 | 128位整数+可控舍入 | 精度可控 | 性能和复杂度居中 |

**实际选择：** 绝大多数交易所核心系统用整数"分"方案 + 严格的舍入规则文档。需要极高精度（如加密货币）或复杂建模（利率衍生品）时才用 BigDecimal 或定点数。

### 大白话解释

你用尺子量东西。用米尺（**二进制**）量一个恰好 1/10 米的东西——米尺的刻度是按 1/2、1/4、1/8、1/16 刻的——永远找不到 1/10 米的精确刻度，只能找最接近的那一格（**近似**）。两段近似拼在一起，误差叠加。

`BigDecimal` = 专门准备了一把十进制刻度的尺子。整数"分"方案 = 你干脆不说"1.99 元"，说"199 分"——全是整数尺子，不需要小数。

### 核心启示

> 浮点数的精度问题不是数学错误——是**"精度-速度"的系统级权衡**。IEEE 754 被设计成能在 CPU 上用单一硬件指令完成 `double` 加法（1 个时钟周期），代价是"不是精确表示所有实数"。**所有工程决策的本质，都是在约束下选择你的非妥协项。** 金融选精确性，游戏选速度。

## Q8 · CPU缓存和Redis缓存——都是"缓存"，一个你完全控制不了

![配图](images/chapter1/q08-cpu-vs-redis-cache.png)





### 你有没有思考过这么一个问题？

你写 `redisTemplate.opsForValue().get("user:12345")`——打穿 Redis → 回源 MySQL → 写回 Redis。你是在**手动管缓存的生命周期**。

但在你的代码执行 `user.getName()` 时，CPU 内部正发生着比你快一百倍的缓存操作——L1/L2/L3 缓存**完全自动、不可编程**。

同样是缓存，一层的淘汰策略在你手里，另一层完全对你不可见。**为什么？**

### 底层发生了什么？

CPU 缓存和 Redis 缓存虽然都叫"缓存"，本质完全不同——关键在**透明性**。

---

**CPU 缓存：硬件级，完全透明。**

当 CPU 执行 `mov rax, [rbx]`（从内存读数据到寄存器）时，MMU 和缓存控制器自动做了：

```
查 L1 → 命中? 2-4 周期返回 → 未命中? 查 L2 → L3 → 主内存(~100周期)
```

整个过程对 CPU 指令完全透明——同一个 `mov` 指令，可能 1ns 完成可能 100ns 完成，取决于数据当时在缓存的什么地方。

**CPU 缓存的替换策略焊接在硅片上——你不能调、不能关、不能用代码干预。**

为什么？两个理由：

- **速度**——任何软件介入都会吃掉缓存的速度优势（L1 命中只有 1ns，任何分支跳转都比这慢）
- **设计哲学**——缓存对程序完全透明，正因此**所有程序自动受益**

---

**软件缓存（Redis/Guava/Caffeine）：显式，可控。**

你决定什么数据放进去、TTL 设多久、怎么淘汰。

这层缓存不透明，恰恰是因为你面对的数据访问模式不是均匀的——热门微博今天不同明天，需要主动刷新；VIP 用户数据需要长 TTL。

尽管 CPU 缓存透明，你仍可以通过代码风格间接影响它——遍历矩阵时**按行访问 vs 按列访问**，缓存命中率差几个数量级。这是**对隐式系统的间接优化**。

### 大白话解释

CPU 缓存 = 你身体的**自动呼吸**。脑干替你做了，你不需要想"现在吸、现在呼"。你只能间接影响——跑步加速时呼吸自动加深。

Redis 缓存 = 你主动拿一个**收纳盒**把常用东西放身边。你知道什么东西常用，你决定放进去的时间和扔掉的标准。代价：每次拿东西都要想"在不在盒子里、满了要不要扔"——CPU 缓存完全不需你做这些思考。

### 核心启示

> CPU 缓存和 Redis 缓存代表了两种截然不同的设计哲学：**"为我做一切"的透明性和"我来决定一切"的可控性**。透明系统自动优化一切但不能适应特殊场景；显式系统让你精细调整但引入认知和操作负担。**这个哲学不仅适用于缓存——微服务的透明通信 vs 显式 RPC、数据库自动优化 vs 手动索引——都是同一场"透明和可控"之间的永恒拉扯。**
