## Q51 · CSRF——为什么你登录了银行网站，点一个链接钱就被转走了？

![配图](images/chapter4/q51-csrf.png)

### 你有没有思考过这么一个问题？

你登录了银行网站，浏览器里还保存着 Cookie。然后你访问了一个恶意网站，这个网站里有一张图片：

```html
<img src="https://bank.com/transfer?to=attacker&amount=10000">
```

你一看是图片，其实浏览器会自动带着你的 Cookie 向银行发起请求。如果这个转账接口没有额外校验，钱就被转走了。这就是 CSRF（Cross-Site Request Forgery，跨站请求伪造）。

### 底层发生了什么？

CSRF 的本质是：**攻击者利用你的浏览器自动携带 Cookie 的机制，诱导你（的浏览器）向受信任网站发起非自愿的请求。** 它不是窃取你的 Cookie，而是"借用"你的登录状态。

CSRF 成立的条件：

1. 你已经登录了目标网站，Cookie 有效。
2. 目标网站的请求只依赖 Cookie 验证身份，没有额外校验。
3. 攻击者知道或能猜测请求参数和 URL。

**防御方法**

- **CSRF Token**：服务器给每个表单或请求生成一个随机 Token，前端提交时带上，后端验证。攻击者无法获取这个 Token（因为受同源策略限制）。
- **SameSite Cookie**：设置 Cookie 的 `SameSite` 属性。
  - `SameSite=Strict`：完全不允许第三方网站携带 Cookie。
  - `SameSite=Lax`：允许部分安全请求（如 GET 链接跳转），但禁止 POST 等写操作跨站携带。
- **双重 Cookie 验证**：Cookie 中放一个随机值，同时请求中也带上这个值，后端比对是否一致。
- **关键操作二次验证**：转账、修改密码等操作要求输入短信验证码、支付密码，即使请求伪造也无法通过。

CSRF Token 是最经典的做法。注意：它不能防御 XSS，因为 XSS 可以直接读取页面上的 Token。

### 大白话解释

你进了一家银行，柜员认识你（Cookie）。攻击者不能冒充你进银行，但他可以写一封信，假装是你签的名，骗邮差（浏览器）把你的印章（Cookie）盖上去，然后寄到银行。银行一看印章对，就照办了。

CSRF Token 相当于银行每次给你一封信时，都附带一个随机暗号。真正的信里必须有这个暗号，攻击者拿不到暗号，就无法伪造。

### 核心启示

CSRF 告诉我们：**Cookie 能验证身份，但不能证明请求是用户本人自愿发起的。** 任何敏感操作都需要额外的请求来源校验。现代浏览器通过 SameSite Cookie 大幅削弱了 CSRF，但后端仍然应该在写操作上加 CSRF Token 或二次验证。

## Q52 · DDoS——为什么带宽够大，网站还是被冲垮了？

![配图](images/chapter4/q52-ddos.png)

### 你有没有思考过这么一个问题？

你公司买了 1Gbps 带宽，结果攻击者用 10Gbps 的流量打过来，网站还是挂了。DDoS（Distributed Denial of Service，分布式拒绝服务）攻击的本质不是网站代码有漏洞，而是**资源消耗战**。攻击者用海量请求占满你的网络、CPU、连接数，让正常用户进不来。

### 底层发生了什么？

DDoS 有很多种类型：

**1. 流量型攻击（Volumetric）**

用大量数据包占满目标带宽。典型代表：UDP Flood、ICMP Flood。攻击者可能利用 DNS 反射、NTP 反射、Memcached 反射放大流量，用很小的请求产生巨大的响应流量。

**2. 协议型攻击（Protocol）**

利用协议本身的缺陷消耗服务器资源。典型代表：SYN Flood（TCP 半连接攻击）、Ping of Death。这类攻击不需要很大带宽，就能让服务器连接表爆满、CPU 耗尽。

**3. 应用层攻击（Application Layer）**

针对 HTTP 业务逻辑，请求看起来正常，但消耗大量后端资源。例如大量请求数据库查询接口、登录接口、搜索接口。这种攻击最难识别，因为请求本身合法。

**防御方法**

- **流量清洗（Scrubbing Center）**：把流量引流到高防机房或云厂商的清洗中心，过滤恶意流量后再把干净流量回注。
- **CDN 和 Anycast**：分散流量到全球边缘节点，避免单点被打爆。
- **速率限制（Rate Limiting）**：单 IP 请求频率限制。
- **WAF 和 Bot 管理**：识别机器人流量，区分人和脚本。
- **SYN Cookie**：防御 SYN Flood，不预分配连接资源。
- **黑洞路由（Blackhole）**：当攻击流量太大时，运营商直接把目的 IP 的流量丢弃，保护整个网络，但代价是目标业务不可用。

DDoS 没有完美的防御。防御成本通常随着攻击规模指数上升。因此云厂商提供高防 IP、高防 CDN，按攻击规模计费。

### 大白话解释

DDoS 就像一群人故意同时挤进一家小餐馆，把门口堵死，让真正的顾客进不来。他们不是来吃饭的，而是来占座的。餐馆越大（带宽越高），能容纳的人越多，但如果来捣乱的人更多，还是会被堵死。

### 核心启示

DDoS 防御的核心是**分层缓解 + 资源储备**：边缘清洗、CDN 分散、入口限流、应用层优化。单点扛不住海量请求，必须借助分布式架构和专业的抗 D 服务。安全预算里，DDoS 防护通常是不可避免的一块。

## Q53 · 风控系统——为什么抢红包时有人被封号？

![配图](images/chapter4/q53-risk-control.png)

### 你有没有思考过这么一个问题？

你抢红包、抽奖、领优惠券，有时候系统会突然弹出"操作异常，账号已被限制"。这不是人工封的，而是**风控系统**根据你行为特征自动判断的。风控系统是互联网公司的"免疫系统"，专门识别异常和欺诈。

### 底层发生了什么？

风控系统通常由三大部分组成：数据采集、规则引擎、模型决策。

**1. 数据采集**

记录用户行为的方方面面：设备指纹、IP 地址、地理位置、操作频率、点击轨迹、历史行为、社交关系等。设备指纹可以识别"同一台设备注册了大量账号"。

**2. 规则引擎**

用硬规则快速判断。例如：

- 同一 IP 在 1 分钟内注册超过 10 个账号 → 拦截。
- 同一个设备在一小时内登录了 100 个账号 → 拦截。
- 收货地址集中在某个异常区域 → 人工审核。

规则引擎响应快，适合明确的风控场景。但规则容易被绕过，需要不断更新。

**3. 模型决策**

用机器学习模型识别更隐蔽的异常。例如：

- 异常登录检测：用户平时在北京登录，突然在境外登录。
- 团伙识别：通过设备、IP、账号关系网络发现批量注册的黑产团伙。
- 交易欺诈：判断一笔交易是否被盗刷。

模型需要大量标注数据和特征工程，输出风险分数，再由业务规则决定是否拦截。

**风控的挑战**

- **误杀**：正常用户被误判为黑产。比如公共 WiFi 下多个用户共用一个 IP。
- **对抗**：黑产会不断尝试绕过规则，风控系统要持续升级。
- **实时性**：有些场景必须在几十毫秒内完成决策（如支付），否则影响用户体验。
- **可解释性**：被拦截的用户需要知道为什么，但复杂模型往往黑盒。

### 大白话解释

风控系统就像小区门口的保安。他认识你，知道你几点出门、几点回家。如果你突然半夜带着一堆陌生人频繁进出，或者有人穿着你的衣服但走路姿势不对，保安就会拦下来问问。

但保安也会犯错：比如你请了几个朋友来家里玩，保安可能误以为是小偷团伙。

### 核心启示

风控的本质是**在用户体验和安全性之间找平衡**。过于严格会误伤正常用户，过于宽松会让黑产得逞。好的风控系统不是一套规则管所有，而是分场景、分等级、分人群，实时调整策略。

## Q54 · OAuth 2.0——为什么你点"用微信登录"就能授权第三方？

![配图](images/chapter4/q54-oauth2.png)

### 你有没有思考过这么一个问题？

你登录某个网站，选择"微信登录"，微信弹出一个授权页："XX 应用想获取你的头像和昵称"。你点同意后，网站就登录成功了。这个过程中，网站并没有拿到你的微信密码，但拿到了你的部分信息。这是怎么实现的？这就是 OAuth 2.0。

### 底层发生了什么？

OAuth 2.0 是一个**授权框架**，它的目标是：让第三方应用在不获得用户密码的情况下，访问用户在另一个服务上的资源。

OAuth 2.0 有几种授权模式，最常用的是**授权码模式（Authorization Code Grant）**：

1. 用户访问第三方应用 A，A 把用户重定向到授权服务器（如微信）。
2. 用户在微信授权页登录并同意授权。
3. 微信返回一个授权码（Authorization Code）给 A。
4. A 用授权码 + 自己的 Client ID 和 Client Secret，向微信换取 Access Token。
5. A 用 Access Token 调用微信 API，获取用户头像、昵称等。

为什么用授权码而不是直接返回 Token？因为步骤 3 的回调会经过浏览器，可能被泄露。授权码本身没有访问权限，必须配合 Client Secret 才能换 Token。Client Secret 只保存在 A 的服务器上，不会暴露给浏览器。

**关键概念**

- **Resource Owner（资源所有者）**：用户本人。
- **Client（客户端）**：第三方应用。
- **Authorization Server（授权服务器）**：微信的认证中心。
- **Resource Server（资源服务器）**：存储用户资源的服务器，通常是授权服务器的一部分。
- **Access Token**：访问令牌，第三方用它调用 API。
- **Refresh Token**：刷新令牌，用于 Access Token 过期后获取新的 Access Token。
- **Scope**：授权范围，比如只获取头像昵称，不获取好友列表。

### 大白话解释

OAuth 2.0 就像你去酒店，前台要复印你的身份证。你不想把身份证交给前台，于是你带前台去派出所（授权服务器）。派出所核实你身份后，给前台开了一张"该客人已实名认证"的证明（Access Token），但证明上只写必要信息，不写你的完整身份证号。前台拿着证明就可以办理入住。

### 核心启示

OAuth 2.0 的核心是**把"认证"和"授权"分离**，让用户在不泄露密码的情况下把部分权限委托给第三方。但要注意：OAuth 2.0 只是授权框架，不是认证协议。如果你需要确认用户身份，要在 OAuth 2.0 之上使用 OIDC（OpenID Connect）。

## Q55 · SSO 单点登录——为什么登录了企业邮箱，OA 也不用重新输密码？

![配图](images/chapter4/q55-sso.png)

### 你有没有思考过这么一个问题？

你登录公司邮箱后，打开 OA 系统、HR 系统、财务系统，发现都不用再输密码。这就是 SSO（Single Sign-On，单点登录）。它解决的是：在多个互相信任的系统中，只需要认证一次。

### 底层发生了什么？

SSO 的核心是**集中认证 + 共享票据（Ticket）**。用户认证交给一个专门的认证中心（Identity Provider，IdP），其他应用（Service Provider，SP）信任这个认证中心。

以 CAS（Central Authentication Service）协议为例：

1. 用户访问应用 A，A 发现用户未登录，重定向到 CAS 认证中心。
2. 用户在 CAS 登录，CAS 给用户一个全局会话 Cookie 和一张 Service Ticket。
3. 用户带着 Ticket 回到应用 A，A 用 Ticket 去 CAS 验证。
4. CAS 验证通过，A 建立本地会话，用户登录成功。
5. 用户访问应用 B，B 也把用户重定向到 CAS。但此时用户已经有 CAS 的全局会话 Cookie，CAS 直接颁发新的 Ticket 给 B，无需再次输入密码。

OIDC 和 SAML 也是常见 SSO 协议。OIDC 基于 OAuth 2.0，返回 ID Token（JWT）作为身份凭证；SAML 用 XML 断言在企业级系统里广泛使用。

**SSO 的优点和风险**

优点：用户体验好，IT 管理方便，密码策略统一。

风险：一旦认证中心被攻破，所有接入系统都受影响。所以认证中心必须重点保护，通常要启用 MFA（多因素认证，Multi-Factor Authentication）。

### 大白话解释

SSO 就像游乐园的通票。你在大门口验一次票（认证中心），园内所有项目（各个应用）看到通票就让你进。不用每个项目门口都排队买票。

但如果大门口有人造假票，所有项目都会被骗。所以大门必须用最严的验票方式。

### 核心启示

SSO 是**身份管理的基础设施**，但它把"单点登录"也变成了"单点失效"。设计 SSO 时，认证中心必须是整个系统里最受保护的部分。多因素认证、异常登录检测、最小权限原则，都是 SSO 必须配套的安保措施。
