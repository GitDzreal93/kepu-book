## Q23 · QUIC / HTTP3——为什么 TCP 被"嫌弃"了？

![配图](images/chapter2/q23-quic-vs-tcp.png)



### 你有没有思考过这么一个问题？

TCP 是互联网最可靠的传输协议，但最近几年大厂们纷纷拥抱 QUIC，Chrome、YouTube、Facebook 的流量大量走 UDP 上的 QUIC。HTTP/3 甚至直接基于 QUIC 构建。TCP 做了什么孽，要被这样嫌弃？

面试里常问：QUIC 解决了 TCP 的什么问题？为什么 QUIC 基于 UDP 而不是直接把 TCP 改掉？HTTP/2 的队头阻塞（Head-of-Line Blocking）和 TCP 队头阻塞有什么区别？QUIC 的 0-RTT 是怎么实现的？

### 底层发生了什么？

QUIC（Quick UDP Internet Connections，快速 UDP 网络连接）是 Google 提出、IETF 标准化的传输协议。它运行在 UDP 之上，但自己在应用层实现了 TCP 曾经提供的很多能力：可靠传输、拥塞控制、流量控制、加密、连接迁移。

TCP 有几个历史包袱，QUIC 一一针对它们做了设计：

**问题一：TCP + TLS 的握手延迟**

TCP 三次握手（1-RTT）之后，TLS 1.2 还要两次握手（2-RTT），总共 3-RTT 才能传数据。QUIC 把传输和加密握手合并，通常 1-RTT 就能开始传数据；如果之前连过这个服务器，甚至可以 0-RTT。

**问题二：TCP 队头阻塞（Head-of-Line Blocking）**

HTTP/2 支持一个连接里同时传多个流（stream），但底层还是 TCP。TCP 是按字节顺序交付的，如果某个流的一个数据包丢了，TCP 会重传，导致后续所有流的数据即使已经收到也必须等着。这就是 TCP 层面的队头阻塞。

QUIC 基于 UDP，每个流独立进行丢包检测和重传。一个流卡住了，不影响其他流。

**问题三：连接迁移困难**

TCP 用四元组（源 IP、源端口、目的 IP、目的端口）标识一个连接。你手机从 WiFi 切换到 4G，IP 变了，TCP 连接就断了，必须重新建连。QUIC 用连接 ID（Connection ID）标识连接，IP 变了没关系，只要连接 ID 还在，就能继续传数据。

**问题四：中间设备僵化**

TCP 协议栈被无数路由器、防火墙、运营商设备识别和优化（有时甚至修改），想改 TCP 几乎不可能。UDP 相对简单，中间设备不怎么动它，所以 QUIC 选择基于 UDP 来实现新机制，绕开网络中间件的惯性阻力。

HTTP/3 就是 QUIC 上跑 HTTP/2 的语义。它不再依赖 TCP，而是把传输层的事交给 QUIC 自己管。

### 大白话解释

TCP 像一条老旧的铁路：非常可靠，但修一条新支线要拆沿线无数房子，根本不可能。于是 QUIC 说：我在旁边另修一条高速公路（UDP），自己在高速公路上实现安检、收费站、车道管理——所有铁路的好处我都要，但不用拆房子。

这条高速公路还能让你在换路（切换网络）时不用重新排队进站，多个车道之间不会互相堵死。

### 核心启示

QUIC 不是"UDP 更好"，而是**在UDP之上重新实现了一套传输层，绕开了TCP改不动的困境**。它牺牲了部分协议栈的内核支持，换取了灵活性和低延迟。当你遇到"协议改不动"的瓶颈时，想想 QUIC 的思路：能不能在更上层重新封装，绕过旧协议的约束？

## Q24 · TCP 拥塞控制——为什么下载速度总是忽快忽慢？

![配图](images/chapter2/q24-tcp-congestion.png)



### 你有没有思考过这么一个问题？

你下载一个大文件，网速有时猛冲到几十 MB/s，有时又跌到几百 KB/s，然后慢慢回升。这不是服务器在逗你，而是 TCP 的拥塞控制算法在动态调节发送速率。它要回答一个难题：网络现在能承载多快的速度？

面试会问：慢启动、拥塞避免、快重传、快恢复分别是什么？TCP 怎么判断网络拥塞？BBR 和 Cubic 有什么区别？

### 底层发生了什么？

TCP 拥塞控制的核心目标是：**在不把网络压垮的前提下，尽量占满可用带宽。** 网络是一个共享资源，每个 TCP 连接如果都盲目发数据，路由器缓冲区会溢出，丢包率飙升，所有人的速度都掉下来。

TCP 用几个关键状态机来解决这个问题：

**慢启动（Slow Start）**

连接刚建立时，发送方不知道网络能扛多少。它从很小的拥塞窗口（cwnd，通常 1 个 MSS）开始，每收到一个 ACK，cwnd 就翻倍。指数增长让它能快速探测可用带宽。

**拥塞避免（Congestion Avoidance）**

当 cwnd 达到一个阈值（ssthresh，慢启动阈值）后，不再指数增长，而是线性增长，每 RTT 增加一个 MSS。这样更保守，避免猛然撞墙。

**快重传（Fast Retransmit）**

发送方收到三个重复的 ACK 时，意识到某个包丢了，不等超时，立刻重传这个包。这减少了丢包后恢复的时间。

**快恢复（Fast Recovery）**

快重传之后，不再从慢启动重新开始，而是把 ssthresh 设为 cwnd 的一半，然后进入拥塞避免阶段，线性增长。这比一下子回到 cwnd=1 更平滑。

**丢包检测：超时 vs 重复 ACK**

- 超时重传（RTO, Retransmission Timeout）：包丢了，ACK 一直没来，触发超时。这通常被认为是严重拥塞，cwnd 会大幅削减。
- 三个重复 ACK：说明网络只是偶发丢包，整体还能跑，所以用快重传/快恢复更温和。

**BBR（Bottleneck Bandwidth and Round-trip propagation time）**

Google 提出的 BBR 算法不再把丢包当拥塞信号，而是直接测量带宽和 RTT。它认为网络真正拥塞时，RTT 会变长、带宽会被占满。BBR 试图把发送速率调整到瓶颈带宽，而不是等到丢包才降速。在高带宽、高延迟网络（如跨国链路）上表现更好。

### 大白话解释

TCP 就像你开车上高速。刚上高速时，你慢慢踩油门（慢启动），发现路况好就快速提速；开到一定程度后，你不再猛踩，而是匀速前进（拥塞避免）。如果前面有点堵（三个重复 ACK），你稍微减速变道；如果完全堵死（超时），你只好先停下来，再重新起步。

BBR 则像是一个装了实时路况导航的车，它直接看前方道路能容纳多少车，而不是等到撞车了才知道减速。

### 核心启示

TCP 拥塞控制不是"越快越好"，而是**在共享网络中公平且高效地利用带宽**。设计任何涉及流量控制的系统（如消息队列、限流、线程池调度），都可以借鉴这个思想：探测容量、温和加速、快速但不过度地响应异常。

## Q25 · HTTP 缓存——浏览器为什么有时候不用请求服务器？

![配图](images/chapter2/q25-http-cache.png)



### 你有没有思考过这么一个问题？

你打开一个之前访问过的网站，很多图片、CSS、JS 文件根本没发网络请求，就直接从本地加载出来了。这就是 HTTP 缓存。但缓存也会带来麻烦：你更新了一个 JS 文件，用户浏览器还是加载旧版本；或者有时候你看到页面内容没变，但浏览器还是发了一次请求确认。这背后的逻辑是什么？

面试高频：强缓存和协商缓存的区别？`Cache-Control`、`Expires`、`ETag`、`Last-Modified` 怎么用？缓存击穿、穿透、雪崩有什么区别？

### 底层发生了什么？

HTTP 缓存分两大类：强缓存和协商缓存。

**强缓存（Fresh Cache）**

浏览器在本地直接判断缓存是否有效，不需要发请求到服务器。主要响应头：

- `Cache-Control: max-age=3600`：这个资源在 3600 秒内有效，过期前直接读本地。
- `Expires: Wed, 21 Oct 2026 07:28:00 GMT`：绝对过期时间，但受客户端时钟影响，现在用得少了。

强缓存期间，浏览器甚至不会向服务器发起请求，网络面板里显示 `from disk cache` 或 `from memory cache`。

**协商缓存（Validation Cache）**

强缓存过期后，浏览器会问服务器："我这份缓存还能用吗？" 服务器根据资源标识判断。主要响应头：

- `Last-Modified` + `If-Modified-Since`：按文件最后修改时间判断。但精度只有秒，且文件内容没变但时间变了会误触发更新。
- `ETag`（Entity Tag）+ `If-None-Match`：按文件内容生成的哈希或版本号判断。更精确，但更消耗服务器资源（尤其大文件）。

如果服务器返回 304 Not Modified，浏览器用本地缓存；如果返回 200，就下载新内容。

**缓存更新策略**

前端项目构建时通常会给文件名加 hash，比如 `app.abc123.js`。当代码更新时，文件名变成 `app.def456.js`，HTML 引用新文件，旧文件可以继续缓存一年。这是"**immutable cache**"策略：文件内容不变，URL 永远不变；内容变了，URL 也变。

**缓存问题的三个经典名词**

- **缓存穿透**：请求一个一定不存在的数据，缓存里没有，数据库里也没有，每次请求都打到数据库。
- **缓存击穿**：一个热点数据缓存过期，大量请求同时打到数据库。
- **缓存雪崩**：大量缓存同时过期，数据库压力瞬间暴增。

### 大白话解释

HTTP 缓存就像你书柜里的书。强缓存是：这本书你上个月才买的，你确定没过期，直接拿下来看。协商缓存是：这本书买了很久了，你不确定内容有没有更新，先打电话给出版社问"最新版还是这个 ISBN 吗？" 如果是，继续看；如果不是，买新版。

缓存击穿、穿透、雪崩则是：一群人同时去书店抢一本绝版书（击穿）；抢一本根本不存在的书（穿透）；书店所有书同时下架，所有人涌入仓库（雪崩）。

### 核心启示

缓存是性能优化的核心杠杆，但它本质上是一个**一致性 vs 性能**的权衡。缓存越久，速度越快，但内容越可能过期。做缓存策略时要问自己：这个资源更新频繁吗？过期了会有什么影响？用户看到旧数据能不能接受？

## Q26 · 负载均衡——为什么一个网站背后通常不是一台服务器？

![配图](images/chapter2/q26-load-balancer.png)



### 你有没有思考过这么一个问题？

双 11 零点，几亿人同时打开淘宝。如果只有一台服务器，就算它性能再强，也会被瞬间压垮。真正的网站背后都是成百上千台服务器。谁来决定你的请求去哪台机器？这就是负载均衡（Load Balancing）。

面试会问：负载均衡有哪些算法？四层负载均衡和七层负载均衡有什么区别？LVS、Nginx、HAProxy 各适合什么场景？会话保持（Session Affinity）怎么解决？

### 底层发生了什么？

负载均衡器是客户端和后端服务器之间的"调度员"。它的核心职责是：**把大量请求合理分配到多个服务器，避免某一台机器过载，并在机器故障时把它踢出。**

**负载均衡算法**

- **轮询（Round Robin）**：依次把请求发给每台服务器。简单公平，但不管服务器当前负载。
- **加权轮询（Weighted Round Robin）**：给性能强的服务器更高权重。
- **最小连接（Least Connections）**：把请求发给当前连接数最少的机器。适合长连接场景。
- **源地址哈希（IP Hash）**：根据客户端 IP 哈希决定后端机器。保证同一客户端总是落到同一台服务器，解决会话保持问题。
- **一致性哈希（Consistent Hashing）**：节点变化时只影响少部分请求映射。常用于缓存集群，避免大量缓存失效。

**四层 vs 七层负载均衡**

- **四层（L4）**：基于传输层信息（IP + 端口）转发。它不解析 HTTP 内容，速度更快。典型实现：LVS、AWS NLB、Google Cloud TCP Load Balancer。
- **七层（L7）**：基于应用层内容（URL、Header、Cookie）转发。可以做更智能的路由，比如把 `/api` 请求发到后端服务，把静态文件请求发到 CDN。典型实现：Nginx、HAProxy、AWS ALB。

**健康检查（Health Check）**

负载均衡器会定期探测后端服务器，如果某台机器不响应，就把它从转发列表中移除。等它恢复了再加回来。这个机制让系统有**自动容错能力**。

**会话保持（Session Affinity / Sticky Session）**

如果用户登录状态存在某台服务器的内存里，后续请求必须落到同一台机器，否则要重新登录。解决方法：用 IP Hash 让同一客户端固定到一台机器；或者把 session 存在共享存储（Redis）里，这样任何机器都能处理。

### 大白话解释

负载均衡就像银行网点的大堂经理。客户来了，经理看哪个窗口人少就把客户往哪引。如果某个柜员请假了（机器故障），经理就不再往他那引客户。如果 VIP 客户每次都固定去同一个理财经理（会话保持），经理就记住这个规则。

### 核心启示

负载均衡不只是"分摊压力"，它是系统**横向扩展（Scale Out）**和**高可用**的基础。单点服务器再强也有上限，负载均衡让多台机器像一个整体一样对外服务。设计分布式系统时，永远假设机器会坏，负载均衡就是你在故障面前的缓冲区。

## Q27 · DNS 劫持 / 污染——为什么输入正确网址会跳到假网站？

![配图](images/chapter2/q27-dns-hijacking.png)



### 你有没有思考过这么一个问题？

你输入正确的银行网址，浏览器却打开了一个钓鱼页面。网址没错，但页面是假的。这可能不是病毒，而是 DNS 被劫持或污染了。它是怎么发生的？又怎么防御？

面试会问：DNS 劫持和 DNS 污染有什么区别？HTTPS 能完全防御吗？DoH / DoT 是什么？

### 底层发生了什么？

DNS 劫持和 DNS 污染都是**把域名解析到错误 IP**的攻击，但实现方式不同。

**DNS 劫持（DNS Hijacking）**

攻击者控制了你的本地网络设备（路由器）、操作系统 DNS 设置，或者运营商 DNS 服务器，把正常域名的解析结果改成恶意 IP。你访问 `bank.com`，实际上被指向了攻击者的服务器。

常见场景：你连了公共场合的免费 WiFi，路由器被篡改；你的电脑中毒，hosts 文件被改；恶意插件修改了系统 DNS 设置。

**DNS 污染（DNS Poisoning / Cache Poisoning）**

攻击者向 DNS 服务器或缓存中注入伪造记录。比如你请求的域名解析报文在传输途中被篡改，返回了一个错误的 IP。DNS 协议本身没有加密，数据包容易被伪造。DNS 缓存服务器拿到错误记录后，还会把它缓存下来，影响更多用户。

**防御手段**

- **DNSSEC（DNS Security Extensions）**：用数字签名验证 DNS 响应的真实性和完整性。防止记录被篡改。
- **DoH（DNS over HTTPS）**：把 DNS 查询封装在 HTTPS 里，加密传输，防止中间人监听和篡改。
- **DoT（DNS over TLS）**：用 TLS 加密 DNS 查询，类似 DoH，但运行在独立端口 853。
- **HTTPS 证书校验**：即使 DNS 被劫持，HTTPS 证书链验证也会失败——攻击者拿不到银行证书的私钥，无法伪造受信任证书。浏览器会报警。

### 大白话解释

DNS 劫持就像有人在电话簿里把医院的电话号码改成了黑诊所的号码。你按正确的名字查，但打过去是骗子。DNS 污染则像有人把电话公司总机的记录本改了，不仅你一个人，所有查这个名字的人都会被骗。

HTTPS 证书就是验证对方身份的最后一道防线：你打电话过去，对方必须出示医院的工作证。黑诊所拿不出真工作证，你就要警惕。

### 核心启示

DNS 是互联网信任的底层基础设施，但它本身并不安全。DNSSEC、DoH、DoT 都是为了修复这个历史漏洞。做安全设计时，不要假设底层协议可信，要在应用层再加一层校验——正如 HTTPS 所做的那样。

## Q28 · 长连接 vs 短连接——什么时候该保持连接？

![配图](images/chapter2/q28-keepalive.png)



### 你有没有思考过这么一个问题？

HTTP 请求默认是短连接：请求一次、响应一次、连接就关闭。如果你要连续请求 100 个资源，是不是每次都要重新 TCP 三次握手？那得多慢？HTTP  keep-alive 长连接就是来解决这个问题的。但长连接也不是越多越好，它占用服务器资源，还有超时问题。怎么选？

面试会问：HTTP 长连接和 TCP 长连接是什么关系？keep-alive 由谁控制？长连接适合什么场景？长连接过多会导致什么问题？

### 底层发生了什么？

TCP 连接建立需要三次握手，关闭需要四次挥手。对于高频请求来说，这个开销不可忽略。HTTP/1.0 默认短连接，每次请求后关闭 TCP。HTTP/1.1 默认开启 keep-alive：一个 TCP 连接可以复用多次，连续发送多个请求和响应。

`Connection: keep-alive` 是协商字段，客户端和服务器都要同意才能保持连接。服务器通常还会设置 `Keep-Alive: timeout=60, max=100`，表示连接最多保持 60 秒，最多复用 100 次请求。

但长连接也会带来问题：

- **资源占用**：每个连接都要消耗服务器内存和文件描述符。几万条长连接可能把服务器拖垮。
- **超时管理**：连接空闲太久，防火墙或 NAT 设备会把它断开。客户端需要心跳机制来保持连接。
- **队头阻塞**：HTTP/1.1 长连接上一个请求没响应，后面的请求必须排队。HTTP/2 通过多路复用解决了这个问题。

适用场景：

- **短连接**：一次性请求、低频交互、无状态 API。
- **长连接**：高频请求、实时通信、推送服务、RPC 调用、WebSocket。

### 大白话解释

短连接就像你每次打电话都拨号、说完挂断。长连接就像你和朋友保持通话，说完一件事先别挂，接着说下一件事。显然，频繁联系时长连接更省时间。但如果你只占线不讲话，对方（服务器）的资源就被白白浪费了，所以对方会设置"多久不说话就自动挂断"。

### 核心启示

连接复用是性能优化，但**复用是有成本的**。做架构设计时，要考虑连接数、连接时长、资源消耗、超时策略之间的平衡。盲目开长连接可能压垮服务器；该复用的时候短连接又浪费 RTT。

## Q29 · 网络分区——为什么分布式系统会出现"脑裂"？

![配图](images/chapter2/q29-network-partition.png)



### 你有没有思考过这么一个问题？

一个数据库集群本来有 5 个节点，因为网络故障，节点 A、B 和节点 C、D、E 互相 ping 不通。两边都以为对方死了，各自继续提供服务。于是同一个用户数据被两边同时修改，导致数据不一致。这就是网络分区导致的**脑裂（Split-Brain）**。为什么分布式系统最怕的不是机器宕机，而是网络故障？

面试会问：CAP 定理里 P 是什么？脑裂怎么解决？ZooKeeper / etcd 怎么保证不发生脑裂？多数派（Quorum）机制是什么？

### 底层发生了什么？

网络分区（Network Partition）是指分布式系统中的节点因为网络故障被分成两个或多个互相无法通信的子集。每个子集里的节点以为自己还在正常工作，不知道其他子集也在运行。

在分布式系统中，网络分区比单机故障更棘手：

- 机器宕机时，你知道它死了，可以把它剔除。
- 网络分区时，机器没死，只是互相联系不上。两边都可能认为自己是"大多数"，继续提供服务。

这就导致**脑裂**：一个集群分裂成两个或多个独立集群，各自做出不一致的决策。比如数据库主从架构中，两个节点都以为自己是主节点，各自接受写请求，数据分叉。

解决脑裂的经典思路是**多数派（Quorum）**。只有拿到超过半数节点承认的节点，才能成为主节点或继续服务。比如 5 个节点的集群，至少需要 3 个节点同意。如果网络分成 2 人和 3 人两组，2 人那组凑不够多数派，必须停止服务；3 人那组继续。这保证任何时候最多只有一个分区能写数据。

ZooKeeper 和 etcd 使用 ZAB（ZooKeeper Atomic Broadcast）和 Raft 协议，都依赖 Quorum 机制来避免脑裂。它们要求读写操作都需要半数以上节点确认。

Redis 的 Sentinel 早期发生过脑裂问题：两个 Sentinel 因为网络分区同时认为主节点挂了，各自把不同从节点提升为主节点。Redis 后来引入了配置版本号、min-slaves 等机制来缓解，但 Redis 集群在强一致性上仍然不如 etcd 等 CP 系统。

### 大白话解释

一个团队本来有一个组长（主节点）。某天办公室断网，A 房间和 B 房间互相联系不上。A 房间说"组长可能挂了，我们重新选一个"；B 房间也这么想。结果两边各选了一个组长。等网络恢复，团队就有两个组长，指令互相冲突。

多数派机制就是规定：新组长必须得到超过一半人投票。如果 A 房间只有 2 人，B 房间有 3 人，A 房间凑不够半数，就不能选新组长。B 房间可以选，但等网络恢复后，A 房间必须承认 B 房间的组长。

### 核心启示

网络分区是分布式系统的**第一性难题**：只要网络不可靠，就不可能同时保证强一致性和高可用。设计分布式系统时，必须提前决定分区发生时的行为——是选一致性（拒绝服务）还是选可用性（允许不一致）。这个选择没有标准答案，取决于业务场景。
