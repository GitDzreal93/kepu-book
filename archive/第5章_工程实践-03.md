## Q74 · 设计RESTful API——资源建模、分页、幂等

![配图](images/chapter5/q74-restful-api.png)

### 你有没有思考过这么一个问题？

你接手一个老系统，看到 API 长这样：`POST /api/doStuff?action=createOrder&data=...`。另一个 API 是 `GET /api/getOrder?id=123`。还有 `POST /api/orderAction` body 里是 `{"action": "cancel", "orderId": 123}`。

你不知道每个 API 该用什么 HTTP 方法、参数放哪里、返回什么格式。你必须翻文档才知道每个接口干什么——而文档还经常过期。

你有没有想过：为什么有些 API 你不看文档就能猜出怎么用？比如 `GET /users/123/orders` ——你一看就知道是"获取用户 123 的订单列表"。这种"自解释"的 API 是怎么设计出来的？

### 底层发生了什么？

**REST（Representational State Transfer，表述性状态转移）** 是 Roy Fielding 在 2000 年博士论文中提出的架构风格。RESTful API 的核心思想是：把系统中的"资源"映射到 URL，用 HTTP 方法表达对资源的操作。

**资源建模（Resource Modeling）**

REST 的核心是"资源"——系统中的名词。URL 是资源的地址。

```
/users          → 用户集合
/users/123      → 特定用户
/users/123/orders → 用户 123 的订单集合
/orders/456     → 特定订单
```

URL 表达的是"资源是什么"，不是"对资源做什么"。

**HTTP 方法表达操作**

| HTTP 方法 | 含义 | 幂等性 | 示例 |
|-----------|------|--------|------|
| GET | 获取资源 | 幂等 | `GET /users/123` |
| POST | 创建资源 | 不幂等 | `POST /users` body={name:"Alice"} |
| PUT | 替换资源（整体） | 幂等 | `PUT /users/123` body={name:"Bob",age:25} |
| PATCH | 修改资源（部分） | 不幂等 | `PATCH /users/123` body={age:26} |
| DELETE | 删除资源 | 幂等 | `DELETE /users/123` |

**幂等性（Idempotency）**：同一个请求执行一次和执行多次，效果相同。GET 是幂等的（获取数据不影响状态）；POST 不是幂等的（执行两次创建两个资源）；DELETE 是幂等的（删除同一个资源两次，第二次没影响）。幂等性对重试很重要——网络超时后重试幂等接口是安全的，重试非幂等接口可能产生重复。

**分页（Pagination）——两种模式**

当资源集合很大时（如百万订单），不能一次全返回——需要分页。

1. **Offset 分页（偏移分页）**：`GET /orders?page=3&page_size=20` → 跳过前 40 条，取 20 条。实现简单（`LIMIT 40, 20`），但有问题：如果在第 2 页和第 3 页之间有新数据插入，第 3 页可能重复显示第 2 页的数据——数据移动导致重复或遗漏。

2. **Cursor 分页（游标分页）**：`GET /orders?cursor=abc123&page_size=20` → 返回游标 abc123 之后的 20 条，并返回新的游标。基于排序字段（如 ID 或时间戳）的"位置"，不受数据插入影响。适合数据量大且实时变化的场景（如 Twitter 的 timeline）。

**幂等性设计——防止重复创建**

用户点了"下单"按钮，请求发出去了但网络超时——用户不知道请求到底成功了没有。用户又点了一次。如果两次请求都创建了订单，用户就被扣了两次钱。

**Idempotency Key（幂等键）**：客户端为每个"创建"操作生成一个唯一 ID（如 UUID），放在请求头 `Idempotency-Key: 7c8f...` 中。服务器收到后：

1. 第一次请求：处理业务，把结果和 Key 一起存入"幂等表"。
2. 第二次请求（相同 Key）：查幂等表发现已处理过——直接返回第一次的结果，不重复执行。

**API 版本管理**

一旦 API 发布，调用方就依赖了当前的行为。修改 API（改字段名、改返回格式）会破坏所有调用方。解决方案：URL 版本化。

```
/v1/users/123   → 旧版本
/v2/users/123   → 新版本
```

新版本上线后，旧版本继续维护一段时间（通常是 6-12 个月），给调用方迁移时间。然后废弃旧版本。

**错误处理——用 HTTP 状态码**

不要所有错误都返回 `200 OK` + `{"error": "..."}`。用 HTTP 状态码表达语义：

- `200 OK`：成功
- `201 Created`：资源创建成功（POST 的响应）
- `400 Bad Request`：请求格式错误（参数缺失、格式不对）
- `401 Unauthorized`：未认证（没带 Token 或 Token 无效）
- `403 Forbidden`：已认证但无权限
- `404 Not Found`：资源不存在
- `409 Conflict`：资源冲突（如重复创建）
- `429 Too Many Requests`：限流
- `500 Internal Server Error`：服务器内部错误

**设计原则总结：**

1. URL 用名词复数（`/users` 不是 `/user` 不是 `/getUsers`）
2. HTTP 方法表达操作（GET 获取、POST 创建、DELETE 删除）
3. 参数用 query string（过滤、排序、分页），body 只用于创建/修改的资源数据
4. 返回用标准 HTTP 状态码
5. POST/PATCH 用 Idempotency-Key 保证幂等
6. URL 版本化，给迁移时间

### 大白话解释

好的 API 设计就像好的餐厅菜单：

- **资源命名** = 菜名清晰。你看到"宫保鸡丁"就知道是什么，不需要问服务员"这道菜是什么"。`/users/123/orders` 一看就是"用户 123 的订单"。
- **HTTP 方法** = 动词和菜名的配合。你不会说"我要一个查看宫保鸡丁"——你说"给我看宫保鸡丁的菜单"（GET），"来一份宫保鸡丁"（POST 创建），"把宫保鸡丁换成不辣的"（PATCH 修改）。
- **分页** = 一页不放 100 道菜。先给你前 20 道，你说"下一页"再看后面的。Offset 分页的问题是：如果厨房突然加了一道新菜在前面，你翻到第二页时会发现第一页的最后一道菜又出现了——数据移动了。Cursor 分页是"从这道菜往后看"，不受新菜插入影响。
- **幂等** = 点了两次同一道菜不会上两份。你不确定第一次点了没有，又点了一次——服务员记住了你已经点过，不会重复下单。
- **错误码** = 不用"出错了"三个字敷衍。200 = "上菜了"，400 = "你点的不在菜单上"，401 = "你还没买单不能点菜"，429 = "你点太快了，等一下"。

### 核心启示

API 设计不是"选 GET 还是 POST"——是**让调用方不需要读文档就能猜出怎么用。** 一致性比灵活性更重要。一个 API 如果需要你解释三遍才能让人理解——不是听众的问题，是 API 设计的问题。

## Q75 · 软件基准测试为什么常不准？

![配图](images/chapter5/q75-benchmark.png)

### 你有没有思考过这么一个问题？

你写了一段性能测试代码，测出"这个函数执行只需要 0.3 纳秒"。你很高兴——0.3 纳秒快得不可思议。但你的同事说"不可能，CPU 一次缓存访问都要 1 纳秒"。你重新跑了一遍——这次变成了 50 纳秒。你又跑了一遍——100 纳秒。三次结果完全不同，没有一个是对的。

你有没有想过：为什么你写的基准测试这么不可靠？你到底测了什么——是代码的真实性能，还是"你的测试方法"的性能？

### 底层发生了什么？

软件基准测试的"不准"不是随机的——是系统底层机制刻意"欺骗"了你的测试。主要有三个陷阱：

**陷阱一：JIT 预热（Just-In-Time Compilation Warmup）**

JVM（Java Virtual Machine，Java 虚拟机）、V8（JavaScript 引擎）、PyPy（Python JIT 编译器）等运行时不是一开始就编译成最优机器码的。它们采用 JIT 编译策略：

- 前几次执行时：解释器逐行解释执行——慢。
- 执行了 N 次后（热点检测）：JIT 编译器把这段代码编译成机器码——快。
- 执行了更多次后：JIT 基于运行时反馈进一步优化（内联、逃逸分析）——更快。

所以如果你只跑 10 次就测——你测的是"解释执行"的速度，不是"优化后"的速度。这就是"预热"问题——你需要先空跑几百次让 JIT 完成编译，然后才开始计时。

**陷阱二：GC 暂停（Garbage Collection Pause）**

你在测试一个分配了 100MB 内存的函数。JVM 在你测量的过程中触发了一次 GC（垃圾回收），暂停了 50ms。你的测试结果显示"函数执行 50ms"——但实际上函数本身只需要 0.1ms，另外 49.9ms 是 GC 暂停。

GC 暂停是"Stop-The-World"——整个应用暂停，GC 线程独占 CPU。如果你的测试恰好跨了一次 GC，结果完全不可信。

**陷阱三：死代码消除（Dead Code Elimination）**

这是最隐蔽的陷阱——编译器发现你的测试代码"算了结果但没用"，于是直接把整段计算删了。

```java
// 你以为在测试 calculateHash 的性能
long start = System.nanoTime();
long result = calculateHash(data);
long end = System.nanoTime();
System.out.println(end - start);
// 编译器发现 result 从未被使用 → 把 calculateHash 调用删掉
// 你的测试结果接近 0 纳秒 → 其实什么都没做
```

JIT 编译器（特别是 C2 Compiler）会做积极的优化——如果计算结果没有被"消费"（写入到文件、网络、volatile 变量），编译器认为这段计算没有副作用，直接删掉。

**正确的基准测试方法——JMH（Java Microbenchmark Harness）**

JMH 是 OpenJDK 出品的微基准测试框架，专门解决上述三个问题：

1. **预热轮次**：JMH 先跑 N 轮不计时（默认 5 轮 × 1 秒），让 JIT 完成编译。
2. **多轮测量取统计**：跑 M 轮（默认 5 轮 × 1 秒），取平均值和误差范围。
3. **防止死代码消除**：JMH 提供 `Blackhole` 类来"消费"计算结果，告诉编译器"这个结果被使用了"。
4. **控制 GC**：JMH 可以在每轮测量间强制 GC，报告 GC 次数和耗时。

**其他陷阱：**

- **CPU 缓存影响**：你的测试数据如果在 L1 缓存里（纳秒级访问），和从内存读（百纳秒级）速度差 100 倍。测试用小数据（全在缓存里）和真实场景（数据量大到进不了缓存）结果完全不同。
- **分支预测**：如果测试数据有规律（如总是 if true），CPU 分支预测器几乎 100% 命中——速度很快。真实数据随机时分支预测失败——性能下降。
- **操作系统调度**：测试过程中操作系统可能把你的线程切换走了（去做其他事），过了一会儿又切回来。你测到的时间包含了"被切走"的等待时间——不是真正的执行时间。

### 大白话解释

测一个跑步运动员的速度：

- **JIT 预热问题** = 他刚起床，穿着拖鞋，没热身，你测出来的速度很慢。这不是他的真实能力——是"刚起床"的状态。你得让他热身跑几圈后才开始计时。
- **GC 暂停问题** = 他跑到一半，鞋带松了停下来系鞋带。你测出来的时间包含了系鞋带的时间——这不是跑步的时间，是"跑步 + 系鞋带"的时间。
- **死代码消除问题** = 你让他跑了一圈，但没在终点放计时器。他跑到终点后——"反正也没人看"，JIT 就直接让他跳过这一圈。你以为他跑了，其实他没跑。

JMH 就像一个专业的裁判——让他先热身（预热）、确保鞋带系好（控制 GC）、在终点放计时器（防止死代码消除）、多跑几圈取平均（多轮测量）。

### 核心启示

看基准测试报告时先问三个问题：**你控制了 JIT 预热吗？控制了 GC 暂停吗？确认了死代码没被消除吗？** 如果三个都做不到，数字再漂亮也没有意义。基准测试不是"跑一下看多快"——是一门需要控制大量变量的实验科学。手写循环测性能等于用尺子量光速——工具不对。

## Q76 · 需求文档为什么永远写不清楚？

![配图](images/chapter5/q76-requirements.png)

### 你有没有思考过这么一个问题？

产品经理给你一份需求文档，上面写着"用户可以搜索商品"。你做了个搜索框，输入关键词返回结果。上线后产品说"不对，搜索结果应该按相关度排序"。你加了排序。产品又说"不对，应该支持按价格、销量筛选"。你又加了筛选。产品又说"不对，搜索时应该自动纠正常见拼写错误"。

你怒了："你需求文档里怎么不写清楚？" 产品也委屈："我以为'搜索商品'这个词的意思就是包含排序、筛选、纠错的——这还需要写吗？"

你有没有想过：为什么需求文档永远写不清楚？是产品经理的能力问题，还是这件事本身就是不可能的？

### 底层发生了什么？

需求文档"写不清楚"的根本原因不是"没写好"——是**隐性知识（Tacit Knowledge）的存在**。

隐性知识是波兰哲学家 Michael Polanyi 提出的概念："我们知道的远比我们能说出来的多。" 产品经理脑中有大量"不言自明"的假设——他觉得"搜索商品"当然包含排序、筛选、纠错，因为这些在他的经验里是搜索的基本要素。但他没有把这些假设写出来，因为他甚至没意识到这些是"假设"——对他来说是"常识"。

但开发者的"常识"和产品经理的"常识"不同。开发者理解的"搜索商品"是"输入关键词 → 返回匹配结果"——最小实现。产品经理理解的是"完整的搜索体验"。

**隐性知识的三个来源：**

1. **领域知识**：产品经理在这个行业干了十年，很多东西他觉得"理所当然"。但开发者可能刚接触这个领域。
2. **用户习惯假设**：产品经理假设"用户都知道搜索结果可以筛选"——但新用户可能不知道。
3. **系统上下文**：产品经理知道"我们的搜索服务已经支持纠错了，所以搜索功能自然包含纠错"——但开发者不知道这个上下文。

**为什么"一次性写清楚"是低效的？**

试图在需求文档中穷举所有场景、所有边界条件、所有异常处理——这是"瀑布模型"（Waterfall Model）的思路。问题在于：

- **假设太多**：一个功能可能有 50 个边界场景，逐一写出来需要几天——而且大部分场景最终不会发生。
- **假设在变**：写文档时的假设，到开发时可能已经变了。市场变了、竞品出了新功能、用户反馈了新需求。
- **理解偏差**：即使文档写了 50 个场景，开发者的理解和产品经理的理解可能还是不同。

**敏捷的答案：用频繁交付暴露假设**

敏捷开发（Agile Development）的核心思路不是"把需求写得更详细"——是**"用可工作的软件暴露隐性知识"**。

- 产品做一个最小版本（MVP）→ 上线 → 看真实用户行为 → 发现"原来用户确实需要筛选" → 加上 → 再看 → 发现"原来排序逻辑不对" → 调整。
- 每次迭代都在"暴露假设"——把产品经理脑中的隐性知识变成显性的、可见的、可验证的需求。
- 这种方式的成本远低于"花一周写需求文档 → 花两周开发 → 上线发现全错了 → 推翻重来"。

**但敏捷 ≠ 没文档——这些必须记录：**

1. **接口契约**：API 的请求/响应格式、字段含义、错误码。这是代码的"合同"——调用方依赖这个契约，改了就违约。
2. **设计决策**：为什么选 A 不选 B？为什么用游标分页不用偏移分页？这些决策的"为什么"如果不记录，后人会重新踩坑。
3. **状态机**：订单从"待支付"→"已支付"→"已发货"→"已完成"，每个状态的转换条件、触发事件、不允许的转换。状态机必须记录，因为它定义了业务的核心规则。

**需求文档的正确期望：**

需求文档不是"把所有事情说清楚"——是"让双方在同一个频道上对话"。它提供一个起点，剩下的通过迭代和反馈来补全。试图写一份"完美文档"然后"严格按文档实现"——这是 1970 年代的瀑布模型思维，在当今快速变化的市场中已经行不通了。

### 大白话解释

你跟你妈说"买个西瓜回来"。她买回来了——你说"不对，我要无籽的。" 她说"你没说啊。" 你说"西瓜不是应该买无籽的吗？"

这里的问题不是"你没说清楚"——是"你以为对方知道的背景信息，对方不一定知道"。你的隐性知识（无籽西瓜更好吃）对你来说是常识，但你妈不一定知道。

你可以写一份"买西瓜需求文档"：品种（8424）、重量（5-8斤）、成熟度（拍一下声音沉闷）、种子（无籽）、温度（冰镇）。但你妈可能还是买错——因为她不知道"拍一下声音沉闷"是什么样的声音。

更好的方式：你跟你妈一起去买一次西瓜，你挑给她看 → 她学会了 → 下次她自己去买就对了。这就是敏捷——通过实际操作来传递隐性知识，比写文档更有效。

### 核心启示

需求沟通的核心难题是**隐性知识——领域专家脑中有大量"不言自明"的假设。** 迭代式暴露假设比一次性文档穷举所有场景成本更低。但接口契约、设计决策、状态机必须记录——这些是"不可通过迭代发现的"，必须在设计时确定。

## Q77 · 可观测性三支柱——Metrics、Tracing、Logging

![配图](images/chapter5/q77-observability.png)

### 你有没有思考过这么一个问题？

你的系统有 50 个微服务。一个用户反馈"下单很慢，等了 10 秒才返回"。你打开了 Grafana（监控面板），看到订单服务的 P99 延迟确实从 200ms 飙到了 5000ms。但为什么？是数据库慢了？还是下游服务慢了？还是网络问题？

你翻了订单服务的日志——日志只记录了"收到请求"和"返回响应"，中间发生了什么没有记录。你又翻了支付服务的日志——但你不知道用户的请求经过了哪些服务、按什么顺序经过的。50 个服务的日志摆在你面前，你不知道哪个和这个请求有关。

你有没有想过：如果有一种手段，能让你一眼看到"这个请求经过了哪些服务、每一步花了多少时间、在哪个环节卡住了"——排查问题不就快了吗？

### 底层发生了什么？

**可观测性（Observability）** 的概念来自控制论——一个系统的可观测性是指"从系统的外部输出推断系统内部状态的能力"。在软件领域，它意味着你能在不修改代码的情况下，通过系统的输出来理解系统内部在发生什么。

可观测性有三大支柱（Three Pillars of Observability），它们各自回答不同的问题：

**Metrics（指标）——回答"有没有问题？趋势是什么？"**

Metrics 是聚合的数值数据，按时间序列存储。比如：

- `http_request_duration_seconds{service="order", endpoint="/api/orders", quantile=0.99}` = 5.2（P99 延迟 5.2 秒）
- `http_requests_total{service="order", status="500"}` = 23（500 错误 23 次）
- `cpu_usage_percent{host="server-42"}` = 92（CPU 使用率 92%）

- **特点**：聚合、低成本、适合趋势分析和阈值告警。存储成本低（一个指标序列每分钟只存一个数值）。
- **工具**：Prometheus（采集+存储）、Grafana（可视化）
- **局限**：Metrics 告诉你"订单服务 P99 延迟 5 秒"——但不告诉你"哪一笔请求慢了？为什么慢？"。

**Tracing（链路追踪）——回答"问题在哪一步？"**

Tracing 记录一次请求在分布式系统中的完整路径。一次请求经过多个服务，每个服务处理的时间、调用的下游服务、在每个环节的耗时——全部记录在一个"Trace"中。

- **TraceID**：每次请求生成一个唯一 ID，贯穿所有经过的服务。不管请求经过 5 个还是 50 个服务，通过 TraceID 可以把它们串起来。
- **Span**：Trace 中的一个操作单元。比如"调用数据库"是一个 Span，"调用支付服务"是另一个 Span。每个 Span 记录开始时间、结束时间、操作名称、标签。
- **Trace 树**：Spans 之间有父子关系——"调用订单服务"是父 Span，"调用数据库"和"调用库存服务"是子 Span。最终形成一棵调用树。

```
[Order Service: 5000ms]
  ├── [DB Query: 4500ms] ← 瓶颈在这！
  ├── [Inventory Service: 200ms]
  └── [Payment Service: 300ms]
```

- **工具**：Jaeger、Zipkin、SkyWalking
- **局限**：Tracing 采样率通常不是 100%（存储成本高），高频请求只采样 1%-10%。所以"被采样的请求"不一定覆盖了你关心的那次请求。

**Logging（日志）——回答"具体发生了什么？"**

Logging 记录离散事件的详细信息。比如"用户 12345 下单，商品 ID 67890，库存检查通过，支付回调成功，订单 ID ORD-2024-001"。

- **特点**：详细、上下文丰富、适合事后排查。但数据量大、搜索成本高。
- **结构化日志**：用 JSON 格式记录日志，方便按字段过滤。`{"timestamp":"2024-01-01T10:00:00","level":"ERROR","service":"order","trace_id":"abc123","message":"DB timeout","query":"SELECT * FROM orders WHERE id=123"}`
- **TraceID 关联**：日志中必须带 TraceID——这样你可以从 Tracing 跳到 Logging，看到具体那一笔请求的详细日志。
- **工具**：ELK Stack（Elasticsearch + Logstash + Kibana）、Loki

**三者如何协同工作：**

1. Metrics 告诉你"有问题"——P99 延迟飙升了
2. Tracing 告诉你"问题在哪"——DB 查询占了 90% 的时间
3. Logging 告诉你"具体什么"——DB timeout，慢查询 SQL 是什么

**关键设计原则——让系统"可被观测"：**

- **每条日志带 TraceID**：没有 TraceID 的日志无法和链路追踪关联——等于没用。
- **Metrics 带维度标签（Labels）**：`http_request_duration` 没有标签 → 只能看全局。加上 `{service, endpoint, method, status}` 标签 → 可以下钻到"哪个服务的哪个接口的哪种请求慢了"。
- **结构化日志**：非结构化日志（纯文本）难以搜索和过滤。JSON 格式的结构化日志可以按字段查询。

**可观测性 vs 监控——关键区别**

- **监控（Monitoring）**：只能回答"已知的已知"（CPU > 90% 了吗？）。你预先定义了"什么算异常"，系统帮你检查。
- **可观测性（Observability）**：能回答"未知的未知"（为什么 P99 突然高了？）。你没有预先定义这个问题，但通过 Metrics + Tracing + Logging 可以临时探索和排查。

监控是"你问了什么，它答什么"——可观测性是"你没问的，它也能告诉你"。

### 大白话解释

想象查案：

- **Metrics** = 警局统计看板："本周入室盗窃案 ↑15%" → 知道有问题
- **Tracing** = 嫌疑人行动轨迹图：从哪条路进的门 → 在哪个房间待了多久 → 从哪走的 → 知道问题出在哪个环节
- **Logging** = 案发现场监控录像：详细记录了每一秒发生了什么 → 知道具体怎么做的

缺一个都有盲区：只有 Metrics（知道有问题但不知道在哪）→ 只能干着急。只有 Tracing（知道在哪但不知道细节）→ 猜。只有 Logging（知道细节但不知道是哪次的）→ 大海捞针。

三者通过 TraceID 关联——就像案号把统计报表、行动轨迹、现场监控串联起来。

### 核心启示

可观测性不是"买一套监控工具"——是**让你的系统在设计时就考虑了"怎么被观测"。** 日志没有 TraceID = 查不了链路。Metrics 没有维度标签 = 只能看全局不能下钻。可观测性是在代码里写出来的，不是靠工具装出来的。当你的系统出了问题时，可观测性决定了你是"5 分钟定位问题"还是"5 小时翻日志"。
