## Q30 · 设计一个短链接系统——从 Demo 到能扛千万级访问量，要做哪些关键决策？

![配图](images/chapter3/q30-short-url.png)



### 你有没有思考过这么一个问题？

你在微博上看到一条 `t.cn/abc123`，点击后跳到了一篇万字长文。这么短的 URL 是怎么对应到长链接的？如果面试官让你"设计一个短链接系统"，你除了"把长链接哈希成短码"，还能说出什么？

这个问题真正的考察点不是你会不会写，而是你在**高并发、低冲突、高性能、防滥用**四个维度上有没有完整的思考。一个能扛千万级访问量的短链接系统，和课堂作业里的哈希表完全是两回事。

### 底层发生了什么？

设计短链接系统，核心要解决四个问题：短码怎么生成？怎么存？怎么查？怎么防止被刷？

**1. 短码生成：唯一性是根基**

最简单的想法是：把长链接做 MD5 或 SHA256，取前 6 位。但哈希会碰撞。6 位 62 进制（0-9、a-z、A-Z）最多只有 62^6 ≈ 568 亿个组合，看起来很多，但生日悖论告诉我们：当你生成 15 万个短码时，碰撞概率就超过 50% 了。对于真实业务，这种方案不可用。

更可靠的方案有两种：

- **自增 ID + Base62 编码**：数据库自增主键，比如 100000000，转成 62 进制得到 "abc123"。优点是绝对唯一、顺序增长；缺点是短码有规律，可能被遍历，而且依赖数据库的自增能力。
- **分布式 ID 生成器 + Base62 编码**：用 Snowflake、数据库号段、Leaf 等算法生成全局唯一 ID，再转成 62 进制。Twitter 的 Snowflake 用 64 位整数，包含时间戳、机器 ID、序列号，每秒能生成上百万个唯一 ID，且大致有序。

大厂的短链服务通常用**预生成短码池**：系统提前生成一大批唯一短码放在 Redis 或内存里，写入时直接取一个可用的，避免写入时实时生成带来的竞争。

**2. 存储：短码和长链接的映射**

最常见的存储方式是：

- **关系型数据库（如 MySQL）**：持久化存储。短码建唯一索引，查询时 `SELECT long_url FROM short_url WHERE short_code = 'abc123'`。
- **Redis**：热数据缓存，短码作为 key，长链接作为 value。Redis 查询是 O(1)，能把重定向延迟压到毫秒级。
- **布隆过滤器（Bloom Filter）**：用于判断"这个短码是否已经被占用"。如果你用预生成短码池，这一步可有可无；如果你是实时生成，布隆过滤器可以快速告诉你"这个短码很可能存在"，减少数据库查询。

如果系统每天有上亿条短链生成，MySQL 单表可能扛不住。这时需要**分库分表**：按短码首字符或哈希取模分片。读取时可以按短码路由到对应分片。

**3. 重定向：性能的关键路径**

用户访问 `t.cn/abc123`，服务器要在最短时间内返回 302 重定向到长链接。这个路径上：

- 先查 Redis，命中直接 302。
- Redis 不命中，查 MySQL 或分片库，然后回填 Redis。
- 如果短码不存在，返回 404，避免被恶意请求击穿数据库（可以用布隆过滤器或空值缓存）。

**4. 防滥用与统计**

短链系统经常被用来做钓鱼链接、垃圾传播。所以还要考虑：

- **链接审核**：生成时先过一遍黑白名单、内容安全检测。
- **访问频率限制**：单个 IP 或用户在短时间内生成大量短链要限流。
- **访问统计**：记录点击量、访问地域、设备、来源。可以用消息队列异步落盘，避免影响重定向主链路性能。
- **过期策略**：可以设置短链有效期，过期后自动清理或失效。

### 大白话解释

短链接系统就像城市图书馆的索书号。一本很厚的书（长链接）名字太长，你记不住，图书馆给它一个简短编号（短码）。你去查这个编号，管理员立刻知道书放在哪一排书架，然后告诉你"请去 A 区 3 架"。

但大城市图书馆每天有几万人借书，所以管理员不能每次都去仓库找。热门书放在前台（Redis），冷门书去仓库（MySQL）。同时，图书馆还要防止有人恶意注册一万个编号来占坑（防刷），以及记录每本书被借了几次（统计）。

### 核心启示

短链接系统的本质是**"用空间换简洁"+"用缓存换速度"**。真正难的不是"把长链接变短"，而是：如何在大量写入时保证短码唯一不冲突，如何在大量读取时把延迟压到最低，如何在开放服务中防止被滥用。这个设计模式可以迁移到很多"键-值映射 + 高并发读取"的场景。

## Q31 · 设计一个登录系统——Session、JWT、单点登录，到底怎么选？

![配图](images/chapter3/q31-login-system.png)



### 你有没有思考过这么一个问题？

你登录淘宝后，关闭浏览器再打开，为什么很多时候还是保持登录？你登录了企业邮箱，为什么点进公司 OA 不用重新输密码？这些体验背后，有 Session、Cookie、JWT、SSO（Single Sign-On，单点登录）四种技术在不同场景下组合。

面试里最常问的是：Session 和 JWT 有什么区别？各有什么优缺点？单点登录的实现原理是什么？如果用户规模大，登录状态存在哪里最好？

### 底层发生了什么？

**1. Session + Cookie 方案**

用户提交用户名密码，服务器验证通过后：

- 在服务器内存或 Redis 中生成一个 Session 对象，包含用户 ID、角色、过期时间等。
- 把这个 Session 的 ID（Session ID）通过 `Set-Cookie` 返回给浏览器。
- 浏览器之后每次请求都带上这个 Cookie，服务器根据 Session ID 查 Redis，确认用户身份。

优点是：状态存在服务器端，可以随时让某个 Session 失效（比如用户点"退出登录"，或者管理员强制下线）。缺点是：服务器集群环境下需要共享 Session，通常用 Redis 集中存储；另外，浏览器端无法自主解析用户信息。

**2. JWT（JSON Web Token）方案**

JWT 把用户信息直接编码成一个 token，结构是 `Header.Payload.Signature`。服务器用私钥签名后发给客户端，客户端之后每次请求都带上这个 token。服务器用公钥验证签名即可，不需要查数据库。

JWT 的优点是：无状态、可扩展、跨服务传递方便。缺点是：一旦签发，服务器无法主动撤销（除非维护一个黑名单，或者设置短有效期 + 刷新机制）。如果 token 泄露，在过期前攻击者都能使用。

所以 JWT 通常设置很短的有效期（比如 15 分钟），配合 Refresh Token 使用。Access Token 负责频繁访问，Refresh Token 负责换取新的 Access Token，且 Refresh Token 的存储和撤销更严格。

**3. 单点登录 SSO**

SSO 解决的是：你登录一次，就能访问多个互相信任的系统。最典型的实现是 CAS（Central Authentication Service）协议和 OAuth 2.0 / OIDC（OpenID Connect，基于 OAuth 2.0 的身份层）。

以 OAuth 2.0 授权码模式为例：

- 用户访问应用 A，应用 A 把用户重定向到认证中心。
- 用户在认证中心登录，认证中心生成一个 authorization code 并重定向回应用 A。
- 应用 A 拿 authorization code 向认证中心换取 Access Token 和 ID Token。
- 应用 A 用 ID Token 确认用户身份。

OIDC 在 OAuth 2.0 基础上定义了 ID Token 的格式（本质上是一个 JWT），明确解决了"用户是谁"的问题，而不是只解决"你能访问什么资源"。

**4. 选型建议**

- 内部管理系统、需要强会话控制：Session + Redis。
- 无状态微服务、移动端 API：JWT + Refresh Token。
- 多系统统一登录：SSO + OAuth 2.0 / OIDC。
- 超大用户规模：认证中心独立，登录状态用分布式存储，Token 分层管理。

### 大白话解释

Session 像你去健身房办了一张会员卡，卡上只有卡号，你的姓名、照片、有效期都存在健身房前台（服务器）。每次你去，前台刷卡查档案。

JWT 像一张带防伪水印的演唱会门票，门票上直接印着座位号和你的头像。检票口只需要验证水印真假，不用再查后台系统。

SSO 像"一卡通"：你刷一次卡，食堂、图书馆、宿舍门禁都能认。

### 核心启示

登录系统的核心是**身份凭证的颁发、验证和撤销**。Session 好控制但依赖集中存储；JWT 去中心化但难以撤销；SSO 解决多系统信任问题。实际系统中，往往是三者混用：认证中心用 Session 或 JWT，业务系统用 JWT 或 Session，根据 revoke 需求和安全级别做分层。

## Q32 · OTA 升级——为什么手机更新系统不能直接把旧版本覆盖掉？

![配图](images/chapter3/q32-ota-upgrade.png)



### 你有没有思考过这么一个问题？

你的手机提示"系统有更新"，下载完安装包后，它不会让你直接删掉旧系统再装新系统。相反，它通常会把新系统写到另一个分区，重启后切换到新分区。如果更新失败，还能回退到旧版本。这就是 OTA（Over-The-Air，空中下载）升级机制。

面试会问：OTA 升级为什么大多用 A/B 分区？差分包是怎么做的？升级失败怎么回滚？OTA 设计中哪些环节最容易出安全问题？

### 底层发生了什么？

OTA 升级的本质是：**在不破坏用户当前使用的前提下，把新固件或软件安全地部署到设备上。** 它涉及三个核心问题：怎么下载？怎么安装？怎么保证安全？

**1. A/B 分区（无缝升级）**

传统做法是单分区：下载新系统 → 替换旧系统 → 重启。问题是：如果替换过程中断电或出错，设备可能变砖。

A/B 分区方案（也叫无缝升级，Seamless Updates）把存储分成两套系统分区：A 和 B。当前运行的是 A，新系统下载到 B，写入完成后重启切换到 B。如果 B 启动失败，自动回退到 A。

Android 从 7.0 开始推广 A/B 分区。它的代价是占用更多存储空间，但大大提高了升级的可靠性。

**2. 差分包（Delta Update）**

完整系统包可能有几个 GB，如果每次都下载完整包，浪费流量和时间。差分包只包含新版本和旧版本之间的差异，通常小得多。生成差分包的方法包括：

- **二进制差分**：如 bsdiff，对可执行文件或压缩包做二进制差分，生成 patch，设备端用 bspatch 合并。
- **文件级差分**：只替换有变化的文件，未变化的文件保留。
- **块级差分**：对文件系统块做差异比较，只传输变化的块。

差分包需要和设备的当前版本精确匹配。如果用户版本五花八门，服务器要维护很多个差分包组合，管理上更复杂。

**3. 安全校验**

OTA 包必须防篡改：

- **签名验证**：OTA 包用厂商私钥签名，设备用内置公钥验证签名，确保包来自官方。
- **哈希校验**：下载完成后检查文件哈希，防止下载过程中被损坏或替换。
- **启动链验证**：从 bootloader 到 kernel 到 system 分区，每一层都验证下一层的签名。这就是 Android 的 Verified Boot。

**4. 状态机和回滚**

OTA 升级通常有状态机：检测更新 → 下载 → 校验 → 安装 → 重启 → 启动成功判定 → 标记新系统可用。如果在任何一步失败，都要能回退。启动成功判定通常由系统多次成功启动后确认，否则下一次重启仍回旧系统。

### 大白话解释

OTA 升级就像给行驶中的汽车换引擎。传统做法是：停车，把旧引擎拆下来，装新引擎，再启动。如果新引擎有问题，车就趴窝了。A/B 分区就像汽车有两套引擎舱，旧引擎还在工作，新引擎在旁边装好，装好后一键切换。如果新引擎不好使，立刻切回旧引擎。

差分包则是：不是给你寄一整个新引擎，而是只寄"和旧引擎不一样的那几个零件"。

### 核心启示

OTA 升级的核心设计是**可回滚的原子替换**。任何不能回滚的更新都是危险的。设计发布系统时，都要有"金丝雀发布"、"蓝绿部署"、"快速回滚"的思维，这和 A/B 分区是同一个底层逻辑：先验证，再切换，失败能回退。

## Q33 · 设备影子模型——为什么物联网设备不可能一直在线？

![配图](images/chapter3/q33-device-shadow.png)



### 你有没有思考过这么一个问题？

你用手机 App 控制家里的智能灯，灯本身可能睡眼惺忪地连在 WiFi 上，时不时掉线。但你的 App 上却始终能看到灯的状态：开着、亮度 80%、颜色暖黄。这个状态是从哪来的？

答案就是**设备影子（Device Shadow）**。它是 AWS IoT、阿里云 IoT 等平台的核心概念。设备不可能一直在线，但应用端需要随时看到设备最近一次的状态。设备影子就是在云端为每个设备维护的一份"状态副本"。

### 底层发生了什么？

物联网设备通常有资源限制（CPU 弱、内存小、电量有限），而且网络不稳定。它们不能一直连接服务器，甚至为了省电会主动休眠。但云端应用需要随时查询设备状态，也需要随时向设备下发指令。

设备影子模型解决这个矛盾的方式是：**在云端和设备之间建立一个双向同步的状态缓冲区。**

每个设备在云端有一个 JSON 格式的影子文档，通常包含三个部分：

- **reported（设备上报状态）**：设备连上来时告诉云端的当前状态。比如 `{"light": "on", "brightness": 80}`。
- **desired（云端期望状态）**：应用端希望设备达到的状态。比如 `{"light": "off"}`。
- **metadata（元数据）**：版本号、时间戳、状态同步时间等。

工作流程：

1. 设备在线时，把自身状态上报到云端，更新 reported。
2. 应用端通过 API 查询设备影子，看到的是最近一次 reported 状态。
3. 应用端下发指令，修改 desired。比如让灯关掉。
4. 设备下次上线时，从云端拉取 desired，发现 desired 和 reported 不一致，就执行关灯动作，并把新的 reported 上报。

这个模型把"设备在线"和"应用可用"解耦了。即使设备当前不在线，应用也能操作 desired；等设备上线，状态会自动同步。

设备影子通常基于 MQTT 协议。MQTT 是轻量级发布/订阅协议，适合物联网设备。设备通过 MQTT 订阅主题来接收 desired 变化，通过发布消息来更新 reported。

### 大白话解释

设备影子就像你给你室友留了一张便签："出门记得关空调，温度调到 26 度。"你室友（设备）现在不在家，但便签贴在冰箱上（云端）。他回来看到便签，执行，然后把结果告诉你"已关，温度 26"。

没有设备影子，你就必须等到室友在家的那一刻才能告诉他要做什么。有了便签，你们不需要同时在线。

### 核心启示

设备影子模型的本质是**用云端状态副本解决弱网、离线、异步通信的问题**。它把"状态同步"和"控制指令"分开，让设备端和应用端各自按自己的节奏运行。任何需要在"不稳定的终端"和"高可用的服务"之间保持一致的系统，都可以借鉴这个模型。

## Q34 · 设计一个秒杀系统——100 万人抢 1000 张票，系统怎么不崩？

![配图](images/chapter3/q34-seckill.png)



### 你有没有思考过这么一个问题？

双 11 零点、演唱会开票、限量球鞋发售，几万人甚至几十万人同时点击购买。如果服务器不做任何处理，瞬间流量可能直接把数据库打挂。秒杀系统的核心不是"让谁先买到"，而是**"怎么从海量请求里精准过滤掉无效的、重复的、恶意的请求，保护下层系统"**。

面试里会追得很细：秒杀系统怎么限流？怎么防止超卖？库存怎么扣？Redis 和消息队列怎么配合？

### 底层发生了什么？

设计秒杀系统，通常要从前端、网关、服务、数据四个层面层层设防。

**1. 前端层：把请求拦在浏览器里**

- **验证码 / 滑块**：挡住简单脚本，让黄牛机器每秒发 1000 次请求变得困难。
- **按钮置灰**：点击后立即禁用按钮，避免用户狂点。
- **答题 / 排队机制**：让部分请求自然错峰，降低峰值。
- **静态资源 CDN**：活动页面提前缓存到 CDN，不让请求打到源站。
- **URL 动态化**：下单 URL 不是固定地址，而是活动开始前才下发，防止脚本提前准备。

**2. 网关层：把请求拦在入口**

- **限流（Rate Limiting）**：按 IP、用户、接口维度限制请求速率。常用令牌桶（Token Bucket）或漏桶（Leaky Bucket）算法。
- **黑名单**：识别恶意 IP 或账号，直接拒绝。
- **请求去重**：同一个用户的重复请求幂等处理，避免重复扣库存。
- **预热缓存**：活动开始前把商品库存、状态等信息写入 Redis，避免实时查数据库。

**3. 服务层：异步化和库存扣减策略**

秒杀不能只靠数据库扛并发。常见做法是：

- **Redis 预扣库存**：用 Redis 的 `DECR` 命令原子扣减库存。`DECR` 是单线程原子操作，返回剩余库存。如果剩余库存 <= 0，直接返回"已售罄"。
- **消息队列异步下单**：真正创建订单、写数据库的操作通过消息队列异步处理。只有 Redis 扣减成功的请求才进入队列，下游系统按自己能处理的速度消费。
- **令牌池**：提前发放固定数量的购买令牌，用户拿到令牌才能下单。没拿到令牌的直接失败。

**4. 数据层：防止超卖**

超卖是最严重的事故。防止超卖的关键是**扣库存的原子性**。数据库层面可以用：

- **乐观锁**：更新库存时带上版本号，只有版本号匹配才更新成功。`UPDATE stock SET count = count - 1, version = version + 1 WHERE id = 1 AND count > 0 AND version = ?`
- **悲观锁**：`SELECT ... FOR UPDATE`，但并发高时性能差。
- **唯一索引/请求去重表**：防止同一用户重复下单。

真正的大厂秒杀，会把库存拆成多份（分桶），每个桶独立扣减，减少单点竞争。Redis 里的库存扣减成功只代表"你抢到了资格"，最终订单能否成立还要经过异步风控、支付校验。

### 大白话解释

秒杀就像春运抢票。1000 张票，100 万人抢。火车站不能同时给 100 万人办业务，所以：

- 先让大家在网上排队（限流）；
- 防止机器人占着窗口（验证码、风控）；
- 真正出票前先扣一个库存名额（Redis 预扣）；
- 然后把成功抢到名额的人名单交给后台慢慢出票（消息队列异步）；
- 最后数据库确认座位，避免同一张票卖给两个人（防止超卖）。

### 核心启示

秒杀系统的本质是**流量削峰 + 资格筛选 + 异步落单**。它不是在几秒内处理所有请求，而是让绝大多数请求在入口处就快速失败，只让真正"有机会成功"的请求进入核心业务链路。这个思路适用于任何突发流量的保护设计。

## Q35 · 消息队列——为什么 Kafka 被称作"高吞吐量的日志管道"？

![配图](images/chapter3/q35-kafka.png)



### 你有没有思考过这么一个问题？

你每天刷的微博、抖音、淘宝，背后都有无数用户行为需要被记录和分析：谁点赞了、谁下单了、谁看了广告。这些事件如果直接写数据库，数据库会瞬间爆炸。于是它们先被塞进一个"队列"，让消费者慢慢处理。这就是消息队列（Message Queue）。

但消息队列不只有一种。RabbitMQ、Kafka、RocketMQ、Pulsar 各自解决什么问题？为什么 Kafka 特别擅长高吞吐？面试会问：Kafka 的 Topic、Partition、Offset 是什么？为什么 Kafka 读写那么快？Kafka 和 RabbitMQ 怎么选？

### 底层发生了什么？

消息队列的核心作用有三个：**异步解耦、削峰填谷、可靠传输**。

**Kafka 的设计哲学：顺序写磁盘 + 零拷贝**

Kafka 把自己定位为"分布式流处理平台"，而不是传统的消息队列。它的底层设计有几个关键决策：

- **Partition（分区）**：一个 Topic 分成多个 Partition，每个 Partition 是独立的有序日志。Producer 可以并发写入不同 Partition，Consumer 也可以并发读取不同 Partition。分区是 Kafka 横向扩展的核心。
- **顺序写磁盘**：Kafka 把消息追加到文件末尾，而不是随机写。机械硬盘顺序写比随机写快几个数量级，SSD 也有优势。
- **Page Cache + 零拷贝**：Kafka 读取消息时，利用 Linux 的 Page Cache 和 sendfile 零拷贝技术，数据从磁盘到网卡几乎不经过用户态，CPU 拷贝次数极少。
- **Offset 消费机制**：每个消费者组在每个 Partition 上记录一个消费位置（Offset）。消息不会被消费后立即删除，而是保留一段时间（可配置）。这允许消费者回溯、重放数据。
- **多副本（Replication）**：每个 Partition 有多个副本，Leader 负责读写，Follower 同步。Leader 挂了，Follower 可以选举为新 Leader，保证高可用。

**Kafka vs RabbitMQ**

- **Kafka**：高吞吐、持久化、适合日志流、事件流、大数据管道。延迟在毫秒到秒级，不强调实时性。
- **RabbitMQ**：传统消息队列，支持丰富的路由模型（direct、topic、fanout、headers），适合任务队列、RPC、需要复杂路由的场景。延迟更低，但单集群吞吐量不如 Kafka。
- **RocketMQ**：阿里开源，兼顾 Kafka 的高吞吐和 RabbitMQ 的灵活，适合金融级消息。
- **Pulsar**：分层架构（存储和计算分离），多租户友好，适合云原生大规模场景。

**Kafka 为什么快？**

总结一句话：它放弃了传统消息队列的随机读写和复杂路由，用**日志追加写 + 分区并行 + 零拷贝 + 批量压缩**把吞吐做到极致。它不是为每条消息的低延迟设计的，而是为"每秒处理百万条消息"设计的。

### 大白话解释

消息队列就像快递分拣中心。全国各地的包裹（消息）先送到分拣中心，不直接送到你家。分拣中心按地址分类，快递员（消费者）一批批取走派送。高峰期包裹太多，分拣中心也不会崩溃，只是让消费者慢慢消化。

Kafka 这个分拣中心特别之处在于：它不拆开每个包裹重新整理，而是把同一地区的包裹按顺序码成一列（Partition），叉车直接整列拉走（顺序写 + 零拷贝）。所以它能处理天量包裹。

### 核心启示

Kafka 的核心不是"消息队列"，而是**持久化的高吞吐日志流**。如果你的场景是事件采集、日志处理、流式计算、数据管道，Kafka 是首选；如果你的场景是任务调度、RPC 异步、复杂路由，RabbitMQ 或 RocketMQ 更合适。选型之前，先问自己：你更关心吞吐，还是延迟和灵活性？
