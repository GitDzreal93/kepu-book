> 从短链接到秒杀系统、从登录到支付——真正的系统设计不是"用什么技术"，而是"在约束下你怎么做取舍"。

## Q30 · 设计一个短链接系统（像t.cn）——从生成到重定向的每一步

### 你有没有思考过这么一个问题？

"设计短链接系统"是经典面试题。你说"哈希一下，存数据库，查出来重定向"——然后面试官补充：**每天100万条、重定向<10ms、短码全局唯一、不能被遍历。**

你的"哈希一下"瞬间暴露了问题。

### 底层发生了什么？

**问题一：短码怎么生成？**

MD5 取前 7 位 → 7 位 Base62 共 3.5 万亿组合 → 但生日悖论下 **150 万条后碰撞概率 > 50%**。哈希方案有天然碰撞风险。

分布式 ID（Snowflake）→ Base62 编码 → **零碰撞 + 各机器独立生成。** 但可遍历——加 XOR 扰动后外人看到的短码"随机"，内部可还原。

**问题二：存储怎么设计？** 双向映射——短码→长URL（重定向主路径）+ 长URL→短码（去重路径）。Redis O(1) 读热点短码，MySQL 持久化全量，布隆过滤器拦截无效查询。

**问题三：重定向 < 10ms？** 302 临时重定向（不用 301——浏览器缓存后会跳过统计）。CDN 边缘节点——热门短链接缓存在边缘。延迟取决于"用户到边缘"的距离。

**问题四：防滥用。** IP 频率限制（网关层）+ 异步恶意链接检测 + 不可预测短码防枚举。

### 大白话解释

短链接 = 图书馆索书号系统。大部头书（长 URL）分配唯一编号（短码）。读者拿编号秒找到书。但难点不是"编号化"——是**每天百万本新书编号不重、读者拿号取书零等待、编号不能顺着猜。**

### 核心启示

> "看起来简单"的系统，约束一收紧就暴露复杂度。**面试官不是在看你能否写一个能跑的 Demo——是在看你在 ID 生成、缓存策略、安全防护三个维度的取舍逻辑。**

## Q31 · 设计一个千万级用户登录系统——Session vs JWT vs OAuth2.0

### 你有没有思考过这么一个问题？

用户登录后，怎么在后续请求中知道"这个人已经登过了"？存 Session？发 Token？用 OAuth2.0 接第三方登录？三个方案各自适合什么场景？

### 底层发生了什么？

**Session 模式：** 服务端在 Redis/内存存 Session（用户ID + 过期时间），客户端只存 SessionID（Cookie 传递）。优点：**服务端可控**（随时踢人）。缺点：**服务端有状态**——水平扩展需要 Session 共享（Sticky Session 或集中 Redis）。

**JWT（JSON Web Token）模式：** 服务端签发自包含 Token（Header.Payload.Signature），客户端自己保管。优点：**服务端无状态**，易扩展。缺点：**Token 发出后无法主动失效**（除非引入黑名单或短过期 + Refresh Token）。

**OAuth2.0（授权码模式）：** 第三方不触碰用户密码，通过授权服务器中转。适用"用微信登录我的网站"。PKCE 扩展防止授权码被拦截——隐式模式已淘汰。

**安全防御：** 暴力破解 → 频率限制 + 账户锁定。撞库 → 多因素认证。CSRF → SameSite Cookie。

### 大白话解释

- Session = 健身房给你一张临时卡。前台拿卡查登记表——确认你卡有效。卡丢了？前台删掉登记表的记录。但你每次来都要查表。
- JWT = 给你一张"防伪身份证"。上面写着你是谁、有效期。门卫看身份证本身判断——不用查表。但身份证被偷了——门卫拦不住——你只能等它过期或上黑名单。
- OAuth2.0 = 你去朋友的健身房，不用办卡——扫朋友分享的二维码确认是客人 → 给你临时访客卡。

### 核心启示

> 登录系统的选择，本质是**在"服务端可控"和"系统可扩展"之间取舍。** Session 给你完全控制但牺牲扩展性；JWT 给你无限扩展但牺牲即时失效能力。**选择不是"哪个更好"，是"你的场景更需要控制还是更需要扩展"。**

## Q32 · 设计App的OTA升级系统——面向千万台设备，怎么保证不"变砖"？

### 你有没有思考过这么一个问题？

手机弹窗"有新的系统更新，3GB"。你点"下载并安装"，20 分钟后手机重启，系统更新好了。但如果更新中途断电了——手机**不能变砖。**

这个"不能变砖"的保证靠的是**A/B 分区设计**。

### 底层发生了什么？

设备存储被分为两个完整系统分区——**slot A 和 slot B。**

当前运行在 slot A。OTA 下载新版本到 slot B → 设置 slot B 为 active → 重启 → 如果 slot B 启动成功，标记成功。如果失败（多次尝试），Bootloader **自动回切到 slot A。**

**增量更新（Delta OTA）：** 不是下载完整 3GB 系统——只传变更的块。bsdiff 算法计算新旧版本的二进制差异 → 生成极小 patch 文件 → 设备端用 patch + 旧版本重建新版本。一个 3GB 的更新可能只需下载 300MB patch。

### 大白话解释

你的房子有两套完全对称的电路（A 区和 B 区）。平时用 A 区。升级时对 B 区施工——A 区照常用。施工完测试通过 → 切换到 B。施工中途被断电了？系统自动切回A区——你甚至没感觉到。

### 核心启示

> A/B 分区的设计哲学：**永远保留一个"已知可工作的版本"，新版本在完全验证通过前不替换原有版本。** 这不仅适用于 OTA——蓝绿部署、金丝雀发布、数据库迁移的"影子表"——都在遵循同一个原则：**回滚路径比升级路径更重要。**

## Q33 · 设计智能家居设备管理平台——设备离线了，指令怎么不丢？

### 你有没有思考过这么一个问题？

你用手机 App 控制家里的智能灯——点"关灯"。但灯恰好离线了（Wi-Fi 断了）。如果指令直接发给灯——丢了。下次灯上线，它不知道你让它关。

**设备影子（Device Shadow）模型解决了这个问题。**

### 底层发生了什么？

设备影子是云端的设备状态缓存（JSON 文档），包含三部分：

- **Reported**（设备上报的实际状态）：灯报告"我现在是开的"
- **Desired**（应用期望的目标状态）：App 设置"请关灯"
- **Delta**（差异）：`{"state": "OFF"}` —— 期望和实际不一致

设备离线时，Desired 被更新但 Reported 不变。设备上线后通过 MQTT 订阅影子 Topic → 获取 Delta → 执行 → 更新 Reported → 同步完成。

**解决了物联网三大难题：** 间歇连接下的指令可靠性、状态冲突（多 App 同时控制 → 最后写入者获胜 + 版本号）、弱网环境（App 随时读最新已知状态，不依赖设备在线）。

### 大白话解释

设备影子 = 一个云端记事本。你写上"关灯"。灯上线后第一件事是翻记事本看"别人对我有什么期望"→ 照做。App 显示的状态也是看记事本——不直接问灯。**灯离线，记事本还在。**

### 核心启示

> 设备影子的本质是**在"不稳定连接"上叠加一层"稳定抽象"。** 物联网设备随时可能离线——影子用"期望 vs 实际"的差异模型，把"指令可靠送达"变成了"状态最终一致"。**所有异步系统的核心模式都是：不要让发送方等待接收方——让中间状态缓存承担"等待"。**

## Q34 · 设计一个秒杀系统——100万人抢1000张票

### 你有没有思考过这么一个问题？

100 万人同时点"购买"，你只有 1000 张票。黄牛脚本比你快 1000 倍。有人篡改请求参数把价格改成 1 分钱。同一 IP 注册 100 个马甲账号。

**你的系统不是在"帮人下单"——是在"从 100 万请求里甄别出 999000 个恶意请求"。**

### 底层发生了什么？

防线分层：

**第一层（网络边缘）：** IP 频率限制 + 验证码/滑块。收集鼠标轨迹和点击特征——脚本的轨迹太"完美"，真人有微抖。设备指纹（Canvas/WebGL 特征）——同一设备换了 100 个 IP？标记。

**第二层（业务层）：** 行为分析——正常用户浏览→加购物车→下单有路径和时间间隔。脚本直接跳到下单接口——异常短路径。下单 Token 防重放——每次请求携带一次性 Token。

**第三层（下单核心）：** 库存原子扣减——Redis DECR 原子操作：缓存没库存了直接返回"售罄"，不再查数据库。数据库用乐观锁兜底。**让"判断有无"和"扣减"成为一个原子操作。**

### 大白话解释

体育场门口排队买票。有人穿不同衣服排十次队（多账号）、雇了 100 个人凌晨来排（机器人脚本）、伪造门票试图混进去（参数篡改）。保安（风控系统）的工作不是"放人进去"——是"识别并拦住这些不守规矩的"。保安拦完坏人，剩下的才是有票的人。

### 核心启示

> 秒杀系统的本质不是"怎么卖得快"——是**"怎么在被攻击的前提下仍然正常服务"。** IP 限流、设备指纹、验证码、库存原子扣减——防线布在四层。任何一层被绕过，下一层接住。**安全不是一道门，是纵深防御。**

## Q35 · 设计一个消息队列——生产者、Broker、消费者之间怎么保证消息不丢？

### 你有没有思考过这么一个问题？

你用 Kafka 发了一条消息。这条消息在 Broker 的磁盘上，在多个副本之间同步，最终被消费者读取。每一步都可能失败。**怎么保证这条消息不丢？**

### 底层发生了什么？

**生产者到 Broker：** 生产者发消息 → Broker 写入并返回 ACK。Kafka 的 `acks` 参数控制："0"——发了不管（可能丢）；"1"——Leader 写入确认；"all"——所有 ISR（同步副本）确认。**"all"模式保证不丢但吞吐最低。**

**Broker 内部：** 消息写入 Page Cache，操作系统定期刷盘。Kafka 靠**顺序写日志**——顺序写极快（HDD 上 600MB/s+），不随机更新索引。

**Broker 到消费者：** 消费者拉取消息，处理完提交 offset。如果先提交 offset 再处理——处理时崩溃了，消息永久丢失。正确做法：**先处理后提交。** 这导致"至少一次"语义——消息可能被重复处理。**幂等性是消费者的责任。**

**顺序保证：** Kafka 同一分区内消息严格有序——同一 Key 的消息发到同一分区。但跨分区不保证全局有序。

### 大白话解释

消息队列 = 一个"写便利贴 + 贴到同事桌上"的通道。你写一张贴过去（发消息），同事撕掉做完（消费），在纸条背面打勾（提交 offset）。如果同事撕掉但没打勾就下班了（崩溃）——下次上班纸条还在——重做。

### 核心启示

> 消息不丢 = ACK + 持久化 + 先处理后提交。消息不乱 = 分区内有序 + Key 哈希。消息不重 = **幂等性——这是消费者必须自己保证的。** 消息中间件可以保证"至少一次"，但只有消费者能保证精确一次。

## Q36 · 缓存穿透、击穿、雪崩——三个问题，三个成因，三个解法

### 你有没有思考过这么一个问题？

缓存崩了，数据库被打挂。但你排查后发现——根本原因可能是三种完全不同的事：查了不存在的数据、热点 Key 刚好过期、大量 Key 同时过期。**名字相似，解法完全不同。**

### 底层发生了什么？

**穿透：** 查询不存在的数据 → 缓存永远不命中 → 每次都打到数据库。防御：布隆过滤器（"绝对不存在"直接拦截）+ 空值缓存（缓存空结果，短 TTL）。

**击穿：** 某个热点 Key 过期 → 瞬间大量并发请求同一 Key → 全打到数据库。防御：互斥锁（第一个请求重建缓存，其他等）+ "永不过期"（物理不过期，异步任务更新值）。

**雪崩：** 大量 Key 同时过期。防御：TTL 加随机偏移 + 多级缓存（本地 Caffeine + 远程 Redis）+ 限流降级。

### 大白话解释

- 穿透 = 你专门找冷门书——图书馆没有，每次都跑去总馆问。防御 = 门口挂牌"这本书不存在"（布隆过滤器）。
- 击穿 = 畅销书过期被下架——一瞬间几百人同时冲总馆。防御 = 书快过期时悄悄续借（互斥锁）。
- 雪崩 = 图书馆大扫除把所有书同时下架——瞬间所有人涌向总馆。防御 = 每本书不同过期时间（随机 TTL）。

### 核心启示

> 缓存问题的根源不是"缓存不够快"——是**缓存和数据库之间的"状态一致性窗口"。** 穿透是因为缓存永远追不上不存在的数据；击穿是因为缓存重建有时间窗口；雪崩是因为窗口被同步放大。**解法不是"把缓存做得更完美"，是"让缓存和数据库之间的窗口更可控"。**

## Q37 · Redis为什么快——是"因为纯内存"吗？

### 你有没有思考过这么一个问题？

"Redis 快是因为纯内存操作"——Memcached 也用内存。Redis 快不仅是内存——**是五个因素叠加。**

### 底层发生了什么？

1. **纯内存操作**：无磁盘 IO，纳秒级。但这只是基础。
2. **单线程消除锁竞争和上下文切换**：核心操作在单线程中完成，不需要给共享数据加锁。多线程（Redis 6.0 IO 线程）只用于网络读写，命令执行仍是单线程。
3. **IO 多路复用（epoll）**：单线程同时监听数千连接的读写事件——不是每连接起一个线程阻塞等。
4. **手工优化的数据结构**：SDS（动态字符串）、ZipList（压缩列表）、SkipList（跳表）——都是为特定场景手工优化，比通用数据结构更高效。
5. **渐进式 rehash**：字典扩容时不一次性拷贝所有数据——分批迁移，不阻塞。

**单线程的瓶颈：** 单个耗时命令（`KEYS *`、大 Key 的 DEL）会阻塞所有其他请求——需要用 SCAN / UNLINK 拆分。

### 大白话解释

Redis = 一个技艺极高的厨师，一个人看着十口锅——动作极快，不需要跟别的厨师抢锅（单线程无锁）。每口锅好了就翻一下（epoll 事件驱动）。十口锅同时管——比十个互相拖后腿的厨师快。

### 核心启示

> Redis 快不是"一件事做对了"——是**五件事同时做对了。** 纯内存、单线程、IO 多路复用、数据结构、渐进式 rehash——去掉任何一个都会掉速。**高性能系统的优化往往是多个因素的叠加，不是单点突破。**
