# 第五章 工程实践

> 软件工程不是"写代码"——是管理复杂度。从 CI/CD 到测试金字塔、从 Git 到技术债、从微服务到可观测性。工程师的日常不是在写新功能，而是在和复杂度搏斗。

## Q66 · 设计CI/CD流水线——代码提交后自动到上线

![配图](images/chapter5/q66-cicd-pipeline.png)

### 你有没有思考过这么一个问题？

你写完一段代码，在自己电脑上跑通了，然后发给同事——他说"在我机器上跑不了"。你花了一下午远程帮他调环境，最后发现是他本地的 Node.js 版本差了一个小版本号。

或者更常见的场景：你们的团队每次发布都是一场"大事件"——凌晨两点的发布窗口，全员待命，部署文档写了三十页，一个人操作一个人盯着，出了问题赶紧回滚，然后再等下个月的窗口。

你有没有想过：为什么有些团队一天能发布十几次，而你们一个月才发一次？差距不在于代码质量——在于有没有一套自动化的流水线，让"代码从提交到上线"这件事变成一个不需要人盯着的、可重复的、安全的过程。

### 底层发生了什么？

CI/CD（Continuous Integration / Continuous Deployment，持续集成 / 持续部署）是一条从代码提交到线上运行的自动化管道。它不是一个工具，而是一套由多个阶段串联起来的流程。

**CI（持续集成）阶段——发现问题越早，修复成本越低。**

代码 push 到远程仓库后，CI 系统自动触发：

1. **代码拉取与隔离环境创建**：CI 系统（如 Jenkins、GitLab CI、GitHub Actions）在干净的容器中拉取代码，确保不受本地环境残留影响。
2. **依赖安装**：根据 `package.json` 或 `pom.xml` 安装依赖，锁定版本号防止"在我机器上能跑"的问题。
3. **静态代码分析（Static Analysis）**：工具如 SonarQube、ESLint、golangci-lint 检查代码风格、潜在 bug、安全漏洞。这个阶段不需要运行代码，只做静态扫描。
4. **编译构建**：把源码编译成可执行文件或打包成 Docker 镜像。如果编译失败，立刻拒绝——不让问题进入下一阶段。
5. **单元测试**：运行所有单元测试，确保单个函数级别的正确性。这一步通常在几秒到几十秒内完成。
6. **集成测试**：启动依赖服务（数据库、消息队列），测试模块间的交互。这一步比单元测试慢，通常在分钟级。

**CD（持续部署 / 持续交付）阶段——让发布从"大事件"变成"日常操作"。**

7. **构建产物**：把编译后的代码打包成 Docker 镜像或可部署的制品（Artifact），推送到镜像仓库。
8. **部署到测试环境**：自动部署到测试环境，跑端到端测试（E2E Test，即模拟真实用户操作的自动化测试）。
9. **部署到预发布环境（Staging）**：预发布环境尽可能模拟生产环境，做最后的验证。
10. **人工审批（可选）**：对于关键业务，需要人工确认后才能部署到生产环境。
11. **灰度发布**：不是一次性全量上线，而是先放 1% 的流量，观察监控指标，如果没有异常，逐步放大到 5% → 25% → 100%。
12. **全量上线 + 自动回滚**：如果灰度阶段发现异常（错误率上升、延迟飙升），自动回滚到上一个稳定版本。

**Pipeline as Code（流水线即代码）**：整个流程用代码描述（如 `.gitlab-ci.yml`、`Jenkinsfile`），和业务代码一起版本管理。这意味着流水线本身也是可审查、可回滚、可演进的。

**关键设计原则：**

- **Fail Fast（快速失败）**：把检查前置——编译错误在编译阶段发现，不要等到部署后才发现。越早发现，修复成本越低。
- **每个阶段都是幂等的**：重复执行同一版本代码，结果必须一致。
- **环境一致性**：Docker 镜像保证了开发、测试、生产环境的一致性。这就是"消除在我机器上能跑"的根本手段。

### 大白话解释

想象一家餐厅的自动化流水线：

厨师把食材放上传送带（代码 push）→ 机器自动检查农药残留（静态分析）→ 自动试吃一小块确认没毒（单元测试）→ 自动装盘打包（构建镜像）→ 送到内部试吃间让资深厨师品尝（集成测试和预发布）→ 经理点头（人工审批）→ 先端到一桌客人尝（灰度发布）→ 没问题就端给所有客人（全量上线）。

关键在于：整个过程没有人需要"盯着"传送带。如果某一步发现问题，传送带自动停止，食材退回给厨师修改。

### 核心启示

CI/CD 的核心价值不是"自动化"——是**让发布从恐惧变成日常。** 当你一天可以发布十次时，每次发布的变更范围从"三个月改了 500 个文件"缩小到"最近一次改了 3 个文件"。变更越小，出问题的概率越低，排查越容易。这不是工具的升级——是工程文化的转变：从"小心翼翼地不发"到"自信地频繁发"。

## Q67 · 测试金字塔——为什么单元测试多而快，E2E少而慢？

![配图](images/chapter5/q67-test-pyramid.png)

### 你有没有思考过这么一个问题？

你们的 CI 流水线跑了 40 分钟，其中 35 分钟花在 E2E 测试上。开发等不起，于是有人开始跳过测试直接提交代码。结果线上出了 bug，大家互相指责"你怎么不跑测试"——但真相是，测试太慢了，没人愿意等。

你有没有想过：为什么测试要分层次？为什么不能全部写 E2E 测试，毕竟那才是最接近真实用户场景的？或者说反过来——为什么不能全部写单元测试，那不是最快吗？

答案在于一个残酷的经济学事实：测试的"真实度"和"速度"是矛盾的。越接近真实场景的测试越慢、越脆弱、越难维护。测试金字塔就是为了在这个矛盾中找到最优解。

### 底层发生了什么？

**测试金字塔（Test Pyramid）** 是一个分层测试策略模型，按测试的粒度从底到顶分为三层：

**底层：单元测试（Unit Test）——占 70-80%**

单元测试测试单个函数或方法的行为。比如你有一个 `calculateDiscount(price, rate)` 函数，单元测试就是验证：输入 `(100, 0.2)` 是否返回 `80`。

- **速度**：毫秒级。一个测试套件几百个用例，几秒内跑完。
- **隔离性**：不依赖外部服务（数据库、网络），用 Mock（模拟对象）替代依赖。
- **定位精度**：测试失败时，你知道是哪个函数出了问题。
- **覆盖率**：分支覆盖率、行覆盖率可以精确度量。
- **维护成本**：低。函数接口稳定时，测试几乎不需要改。

**中层：集成测试（Integration Test）——占 15-20%**

集成测试测试多个模块协作时的行为。比如"用户下单"这个流程涉及：订单服务 → 库存服务 → 支付服务。集成测试验证这三个服务协作时数据是否一致。

- **速度**：秒级。需要启动依赖服务（如内存数据库、Mock 服务）。
- **覆盖范围**：验证模块间接口契约、数据流转。
- **维护成本**：中等。接口变化时需要同步修改。

**顶层：端到端测试（E2E Test，End-to-End Test）——占 5-10%**

E2E 测试模拟真实用户的完整操作路径。比如用 Selenium 或 Playwright 打开浏览器，点击"登录"→输入账号密码→点击"下单"→验证订单创建成功。

- **速度**：分钟级。需要启动完整环境（浏览器、前端、后端、数据库）。
- **脆弱性**：高。网络抖动、UI 渲染延迟、第三方服务超时都可能导致测试随机失败。
- **维护成本**：高。UI 改动、接口变化都需要更新测试用例。
- **调试难度**：失败时只知道"这条路径没走通"，需要逐段排查。

**反金字塔（Ice Cream Cone）的陷阱**

很多团队的测试结构是倒过来的——大量 E2E 测试，少量单元测试。这叫"冰淇淋模型"（Ice Cream Cone Pattern）：

- E2E 跑 2 小时 → 开发跳过测试 → 问题漏到线上
- E2E 随机失败 → 大家开始忽略失败 → 测试形同虚设
- 修一个 bug 写一个 E2E → 测试越来越多越来越慢 → 恶性循环

**为什么会形成反金字塔？** 因为写 E2E 测试在心理上更容易——"我要测整个流程"。但每个 E2E 测试都隐式地测了大量底层逻辑，这些逻辑本该由单元测试覆盖。当 E2E 失败时，你不知道是哪个环节的问题，排查成本极高。

**正确的分层策略：**

- 核心业务逻辑（计算、状态机、规则引擎）→ 单元测试全覆盖
- 模块间接口契约 → 集成测试
- 关键用户路径（登录、下单、支付）→ 少量 E2E
- 边界场景和异常处理 → 单元测试

### 大白话解释

想象盖一栋大楼：

- **单元测试** = 检查每块砖是否合格——砖的强度、尺寸、材质。每块砖检查几毫秒，发现问题立刻换。大楼有几万块砖，但检查很快。
- **集成测试** = 检查砖之间的水泥是否牢固、墙壁是否垂直、管道接口是否密封。需要几秒到几分钟。
- **E2E 测试** = 检查整栋大楼所有系统协同工作——打开总闸看灯亮不亮、开水龙头看有没有水、开电梯看能不能到顶楼。需要几十分钟，而且偶尔电梯会卡——但你不确定是电梯的问题还是电路的问题。

大部分时间花在检查砖头（单元），少量时间检查墙壁（集成），极少量时间检查整栋楼（E2E）。如果你每块砖都检查了，墙壁出问题的概率就很低；如果墙壁也检查了，整栋楼出问题的概率就更低。

### 核心启示

测试金字塔不是教条——是**经济学。** 单元测试便宜且定位精准→堆量。E2E 昂贵且脆弱→只覆盖核心路径。把测试成本放在"最可能出问题且修起来最便宜"的层。如果你的 CI 跑了 30 分钟还没完，不是该买更快的机器——是测试结构出了问题。

## Q68 · Git Rebase vs Merge——提交历史的两种哲学

![配图](images/chapter5/q68-rebase-vs-merge.png)

### 你有没有思考过这么一个问题？

你和一个同事同时在一个 feature 分支上开发。你先 push 了代码，他 pull 下来继续改。改完 push 时发现冲突——你们的代码改了同一个文件。他选了 `git pull --rebase`，你选了 `git pull`（默认 merge）。结果他的提交历史是一条直线，你的多了一个 merge commit。同一个功能，两种历史——哪种对？

你有没有想过：为什么 Git 提供两种合并方式？它们的区别不只是"历史好不好看"——而是对"历史是什么"这个问题的不同回答。

### 底层发生了什么？

**Merge（合并）——保留真实的时间线**

`git merge feature` 会在目标分支上创建一个新的 commit——Merge Commit。这个 commit 有两个父节点，记录了"两条线在这里汇合"的事实。

```
main:    A --- B --- C --- M (merge commit)
                  \     /
feature:           D --- E
```

- **保留真相**：你可以看到 D 和 E 是在 C 之后、M 之前并行开发的。时间线是真实的 DAG（Directed Acyclic Graph，有向无环图——一种数据结构，节点间的依赖关系不形成环）。
- **不改写历史**：每个 commit 的 SHA-1（Secure Hash Algorithm 1，一种生成 40 位十六进制指纹的哈希函数，Git 用它唯一标识每个提交）不变。这意味着任何人基于某个 commit 的工作不会因为 merge 而失效。
- **历史复杂**：如果团队有多个 feature 分支并行开发，merge commit 会让历史图变成网状，看起来"乱"。

**Rebase（变基）——重写历史为一条直线**

`git rebase main` 会把 feature 分支上的每个 commit "摘下来"，重新 apply 到 main 分支的最新 commit 之后。

```
Before:
main:    A --- B --- C
                  \
feature:           D --- E

After rebase:
main:    A --- B --- C
                      \
feature:               D' --- E'
```

- **D' 和 E' 是全新的 commit**：虽然代码内容相同，但它们的 SHA-1 变了（因为父节点变了）。这就是"改写历史"的含义。
- **历史是一条直线**：看起来像是 feature 分支是在 C 之后线性开发的。
- **丢掉了并行开发的信息**：你看不出来 D 和 E 实际上是和 C 同时进行的。

**Rebase 的内部机制：**

Rebase 实际上做了三步操作（对每个 commit）：
1. 找到 feature 分支和 main 分支的共同祖先（Common Ancestor）
2. 把 feature 上每个 commit 的变更"打补丁"（Patch）提取出来
3. 依次应用到 main 最新 commit 之后，生成新的 commit

如果有冲突，Rebase 会暂停，让你手动解决冲突后用 `git rebase --continue` 继续。这个过程中，你可以用 `git rebase --abort` 放弃回到 rebase 前的状态。

**黄金法则：永远不要 Rebase 已 push 的公共分支**

为什么？假设你 push 了 commit X，同事基于 X 开始工作。然后你 rebase 了 X——X 变成了 X'（新 SHA）。你 force push 后，同事 pull 下来发现他的 X 和你的 X' 冲突——因为它们是同一个变更的两个不同版本。这会让同事的工作区崩溃。

**Squash Merge——第三种选择**

`git merge --squash` 把 feature 分支上的所有 commit 压缩成一个 commit。适合"feature 分支有很多中间 commit（fix typo、fix test），但主分支只需要一个干净的 feature commit"的场景。GitHub PR 默认提供 Squash and Merge 选项。

**实际团队规范示例：**

- 个人 feature 分支：用 rebase 保持整洁，合并到 main 前清理 commit
- 公共 main 分支：用 merge 或 squash merge 保留真相
- 已 push 的分支：永远只 merge，不 rebase

### 大白话解释

Merge 和 Rebase 就像两种记日记的方式：

- **Merge** = "周一我做了 A，周二同事做了 B，周三我做了 C，然后我们把 A 和 C 合在一起。"——记录了真实的并行开发时间线。日记看起来"乱"，但每件事都是真的。
- **Rebase** = "我先做了 A，然后做了 C，然后同事做了 B。"——把别人做的事"嫁接"到你的最新版本上，假装是一直从头到尾一条线做的。日记看起来整洁，但时间线是假的。

**铁律**：你自己写的日记可以重新整理（个人分支 rebase），但已经交给别人的日记不能改（公共分支不能 rebase）——因为别人已经基于旧版本开始工作了，你改了版本号，他们的工作就全乱了。

### 核心启示

选 Merge 还是 Rebase = **你更在乎"历史的真相"还是"历史的可读性"。** 个人 feature 分支用 rebase 保持整洁；公共 main 分支用 merge 保留真相。这不是技术选择——是对"代码历史该怎么被阅读"的价值判断。理解了这一点，你就不会再纠结"哪种方式更好"——它们服务于不同的目的。

## Q69 · 蓝绿部署、金丝雀发布、灰度发布

![配图](images/chapter5/q69-deployment-strategies.png)

### 你有没有思考过这么一个问题？

你上线了一个新版本，10 分钟后发现一个严重 bug——所有用户都受影响。你赶紧回滚，但回滚也需要时间（停服务、切旧版本、重启），这期间用户骂声一片。整个过程持续了 30 分钟，30 分钟的完全宕机。

你有没有想过：为什么不能上线后只让 1% 的用户用新版本？发现问题立刻切回旧版本，影响范围只有 1%？或者更激进——为什么不能同时跑两套版本，秒级切换？

这就是蓝绿部署、金丝雀发布和灰度发布要解决的问题。它们的共同目标不是"怎样上线更快"——而是"出问题了怎样以最小损失回退"。

### 底层发生了什么？

**蓝绿部署（Blue-Green Deployment）——两套完整环境，秒级切换**

- **Blue 环境**：当前线上运行的生产环境
- **Green 环境**：新版本部署的待切换环境

流程：把新版本部署到 Green 环境 → 在 Green 环境上跑完整测试 → 通过路由规则把流量从 Blue 切到 Green → 观察监控 → 如果出问题，把流量切回 Blue（秒级回滚）

- **优点**：回滚速度极快——只需要切路由，不需要重新部署。
- **缺点**：需要双倍的基础设施成本。两套环境同时运行，资源利用率低。
- **适用场景**：对回滚速度要求极高的核心服务、有足够预算的场景。

**金丝雀发布（Canary Release）——逐步放量，暴露面小**

名字来自矿工带金丝雀下矿——先让金丝雀试，它没事矿工才进。

流程：新版本部署到少量实例（如 1% 的服务器）→ 把 1% 的流量导入新版本 → 观察监控指标（错误率、延迟、业务指标）→ 如果正常，逐步放大到 5% → 25% → 50% → 100% → 如果异常，立刻切回旧版本

- **优点**：风险可控，出问题只影响少量用户。
- **缺点**：需要流量路由能力（Nginx、Service Mesh）、需要细粒度监控来发现问题、放量过程需要时间。
- **自动回滚**：现代 CI/CD 系统可以设置自动回滚阈值——如果错误率超过 0.5%，自动切回旧版本，不需要人工介入。

**灰度发布（Gray Release / Feature Flag）——按维度控制功能可见性**

灰度发布不切换整个版本，而是用功能开关（Feature Flag / Feature Toggle）控制某个功能对哪些用户可见。

流程：新代码部署到所有服务器（和旧代码共存）→ 功能默认关闭 → 逐步对部分用户打开（如先对内部员工、再对 1% 用户、再对全部用户）→ 出问题直接关掉开关，不需要重新部署

- **优点**：不需要额外环境，回滚是"关开关"而不是"切版本"。
- **缺点**：代码中混入了 Feature Flag 逻辑，增加了复杂度。需要 Feature Flag 管理平台（如 LaunchDarkly、Unleash）。
- **维度控制**：可以按用户 ID、地域、设备类型、VIP 等级等维度灰度。比如"先让北京用户用新功能，没问题再开放给全国"。

**三者可以组合使用：**

```
蓝绿切换环境 + 金丝雀逐步放量 + Feature Flag 控制功能
```

- 用蓝绿保证秒级回滚能力
- 用金丝雀控制流量比例
- 用 Feature Flag 在功能粒度做更细的控制

**实际案例：** 大厂的一次发布通常这样走：代码合并 → 自动构建镜像 → 部署到预发布环境验证 → 部署到生产环境（但不切流量）→ 用 Feature Flag 开给内部员工 → 1% 灰度 → 5% → 25% → 100% → 关闭旧版本。整个过程从代码合并到 100% 上线可能需要 2-4 小时，但全程自动且可回滚。

### 大白话解释

- **蓝绿部署** = 有两套宴会厅——蓝色厅正在接待客人，绿色厅在准备新菜单。新菜单在绿色厅试做完毕后，直接把客人引导到绿色厅。如果客人投诉菜不好吃，一秒钟切回蓝色厅。
- **金丝雀发布** = 新厅先放 5% 的客人进去吃。不好只影响 5% 的客人——剩下 95% 还在旧厅正常吃。如果 5% 的客人反馈正常，再放 25% 进去，直到全部切换。
- **灰度发布** = 菜单上一道新菜标着"仅 VIP 可见"。菜已经做好了放在所有桌子上，但只有 VIP 才能看到和点。出了问题直接在菜单上划掉，不需要撤桌。

### 核心启示

发布策略的核心不是"怎样上线"——是**"出问题了怎样以最小损失回退"。** 回滚速度比上线速度更重要。蓝绿提供秒级回滚但成本高；金丝雀控制暴露面但需要时间放量；Feature Flag 提供功能级开关但增加代码复杂度。选择哪种策略取决于你的业务能承受多大的"出问题窗口"——金融系统可能需要蓝绿（秒级回滚），社交应用可能金丝雀就够了（1% 的问题用户可以接受）。
