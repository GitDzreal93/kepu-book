# 第六章 分布式系统

> 当一台机器不够用时——CAP 的取舍、共识的协商、熔断的隔离、可观测性的全貌。分布式系统的根本挑战不是"让多台机器协同工作"，而是"在不可靠的网络上模拟可靠的系统"。

## Q78 · CAP定理——为什么银行选CP，朋友圈选AP？

![配图](images/chapter6/q78-cap-theorem.png)

### 你有没有思考过这么一个问题？

你有两个数据库节点——一个在北京，一个在上海。用户在北京节点存了一笔钱，但网络突然断了——北京和上海之间的光纤被挖断了。这时候：

- 如果上海节点继续提供服务，它不知道北京刚存了多少钱——数据不一致了。
- 如果上海节点拒绝服务以保证数据一致，用户就"存不了钱"——系统不可用了。

你必须选一个：要么允许不一致（牺牲 C），要么拒绝服务（牺牲 A）。你有没有想过：为什么不能两个都要？在分布式系统中，这个"既要又要"为什么做不到？

### 底层发生了什么？

**CAP 定理（CAP Theorem）** 由 Eric Brewer 在 2000 年提出，2002 年由 Gilbert 和 Lynch 正式证明。CAP 是三个词的缩写：

- **C（Consistency，一致性）**：所有节点在同一时刻看到的数据相同。读操作无论访问哪个节点，都能读到最新的写入数据。注意：这里的 C 不是 ACID 中的 Consistency（事务一致性），而是**线性一致性（Linearizability）**——所有操作看起来像是排好序逐个执行的，读到的总是"最新"的值。

- **A（Availability，可用性）**：系统持续对外提供服务。每个请求都能在合理时间内得到非错误响应——不保证返回的是最新数据，但保证有响应。

- **P（Partition Tolerance，分区容错）**：网络可能分区——节点间的网络通信可能中断。系统在网络分区时仍能运行。

**CAP 的真正含义——不是"三选二"，而是"P 必选，C 和 A 二选一"**

P（分区容错）不是可选项——在真实分布式系统中，网络一定会出问题。你不能假设网络永远可靠。所以实际选择是：

- **网络正常时**：C 和 A 可以同时满足。
- **网络分区时**：你必须在 C 和 A 之间选一个。

为什么分区时不能两个都要？因为分区意味着节点间无法通信：

```
北京节点          上海节点
  |                 |
  |  ← 光纤断了 →   |
  |   无法通信       |
  v                 v
用户存了100万    用户来查询余额
```

- 如果上海节点继续返回（可能是旧的）余额 → A 满足，C 违反（因为北京存了新数据但上海不知道）。
- 如果上海节点拒绝返回 → C 满足（等网络恢复后同步，保证一致性），A 违反（用户查不了）。

**CP 系统——一致性优先**

银行核心交易系统选 CP。宁可"服务暂时不可用"（用户查不了余额），也不能"数据不一致"（余额算错）。

- 当分区发生时：节点发现无法和多数派通信 → 拒绝服务。
- 网络恢复后：同步数据，恢复服务。
- 典型系统：ZooKeeper、etcd、HBase（强一致性数据库）、MongoDB（默认 CP）。

**AP 系统——可用性优先**

朋友圈、微博选 AP。宁可"短暂看到旧数据"（少看了一条新朋友圈），也绝不能"加载失败"（应用白屏）。

- 当分区发生时：节点继续提供服务，返回可能过时的数据。
- 网络恢复后：异步同步，最终一致。
- 典型系统：Cassandra、DynamoDB（默认 AP）、Redis（AP，集群模式下可能丢数据）、Eureka（服务发现）。

**BASE 理论——AP 系统的实践指南**

CAP 选择了 AP 后，如何保证数据"最终一致"？BASE（Basically Available, Soft state, Eventually consistent）理论给出答案：

- **Basically Available（基本可用）**：系统在故障时允许损失部分可用性（响应变慢、降级服务），但核心功能仍可用。
- **Soft state（软状态）**：允许数据存在中间状态——不一致是暂时的。
- **Eventually consistent（最终一致性）**：如果没有新的写入，经过一段时间后，所有节点最终会达到一致。

最终一致性有不同的变体：
- **读己之所写（Read Your Writes）**：你写入的数据，你自己马上能读到。但其他人可能延迟才能读到。
- **单调读（Monotonic Read）**：如果你已经读到了某个版本的数据，不会再读到更旧的版本。
- **因果一致性（Causal Consistency）**：有因果关系的操作保持顺序——如果你先发帖再评论，别人看到的顺序一定是先帖后评论。

**现实中的细粒度取舍**

现代系统不在 C 和 A 之间做粗粒度的二元选择——而是在不同子系统、不同操作上做细粒度取舍：

- 银行系统：转账选 CP，余额查询可以选 AP（查到稍微延迟的余额可以接受）
- 电商系统：下单选 CP（不能超卖），商品详情页选 AP（旧数据无所谓）
- 社交系统：发帖选 AP（先发出去再同步），点赞计数选 AP（最终一致就行）

### 大白话解释

两本记账本，两个人各持一本，通过电话保持同步。平时一切正常——一个人记账后打电话告诉另一个人。

但电话线断了（网络分区）：
- **选 CP**（银行）：放下笔，挂出"暂停服务"牌子。等电话修好了再继续。——数据绝对不会错，但期间不能干活。
- **选 AP**（朋友圈）：继续记账，但不管另一本账本了。等电话修好了再对账。——期间两本账本可能不一样，但随时都能记。

银行选 CP 是因为"钱不能错"——宁可暂停也不能算错。朋友圈选 AP 是因为"晚看到一条朋友圈不影响什么"——但不能让用户打不开应用。

### 核心启示

CAP 不是"三选二"——是**网络分区发生时，在 C 和 A 之间二选一。** P 是分布式系统的前提——你无法假设网络永远可靠。现代系统做的是细粒度取舍：核心交易选 CP，非核心展示选 AP。不存在"完美的分布式系统"——只有"在你业务场景下最合理的取舍"。

## Q79 · 设计分布式锁——Redis SETNX vs ZooKeeper临时节点

![配图](images/chapter6/q79-distributed-lock.png)

### 你有没有思考过这么一个问题？

你有一个电商系统，3 个订单服务节点同时运行。用户下单时需要扣减库存——如果三个节点同时读到"库存=1"并同时下单，就超卖了。

单机系统用一个 `synchronized` 或 Mutex 就能解决——同一时刻只有一个线程能进入临界区。但在分布式系统中，3 个节点在不同的机器上，各自的 `synchronized` 互相看不见——你锁的是自己进程内的线程，不是别的进程。

你有没有想过：在多台机器之间，怎么实现"同一时刻只有一个能执行"？

### 底层发生了什么？

分布式锁的核心思路：把锁放在所有节点都能访问的"第三方"存储中——谁先写入成功谁获得锁。

**方案一：Redis SETNX + 过期时间**

```
SETNX lock_key request_id NX PX 30000
```

- `SETNX`（Set if Not eXists）：只有当 `lock_key` 不存在时才能设置成功。
- `NX`：不存在才设置。
- `PX 30000`：设置 30 秒过期时间——防止持锁进程崩溃后锁永远不释放。
- `request_id`：用 UUID（Universally Unique Identifier，通用唯一识别码）标识持锁者——释放锁时需要验证"是不是我的锁"。

**获得锁**：`SETNX` 返回成功 → 获得锁，执行业务。
**释放锁**：用 Lua 脚本验证 `request_id` 匹配后删除 key（必须原子操作，否则可能误删别人的锁）。

**问题一：锁过期但业务没执行完**

你设了 30 秒过期时间，但业务执行了 40 秒——锁在第 30 秒时被自动释放了，第二个节点获得了锁。现在两个节点同时执行——超卖了。

**Redisson 看门狗（Watchdog）**：Redis 的 Java 客户端 Redisson 提供了自动续期机制——获得锁后启动一个后台线程，每隔 10 秒检查是否还持有锁，如果是就续期到 30 秒。只要进程还活着，锁就不会过期。

**问题二：Redis 主从切换可能丢锁**

Redis 集群是 AP 系统——主节点写入后异步同步到从节点。如果主节点在写入锁后、同步到从节点前挂了，从节点被提升为主——新主没有这个锁记录 → 第二个节点可以成功获取锁 → 两个节点同时持锁。

**RedLock 算法**：Redis 作者 Antirez 提出的多节点锁——在 N 个（通常 5 个）独立 Redis 实例上同时获取锁，只有获得多数（N/2+1）锁才算成功。但 RedLock 也有争议——Martin Kleppmann（《Designing Data-Intensive Applications》作者）认为它在 GC 暂停和时钟漂移场景下仍然不安全。

**方案二：ZooKeeper 临时顺序节点**

```
1. 在 /locks 下创建临时顺序节点：/locks/lock-00000001
2. 检查自己是否是序号最小的节点
3. 如果是 → 获得锁
4. 如果不是 → 监听前一个节点的删除事件
5. 前一个节点删除时（持锁者释放或崩溃）→ 自己变成最小的 → 获得锁
```

- **临时节点（Ephemeral Node）**：客户端连接断开时自动删除——天然的"持锁者崩溃自动释放锁"机制。不需要设过期时间。
- **顺序节点（Sequential Node）**：ZooKeeper 保证同一路径下创建的节点按序号递增——天然实现"排队等待"。
- **CP 保证**：ZooKeeper 是 CP 系统（基于 ZAB 协议）——所有写入经过 Leader 并复制到多数节点。主节点切换不会丢锁。

**对比：**

| 特性 | Redis SETNX | ZooKeeper |
|------|------------|-----------|
| 性能 | 高（内存操作） | 中（需要磁盘持久化） |
| 一致性 | AP（主从切换可能丢锁） | CP（强一致） |
| 崩溃恢复 | 靠过期时间或看门狗 | 临时节点自动删除 |
| 公平性 | 非公平（抢锁） | 公平（FIFO 排队） |
| 复杂度 | 低 | 中（需要理解 ZK 模型） |
| 适用场景 | 性能优先、可容忍偶尔失效 | 正确性优先、不可失效 |

**实际选择：**

- 秒杀扣库存 → Redis（性能优先，偶尔超卖 1-2 单可接受）
- 金融转账 → ZooKeeper 或 etcd（正确性优先，绝对不能错）
- 定时任务调度 → ZooKeeper（保证只有一个实例执行）

### 大白话解释

公共厕所的两种管理方式：

- **Redis 锁** = 你进去后锁门，门锁有个定时器（过期时间）。如果你在里面待太久，门锁自动打开——下一个人可以进去了。但问题是：你还没完事就开门了，两个人同时在里面。

  看门狗 = 你每 10 秒拍一下门说"我还在"——只要有人拍门就重新计时。但如果你的手机没电了（进程崩溃），没人拍门了——门就开了。

- **ZooKeeper 锁** = 门口取号排队。叫到你的号才进去。你出来（释放锁）或者你跑了（连接断开 → 临时节点删除）→ 自动叫下一个号。不会出现两个人同时在里面。

  代价是：取号要等 ZooKeeper 集群达成共识——比 Redis 慢。

### 核心启示

选 Redis 还是 ZooKeeper = **你能容忍多大程度的"锁偶尔失效"。** Redis 快但可能丢锁（适合性能优先，容错可接受的场景）；ZooKeeper 慢但绝对可靠（适合正确性优先，不可出错的场景）。没有完美的分布式锁——只有"在你的业务场景下，错误率的可接受范围"。

## Q80 · Raft共识算法——Leader挂了，新Leader怎么知道哪些日志已提交？

![配图](images/chapter6/q80-raft-consensus.png)

### 你有没有思考过这么一个问题？

一个 5 节点的集群，Leader（领导者）把一条日志复制到了 3 个节点（多数派）——这条日志算"已提交"了。但就在这时候 Leader 挂了。

新 Leader 选举出来——它怎么知道哪些日志已经提交了？如果新 Leader 的日志比某些节点旧，它怎么保证不丢掉已提交的日志？

这是分布式共识算法最核心的问题——在 Leader 频繁切换的情况下，如何保证"已提交的日志永远不会丢失"。

### 底层发生了什么？

**Raft** 是 Diego Ongaro 在 2014 年提出的共识算法（Consensus Algorithm），比 Paxos 更易理解。它是 etcd、Consul、TiKV 等系统的核心。共识算法的目标是：让多个节点对"同一序列日志"达成一致。

**Raft 的三个核心机制：**

**1. Leader 选举（Leader Election）**

Raft 集群有一个 Leader 和多个 Follower。Leader 负责接收所有客户端请求，Follower 只负责响应 Leader 的日志复制请求。

- 每个 Term（任期）最多有一个 Leader。Term 是一个单调递增的整数——每次选举开始时 Term +1。
- Leader 定期发送心跳给所有 Follower。如果 Follower 在超时时间内没收到心跳 → 认为 Leader 挂了 → 变成 Candidate → 发起选举。
- Candidate 递增 Term，给自己投票，然后向其他节点发送投票请求。
- 收到多数票（N/2+1）的 Candidate 成为新 Leader。

**选举约束：Candidate 的日志必须至少和多数节点一样新**

投票时，Follower 会比较 Candidate 的日志和自己的日志：
- 比较最后一条日志的 Term——Term 大的更新。
- 如果 Term 相同，比较最后一条日志的 Index——Index 大的更新。

只有 Candidate 的日志"至少和自己一样新"时，Follower 才投票。这个约束保证了：**新 Leader 的日志一定包含了所有已提交的日志。**

为什么？因为"已提交"= 日志已复制到多数节点。"新 Leader 获得多数票"= 多数节点认为新 Leader 的日志至少和自己一样新。而"多数节点持有某条日志"和"多数节点投票给新 Leader"——这两个多数派必有交集（鸽巢原理）。所以新 Leader 一定持有了已提交的日志。

**2. 日志复制（Log Replication）**

- Leader 收到客户端请求 → 追加到自己的日志 → 并行发送给所有 Follower。
- 当多数 Follower 确认收到后 → Leader 标记该日志为"已提交"（Committed）→ 应用到状态机 → 返回客户端成功。
- Leader 在后续心跳中通知 Follower 哪些日志已提交。

**日志格式**：每条日志包含 Term、Index、Command。

```
日志条目：[Term=3, Index=5, Command="SET x=1"]
```

**日志一致性保证**：如果两条日志的 Term 和 Index 相同，那么它们的 Command 也相同，且之前的所有日志也相同。这是 Raft 的核心不变量——通过 Leader 只追加日志（不改写）+ AppendEntries 的一致性检查来保证。

**3. 安全性（Safety）——保证已提交的日志不丢**

Raft 的 Safety 规则：**已提交的日志永远不会被覆盖。**

这是通过选举约束实现的——新 Leader 的日志至少和多数节点一样新，而已提交的日志一定在多数节点上。所以新 Leader 一定有已提交的日志，不会覆盖它。

但有一个微妙的边界情况：**Leader 不能通过计数来提交前任 Term 的日志。**

假设 Term 2 的 Leader 复制了一条日志到 2 个节点（不够多数），然后挂了。Term 3 的新 Leader 选出后，发现这条日志在 2 个节点上——它不能直接标记为"已提交"（因为不满足多数），但也不能删除它（因为它可能在其他节点上已被提交）。Raft 的处理：新 Leader 会保留这条日志并继续复制，只有当新 Leader 自己的 Term 的日志被提交时，之前的日志才"顺便"被提交。

**日志恢复（Log Recovery）：**

新 Leader 选出后，它的日志可能比某些 Follower 更多或更少。Raft 的处理：

- Leader 发送 AppendEntries 时带上"前一条日志的 Term 和 Index"（prevLogTerm, prevLogIndex）。
- Follower 检查自己的日志在 prevLogIndex 处的 Term 是否等于 prevLogTerm。
- 如果不匹配 → Follower 拒绝 → Leader 递减 nextIndex 重试 → 直到找到一致点。
- 从一致点开始，Leader 用自己的日志覆盖 Follower 的冲突日志。

### 大白话解释

议会模型：

- **Leader 选举** = 议会选议长。议长挂了 → 新选举 → 得票过半者当新议长。候选人必须"至少和多数议员一样新"——即候选人必须知道多数议员知道的提案，否则议员不投票。
- **日志复制** = 议长提案，议员投票。超过半数同意 → 提案通过（已提交）。
- **Safety** = 已通过的提案永远不会被推翻。因为新议长一定知道所有已通过的提案（因为它获得了多数票，而已通过的提案也在多数人手中——两个多数派必有交集）。

**鸽巢原理**：5 个人投票，超过半数=3 人。已通过的提案至少在 3 人手里。新议长获得 3 人支持——这 3 人中至少有 1 人持有那条已通过的提案。所以新议长一定知道它。

### 核心启示

Raft 的核心不是"选举 Leader"——是**"确保任何时候，已提交的日志永远不会丢失"。** 这个保证来自多数派原则——任何两个多数派必有交集。这就是为什么分布式共识需要"多数派"而不是"全部同意"——少数节点可以宕机，但只要多数活着，已提交的数据就不会丢。

## Q81 · 容器 vs 虚拟机——不仅仅是"轻量vs重量"

![配图](images/chapter6/q81-container-vs-vm.png)

### 你有没有思考过这么一个问题？

你在云服务器上部署了 10 个微服务。如果用虚拟机，每个服务一台 VM——10 台 VM，每台至少 2GB 内存，总共 20GB。但用 Docker 容器，10 个容器总共只要 2GB。

你有没有想过：为什么容器比虚拟机轻这么多？它们不都是"隔离运行环境"吗？区别到底在哪？这种轻量化的代价是什么——牺牲了什么？

### 底层发生了什么？

**虚拟机（Virtual Machine, VM）**

虚拟机通过 Hypervisor（虚拟机监控器）在硬件层实现虚拟化。每台 VM 运行完整的 Guest OS（客户操作系统）——包括自己的内核、驱动、系统库。

```
物理硬件
  ↓
Hypervisor (VMware ESXi / KVM / Hyper-V)
  ↓
VM1                VM2              VM3
  ├ Guest OS        ├ Guest OS       ├ Guest OS
  ├ 系统库          ├ 系统库         ├ 系统库
  └ 应用            └ 应用           └ 应用
```

- **隔离级别**：硬件级隔离。每台 VM 有自己的虚拟硬件（CPU、内存、磁盘、网卡）。
- **资源开销**：大。每台 VM 需要完整的 OS——内核、驱动、系统服务——通常 1-2GB 内存。10 台 VM = 10 个 OS = 10-20GB 仅系统开销。
- **启动时间**：分钟级。需要启动完整的 OS（BIOS → bootloader → kernel → init → 系统服务）。
- **隔离强度**：强。一台 VM 被入侵不影响其他 VM——因为它们有不同的内核。

**容器（Container）**

容器通过 Linux 内核的两个特性实现隔离：Namespace 和 Cgroups。所有容器共享 Host OS 的内核——不需要每个容器一个 OS。

```
物理硬件
  ↓
Host OS 内核 (Linux)
  ↓
容器运行时 (Docker / containerd)
  ↓
容器1        容器2        容器3
  ├ App        ├ App        ├ App
  └ Libs       └ Libs       └ Libs
```

- **Namespace（命名空间）**：隔离"能看到什么"。每个容器有自己的 PID namespace（只能看到自己的进程）、网络 namespace（自己的 IP/端口）、挂载 namespace（自己的文件系统）、用户 namespace（自己的用户列表）。容器内的进程以为自己是系统的唯一进程。
- **Cgroups（控制组）**：限制"能用多少资源"。CPU 配额、内存上限、磁盘 IO 限制、网络带宽限制。
- **资源开销**：小。不需要 Guest OS——容器只包含应用和依赖库。通常 50-200MB。
- **启动时间**：秒级（甚至毫秒级）。不需要启动 OS，只需要启动应用进程。
- **隔离强度**：弱。所有容器共享 Host OS 内核——如果 Host 内核有漏洞（如 Dirty COW），所有容器都可能被攻破。

**关键区别：隔离的层级不同**

- VM 是"硬件级隔离"——每台 VM 有自己的虚拟硬件和内核。跨 VM 的隔离是"物理"的。
- 容器是"进程级隔离"——容器只是 Host 上的特殊进程。跨容器的隔离是"逻辑"的（靠 Namespace 保证）。

**互补而非替代——实际架构**

云厂商的标准做法是：物理机上跑 VM 做租户间强隔离，VM 内部跑容器做应用级高效编排。

```
物理机
  ↓
VM（多租户隔离——不同客户在不同 VM）
  ↓
K8s 集群（容器编排——在 VM 内高效部署应用）
  ↓
容器（应用隔离——每个微服务一个容器）
```

这就是"两层虚拟化"——VM 负责安全隔离，容器负责高效部署。AWS 的 Fargate、Azure 的 Container Instances 都是这种模式。

**Seccomp 和 AppArmor**：为了弥补容器隔离弱的缺点，可以加 Seccomp（限制系统调用——容器只能调用部分系统调用）和 AppArmor（强制访问控制——限制容器能访问的文件路径）。这增加了安全性，但仍不如 VM 的硬件级隔离。

### 大白话解释

- **VM** = 同一块地上建 N 栋独立别墅。每栋别墅有独立地基、独立水电系统（独立 OS）。盖一栋别墅很慢、成本高，但隔音好、安全性强——一栋着火不影响其他栋。
- **容器** = 同一栋大楼里隔 N 个房间。所有房间共享大楼的水电系统（共享内核）。隔一个房间很快、成本极低，但隔音差——一个房间的水管爆了可能影响整栋楼。

实际选择：
- 多租户云环境（不信任的租户之间）→ VM（别墅——隔离强）
- 企业内部微服务（信任的服务之间）→ 容器（房间——效率高）
- 大厂混合架构 → VM + 容器（别墅里再隔房间）

### 核心启示

容器不是"更轻的虚拟机"——是**完全不同的隔离机制。** VM 是硬件级隔离（安全但重），容器是进程级隔离（轻但隔离弱）。容器共享内核是它轻量的根本原因——也是它隔离弱的根本原因。理解了这一点，你就知道为什么多租户云用 VM 而企业内部用容器——不是"成本"而是"信任边界"。

## Q82 · 设计分布式消息系统——百万QPS写入、消息不丢、不乱序

![配图](images/chapter6/q82-distributed-mq.png)

### 你有没有思考过这么一个问题？

微博热搜实时更新——每秒百万级"阅读量+1"的写请求。如果直接写数据库——MySQL 扛不住百万 QPS。

但如果用消息队列削峰——百万消息先入队列，然后按数据库能承受的速度消费。问题来了：百万 QPS 写入时，消息怎么保证不丢？同一用户的操作怎么保证顺序？消费者挂了消息会不会丢？

### 底层发生了什么？

**分布式消息系统的核心挑战：**

1. **高吞吐**：百万 QPS 写入
2. **不丢消息**：Broker（消息代理）宕机后消息不丢
3. **顺序保证**：同一 key 的消息按顺序消费
4. **Exactly-Once（精确一次）**：消息不会被重复消费

**Kafka 的架构设计：**

**Partition（分区）——并行度的核心**

Kafka 的 Topic 被分成多个 Partition。每个 Partition 是一个有序的、不可变的、追加写入的日志。

- Producer 写入时指定 key → 同一 key 的消息进同一 Partition → 保证同一 key 内有序。
- 每个 Partition 分配到一个 Broker → 多 Partition 分布在多 Broker → 并行写入和消费。
- 消费者组（Consumer Group）中，每个 Partition 只被一个 Consumer 消费 → 保证同一 Partition 内有序消费。

```
Topic: user_actions (3 Partitions)
  P0: [msg1, msg4, msg7, msg10...]  → Consumer 0
  P1: [msg2, msg5, msg8, msg11...]  → Consumer 1
  P2: [msg3, msg6, msg9, msg12...]  → Consumer 2
```

同一个用户的操作（key=user_id）一定进同一个 Partition，按写入顺序消费——保证顺序性。

**不丢消息——三段保证**

1. **Producer 端**：`acks=all`（所有 ISR 副本确认后才算写入成功）+ 重试机制。如果写入失败自动重试。
2. **Broker 端**：每个 Partition 有多个副本（Replica）——1 个 Leader + N 个 Follower。Leader 收到消息后同步到所有 ISR（In-Sync Replicas，同步中的副本）才标记为 committed。Leader 宕机时从 ISR 中选新 Leader——新 Leader 有所有 committed 消息。
3. **Consumer 端**：手动提交 offset（消费位置）。消费成功后才提交 offset——如果消费失败不提交，下次重启从上次 offset 继续。

**ISR（In-Sync Replicas）机制**：Follower 必须在一定时间内追上 Leader 的日志——如果落后太多会被移出 ISR。Leader 只等待 ISR 中的副本确认——不等待落后太多的副本。这平衡了可靠性和延迟。

**百万 QPS 的实现——批量 + 顺序写**

- **批量写入**：Producer 攒一批消息一起发送（如 16KB 一批），减少网络往返。
- **顺序磁盘写**：Kafka 追加写日志——磁盘顺序写速度接近内存随机写（磁盘顺序写 600MB/s vs 随机写 100MB/s）。
- **零拷贝（Zero Copy）**：Consumer 读消息时用 `sendfile()` 系统调用——数据从磁盘直接到网卡，不经过用户空间，减少 CPU 开销。

**Exactly-Once（精确一次）——最难的问题**

默认 Kafka 提供 At-Least-Once（至少一次）——消息可能被重复消费（Consumer 处理完但提交 offset 前挂了 → 重启后重新消费）。

Exactly-Once 的实现：
- **Producer 端**：分配 PID（Producer ID）和 Sequence Number。Broker 根据 Sequence Number 去重——相同的消息只保留一条。
- **Consumer 端**：把消费 offset 和业务操作放在同一个事务中——要么都成功，要么都回滚。Kafka 的事务 API 支持跨 Topic 的事务。

**分区扩容时的顺序问题**

当你增加 Partition 数量时——原有的 key-Partition 映射会变。同一个 key 的消息可能从 P0 被路由到 P3——但 P0 之前已有的消息不会被移动。这导致同一 key 的消息分散在两个 Partition——顺序被打破。

解决方案：预先规划足够的 Partition 数量（Kafka 不建议减少 Partition，且增加 Partition 会打乱顺序）。或者用"一致性哈希"路由，减少扩容时的影响。

### 大白话解释

消息队列就像邮局的分拣中心：

- **Topic** = 邮编（目的地分类）
- **Partition** = 分拣通道（一个通道一个分拣员）
- **Producer** = 寄信人 → 按收件人邮编投到对应通道
- **Consumer** = 取件人 → 一个通道一个取件员，按到达顺序取

**不丢信** = 每封信抄 3 份存在 3 个分拣中心（副本）。一个中心着火了，另外两个还有。
**顺序保证** = 同一个人的信都走同一个通道 → 分拣员按到达顺序处理。不同人的信可以并行处理。
**百万 QPS** = 100 个通道同时收信（Partition 并行）+ 每次收一捆信而不是一封（批量）+ 信直接从仓库到取件人手中不经过分拣员手（零拷贝）。

### 核心启示

分布式消息系统的核心不是"队列"——是**"在吞吐、可靠性和顺序性之间找到平衡"。** 分区并行换吞吐，副本换可靠性，key 路由换顺序。没有免费的午餐——高吞吐靠分区数量，多副本降低吞吐，顺序保证限制了并行度。Kafka 的设计哲学是"把不可能三角变成可调的参数"——你可以在不同场景下调到不同的平衡点。

## Q83 · 设计海量日志实时搜索——Elasticsearch能做什么，不能做什么？

![配图](images/chapter6/q83-elasticsearch.png)

### 你有没有思考过这么一个问题？

你的系统每天产生 50GB 日志。你想搜索"昨天 14:00-15:00 之间所有包含 NullPointerException 的日志"。用 MySQL——`SELECT * FROM logs WHERE message LIKE '%NullPointerException%' AND time BETWEEN ...` ——50GB 数据的全表扫描，跑 10 分钟。

但用 Elasticsearch，同样的搜索在 1 秒内返回。你有没有想过：Elasticsearch 为什么这么快？它的原理和 MySQL 有什么不同？以及——它有什么做不到的事？

### 底层发生了什么？

**倒排索引（Inverted Index）——Elasticsearch 的核心**

MySQL 用 B+ 树索引——从"行"找"内容"（给定 ID 找到这一行的数据）。Elasticsearch 用倒排索引——从"内容"找"行"（给定关键词找到包含它的所有文档）。

```
倒排索引：
NullPointerException → [doc1, doc5, doc12, doc47, ...]
OutOfMemoryError    → [doc3, doc8, ...]
```

搜索"NullPointerException"时，不需要扫描所有文档——直接查倒排索引，拿到包含这个词的文档列表。复杂度从 O(N) 降到 O(1)。

**分词（Tokenization）——建立倒排索引的前提**

文本不是天然被分成词的。`NullPointerException in thread main` 需要被分成 `NullPointerException`、`in`、`thread`、`main` 四个 Token。不同的分词器（Analyzer）有不同的分词策略：

- **Standard Analyzer**：按空格和标点分词，小写化。"Hello World" → `hello`, `world`
- **IK Analyzer**：中文分词。"分布式系统设计" → `分布式`, `系统`, `设计`
- **Keyword Analyzer**：不分词，整个作为一个 Token

**Elasticsearch 的分布式架构：**

- **Index（索引）**：相当于 MySQL 的表。但 ES 的 Index 被分成多个 Shard（分片），分布在多个节点上。
- **Shard（分片）**：每个 Shard 是一个完整的 Lucene 实例，有自己的倒排索引。写入时根据 `routing` 决定进哪个 Shard。
- **Replica（副本）**：每个 Shard 有副本——主 Shard 挂了从副本读。读请求可以分散到副本上——提高读吞吐。

```
Index: logs (3 primary shards + 3 replicas)
  Node 1: P0, R1
  Node 2: P1, R2
  Node 3: P2, R0
```

搜索流程：客户端发搜索请求 → 任意节点作为协调节点 → 并行查询所有 Shard → 合并结果 → 返回。

**Elasticsearch 能做什么：**

- **全文搜索**：搜索包含关键词的文档，按相关度排序。支持模糊搜索、短语搜索、布尔组合搜索。
- **聚合分析**：按字段分组统计——如"按服务名分组，统计每个服务的错误数"。
- **近实时搜索**：写入后 1 秒内可搜（refresh interval 默认 1 秒）。

**Elasticsearch 不能做什么（或者做得不好）：**

1. **不支持事务**：ES 没有 ACID 事务。你不能保证"写 Index A 成功后写 Index B 也成功"。需要事务的场景用 MySQL。

2. **不适合作为主数据库**：ES 的写入是"先标记再合并"——删除是标记删除（不立即释放空间），频繁更新会导致大量标记删除的文档，需要定期合并（Merge）。

3. **Like 搜索受限**：`%keyword%` 这种前后通配的搜索不走倒排索引——退化成全扫描。ES 的 wildcard query 性能很差。

4. **精确匹配不如关系型数据库**：ES 的 `term` 查询可以精确匹配，但不如 MySQL 的 B+ 树索引高效。ES 强项是全文搜索，不是精确查找。

5. **Join 能力弱**：ES 支持嵌套文档和父子文档，但 Join 查询性能远不如 MySQL。复杂关联分析应该在应用层做。

**容量规划：**

- 倒排索引的膨胀：原始数据 1GB → 倒排索引可能 2-3GB（取决于字段数量和分词策略）。
- 堆内存限制：单个 Shard 建议 < 50GB，超过后查询性能下降。
- Shard 数量：不是越多越好——太多 Shard 增加搜索时的 fan-out 开销。建议每 GB 数据 1-2 个 Shard。

### 大白话解释

- **MySQL 的 B+ 树索引** = 图书馆的编号目录。你知道书号（ID）→ 直接去对应书架找到书。但你不知道书号时——只能一本书一本书翻（全表扫描）。
- **Elasticsearch 的倒排索引** = 图书馆的关键词索引。你想找"包含分布式系统的书" → 查索引卡"分布式系统" → 直接列出所有包含这个词的书号。

所以 MySQL 适合"按 ID 查具体记录"——精确查找快。Elasticsearch 适合"按内容找匹配的文档"——全文搜索快。

### 核心启示

Elasticsearch 不是"更好的 MySQL"——是**完全不同的数据结构和查询模型。** MySQL 是 B+ 树 + 行存储，擅长精确查找和事务。ES 是倒排索引 + 文档存储，擅长全文搜索和聚合分析。两者互补：MySQL 做事务数据库，ES 做搜索和分析引擎。数据从 MySQL 同步到 ES（通过 Canal 或 Logstash），各自做各自擅长的事。

## Q84 · 熔断器——一个服务崩了，怎么阻止全链路雪崩？

![配图](images/chapter6/q84-circuit-breaker.png)

### 你有没有思考过这么一个问题？

你的系统有 5 个微服务：A → B → C → D → E。E 是一个第三方短信服务。某天 E 的服务器挂了，所有请求超时。

A 调用 B → B 调用 C → C 调用 D → D 调用 E（超时 30 秒）。由于 D 在等 E 响应，D 的线程被占满。D 的所有线程都在等 E → D 不可用了。C 调用 D 也超时 → C 的线程被占满 → C 也不可用了。最终整个链路雪崩——5 个服务全挂了，因为最末端的 E 挂了。

你有没有想过：能不能在 D 调用 E 失败几次后，直接停止调用 E——快速失败而不是等 30 秒超时？这就是熔断器（Circuit Breaker）的思想。

### 底层发生了什么？

**熔断器（Circuit Breaker）** 的灵感来自电路中的保险丝——当电流过大时保险丝自动熔断，切断电路保护电器。软件中的熔断器在"服务调用失败率过高时"自动切断调用，防止故障蔓延。

**熔断器的三种状态：**

1. **Closed（关闭/正常）**：正常调用。记录成功和失败次数。
2. **Open（打开/熔断）**：失败率达到阈值后切换到此状态。直接返回错误或降级响应——不再调用下游服务。给下游恢复的时间。
3. **Half-Open（半开/试探）**：经过一段冷却时间后切换到此状态。放少量请求试探下游是否恢复——如果成功则切换回 Closed，如果失败则切换回 Open。

```
     请求成功              失败率 > 阈值
  ┌──────────┐         ┌──────────────┐
  │          ▼         │              ▼
Closed ◄───────── Half-Open ──────► Open
                          ▲               │
                          │   冷却时间后   │
                          └───────────────┘
```

**关键参数：**

- **失败率阈值**：如 50%——当滑动窗口内失败率超过 50% 时触发熔断。
- **最小请求数**：如 20——只有在最近 20 次请求中统计失败率才有意义。不能因为前 3 次都失败就熔断——可能只是偶然。
- **熔断持续时间**：如 10 秒——Open 状态持续 10 秒后进入 Half-Open。
- **半开试探请求数**：如 3——Half-Open 状态放 3 个请求试探。

**降级策略（Fallback）**：熔断后不是直接报错，而是返回降级响应：

- 返回缓存数据（"用户信息暂时不可用，返回上次的缓存"）
- 返回默认值（"推荐服务不可用，返回热门商品列表"）
- 返回简化结果（"搜索服务不可用，返回热门搜索词"）

**熔断 vs 限流 vs 重试——三个不同的保护机制：**

| 机制 | 解决什么问题 | 触发条件 | 动作 |
|------|------------|---------|------|
| 熔断 | 下游服务故障 | 失败率过高 | 停止调用下游 |
| 限流 | 自身容量保护 | QPS超限 | 拒绝部分请求 |
| 重试 | 偶发性失败 | 单次请求失败 | 重发请求 |

三者关系：限流是"保护自己不被冲垮"；熔断是"保护下游不被自己打死"；重试是"处理偶发抖动"。但如果熔断器已经打开——重试是危险的（重试会放大流量，加重下游负担）。

**Service Mesh（服务网格）中的熔断：**

在微服务架构中，每个服务实例都可能有熔断器。传统做法是在代码中嵌入熔断逻辑（如 Hystrix、Resilience4j）。现代做法是在 Service Mesh（如 Istio）层面做——熔断逻辑在 Sidecar Proxy（Envoy）中实现，业务代码不需要感知。

**超时设置的重要性：**

没有超时设置的调用是危险的——如果下游永远不响应，调用方线程永远被占用。每个远程调用必须设置超时：
- 连接超时（Connect Timeout）：建立连接的超时
- 读取超时（Read Timeout）：等待响应的超时
- 超时时间应合理——太短导致偶发抖动误判，太长导致线程占用过久

### 大白话解释

想象一条流水线：A 加工 → B 加工 → C 加工 → D 加工 → E 包装。

E 突然停了（下游故障）。D 在等 E → D 堆满了未完成的半成品 → D 也停了 → C 堆满了 → C 也停了 → 整条线停了。

**熔断器** = D 在发现 E 连续 3 次没响应后，停止往 E 送货——直接把半成品扔到"降级仓库"（返回默认值）。虽然这批半成品没有完成包装，但流水线不会停——D 可以继续处理来自 C 的半成品（送去降级仓库而不是 E）。

10 秒后（冷却时间），D 试着往 E 送一件——如果 E 响应了，恢复正常送货。如果 E 还是没响应，继续熔断。

### 核心启示

熔断器的核心不是"检测故障"——是**"阻止故障蔓延"。** 一个服务的故障不应该拖垮整条调用链。熔断器做的是"断臂求生"——宁可损失一个服务的功能，也不要整个系统崩溃。这是分布式系统"韧性设计"（Resilience）的核心原则：假设故障会发生，设计系统让它在局部故障时仍能降级运行，而不是全面崩溃。

## Q85 · 设计可横向扩展的实时计数器——微博热搜的阅读量

![配图](images/chapter6/q85-realtime-counter.png)

### 你有没有思考过这么一个问题？

微博热搜上一个话题的阅读量——"5.2 亿次阅读"。全国几亿用户同时刷新这个数字。如果每次有人"阅读"都 +1 直接写 Redis——单 Redis 扛不住几十万 QPS 的 INCR 操作。如果写 MySQL——更扛不住。

而且你看到的数字不需要"精确到个位数"——5.2 亿和 5.2001 亿对用户来说没区别。但如果你用"近似计数"，用户会不会觉得数据造假？

你有没有想过：怎么在不压垮数据库的前提下，实现高并发、近实时的计数器？

### 底层发生了什么？

**方案一：直接 Redis INCR（扛不住的场景）**

```
INCR topic:xxx:views  → 每次阅读 +1
```

- 单 Redis 实例的 INCR 性能约 10 万 QPS——对于几十万 QPS 刚好够。但如果多个话题同时热搜——单实例扛不住。
- Redis Cluster 可以分片——把不同话题路由到不同节点。但"计数"本质是写密集操作，Redis 写性能受限于单线程。

**方案二：本地缓存 + 定期批量上报（推荐方案）**

核心思路：不要每次都写——先在本地攒着，攒够了批量写。

```
应用层（每个实例）：
  本地计数器 topic_views = {}
  每次阅读 → topic_views["xxx"] += 1  （内存操作，纳秒级）
  每 5 秒 → 批量上报到 Redis
```

```
Redis 层：
  INCRBY topic:xxx:views 1234  （一次性加 1234，减少 Redis 写次数）
```

这样 Redis 的写 QPS = 总 QPS / 批量大小。百万 QPS ÷ 每批 1000 = Redis 只需 1000 QPS——轻松扛住。

**精度损失**：本地攒了 5 秒的计数——在 5 秒窗口内，计数不是实时的。但用户看到的数字"慢了 5 秒"完全可接受——5.2 亿和 5.2000123 亿没有区别。

**方案三：多级缓存 + 定时同步到数据库**

```
应用本地缓存 → Redis（每5秒批量写）
                   → 数据库（每分钟同步一次，持久化）
```

Redis 做实时读，数据库做持久化。即使 Redis 宕机，从数据库恢复——最多丢 1 分钟的数据（可接受）。

**方案四：HyperLogLog（超大基数近似计数）**

如果计数的是"不重复用户数"（如"5.2 亿人看过"），精确计数需要存每个用户 ID——内存爆炸。HyperLogLog 是一种概率数据结构，用 12KB 内存近似计算不重复元素数量，误差约 0.81%。

```
PFADD topic:xxx:unique_users user_123  → 添加用户
PFCOUNT topic:xxx:unique_users        → 近似不重复用户数
```

5.2 亿用户只需 12KB 内存——精度 99.19%。对于热搜这种场景，完全够用。

**方案五：Count-Min Sketch（频率估计）**

如果需要统计"每个关键词出现了多少次"——精确统计需要存每个关键词的计数。Count-Min Sketch 用多哈希函数 + 二维数组近似统计频率，内存占用极小，适合"Top N 热搜"这种场景。

**最终架构——微博热搜阅读量：**

```
用户浏览 → CDN 缓存页面（计数JS异步上报）
     ↓
应用层本地计数器（内存，每5秒批量上报）
     ↓
Redis（INCRBY批量写入，实时读取）
     ↓
定时任务（每分钟同步到MySQL，持久化）
```

读取时从 Redis 读（毫秒级），展示给用户。MySQL 只做持久化备份。

**一致性保证：**

本地批量上报可能丢数据——如果应用实例在攒满 5 秒数据前崩溃，这 5 秒的本地计数会丢失。解决方案：

- 本地计数器写到文件而非纯内存——崩溃后从文件恢复
- 或者接受这个精度损失——热搜阅读量少 0.01% 没人在乎

### 大白话解释

想象一家商场的客流统计：

- **直接计数** = 每进一个人，保安就按一下计数器（直接写 Redis）。中午高峰期每秒进 50 人——保安手忙脚乱，计数器可能按不过来。
- **批量上报** = 保安拿个小本子记人数，每 5 分钟往主系统报一次总数。每秒只需要记在本子上（内存操作），5 分钟才往系统写一次。虽然数字"慢了 5 分钟"，但完全可接受——商场的"今日客流"不需要精确到秒。
- **HyperLogLog** = 如果你要统计"今天有多少不同的人来过"——不需要记每个人的脸，只需要用一个特殊算法估算——12 张照片就能估出 5 亿人的不重复数，误差不到 1%。

### 核心启示

高并发计数器的核心不是"怎么算得准"——是**"在精度和性能之间找到平衡"。** 用户不需要精确数字——他们需要"看起来在实时增长的大致数字"。批量上报把百万 QPS 降到千级 QPS；HyperLogLog 把 5 亿用户降到 12KB 内存。这些"近似"不是偷懒——是工程智慧：在用户可接受的误差范围内，用最少的资源解决问题。

## Q86 · K8s为什么能自愈——不是AI，是声明式API + 控制器循环

![配图](images/chapter6/q86-k8s-self-healing.png)

### 你有没有思考过这么一个问题？

你在 Kubernetes 上部署了一个应用——3 个副本。某个节点的网络突然故障了——这个节点上的容器还在运行，但网络不通了。几分钟后，K8s 自动在另一个节点上启动了一个新容器——你的应用又恢复了 3 个可用副本。

整个过程没有人干预——K8s "自己"发现问题并修复了。你有没有想过：K8s 是怎么做到"自愈"的？它用了什么魔法？

### 底层发生了什么？

K8s 的"自愈"不是 AI——是**声明式 API + 控制器循环（Reconciliation Loop）**的组合。

**声明式 API（Declarative API）——你描述"想要什么"，不是"怎么做"**

传统命令式（Imperative）运维：
```bash
# 启动 3 个容器
docker run --name app1 myapp
docker run --name app2 myapp
docker run --name app3 myapp

# app2 挂了 → 人工修复
docker run --name app2_replacement myapp
```

K8s 声明式：
```yaml
# Deployment: 我要 3 个副本
apiVersion: apps/v1
kind: Deployment
spec:
  replicas: 3
  template:
    spec:
      containers:
      - name: myapp
        image: myapp:v1
```

你告诉 K8s "我要 3 个副本"——至于怎么启动、在哪台机器上启动、挂了怎么办，K8s 自己解决。

**控制器循环（Reconciliation Loop）——不断对比"期望状态"和"实际状态"**

K8s 的核心设计模式是"控制器循环"——每个控制器持续做一件事：

```
while True:
    desired_state = read_from_etcd()   # 期望状态（如 replicas=3）
    actual_state = observe_cluster()   # 实际状态（当前运行了几个）
    if actual_state != desired_state:
        act_to_reconcile()             # 采取措施让实际=期望
```

**Deployment Controller 的工作流程：**

1. 读取 etcd 中的 Deployment 定义 → 期望 3 个副本
2. 查询集群中当前有多少 Pod 匹配 → 实际 2 个（一个挂了）
3. 期望 ≠ 实际 → 创建 1 个新 Pod → 实际变成 3

**这个循环是持续运行的**——不是"出问题才跑一次"，而是每隔几秒就检查一次。所以当 Pod 挂了后，最多几秒内 Controller 就会发现并修复。

**健康检查——K8s 怎么知道 Pod 挂了？**

K8s 不是靠"容器是否在运行"判断健康——容器可能在运行但应用已经 hang 住了。K8s 用两种探针：

- **Liveness Probe（存活探针）**：K8s 定期调用应用的健康检查接口（如 `GET /health`）。如果返回非 200 或超时 → K8s 认为应用不健康 → 重启容器。
- **Readiness Probe（就绪探针）**：K8s 定期调用应用的就绪检查接口。如果返回非 200 → K8s 认为应用没准备好 → 从 Service 的负载均衡中摘除（不转发流量），但不重启。

**网络故障的自愈流程：**

1. Node-1 的网络故障 → Node-1 上的 Pod 还在运行，但网络不通
2. K8s 的 Node Controller 发现 Node-1 不上报心跳（超过 40 秒）→ 标记 Node-1 为 NotReady
3. NotReady 节点上的 Pod 被标记为 Terminating
4. Deployment Controller 发现期望 3 副本但只有 2 个 Ready → 创建新 Pod 在其他节点
5. 新 Pod 启动 → 通过 Readiness Probe → Ready → 加入 Service 负载均衡
6. 应用恢复 3 个可用副本

整个过程自动完成——不需要人工介入。

**其他 K8s 控制器——都遵循"控制器循环"模式：**

- **ReplicaSet Controller**：确保 Pod 副本数 = 期望数
- **Service Controller**：确保每个 Service 有对应的负载均衡规则
- **Job Controller**：确保批处理任务完成
- **Horizontal Pod Autoscaler (HPA)**：根据 CPU/内存使用率自动扩缩容——本质也是"期望 = 实际"的循环

**为什么声明式 + 控制器循环比命令式好？**

- **幂等性**：同一个 YAML apply 一次和十次效果相同——Controller 每次 compare 后只做"必要的"操作。
- **自愈**：任何偏离期望状态的情况都会被 Controller 自动修复——不需要人工告警和介入。
- **可审计**：期望状态存在 etcd 中（有版本），每次修改都有记录。
- **可回滚**：`kubectl rollout undo` → 切换到上一个版本——因为之前的 YAML 也在 etcd 中。

### 大白话解释

声明式 vs 命令式的区别：

- **命令式** = 你是项目经理，你要一个个喊："小王去启动服务器1""小李去启动服务器2"——如果小王的机器坏了，你得自己发现并叫人重新启动。
- **声明式** = 你贴一张告示："我要 3 台服务器运行"。然后有个"巡检员"（Controller）每隔几秒巡一次——数一数现在有几台在跑，不够 3 台就补上。你不需要管"谁在哪台机器上跑""坏了怎么办"——巡检员自动处理。

巡检员不是 AI——他只是一个死循环：看告示（期望状态）→ 数机器（实际状态）→ 不够就补、多了就删。简单但可靠。

### 核心启示

K8s 的自愈不是魔法——是**"声明期望状态 + 持续对比实际状态 + 自动修复偏差"的工程模式。** 这个模式的威力在于：它不需要"知道故障是什么"——只需要知道"实际 ≠ 期望" → 修复到期望。不管故障是进程崩溃、网络分区、节点宕机还是配置错误——Controller 都能用同一种方式处理：让实际状态回到期望状态。这就是声明式系统的核心价值——你定义"正确"，系统保证"始终正确"。
