## Q56 · 验证码的演变——从扭曲字符到无感验证

### 你有没有思考过这么一个问题？

十年前：输入 4 位扭曲数字。五年前："点击所有红绿灯"。现在：滑块验证——不仅看位置，还分析鼠标轨迹。AI 每破一种，就要发明下一种。**这不是技术问题，是一场军备竞赛。**

### 底层发生了什么？

- **文本型**：扭曲字符 → OCR/CNN 攻破
- **图片型**：选择含 XX 的图片 → CNN 超过人类准确率
- **行为型**：滑块轨迹分析（人手有微抖和不匀速，脚本轨迹完美线性）+ 设备传感器（陀螺仪——手机有微抖，模拟器没有）
- **无感型**：不弹验证码——后台行为分析评分。浏览器指纹 + 鼠标热图 + 页面停留 → 低分放行、中等出滑块、高分直接拒

### 大白话解释

验证码 = 逆向图灵测试。"证明你是人"比"证明你是机器"更难。每次 AI 破解一种，就要发明下一种。最后出口在哪？**无感验证——不需要你"证明你是人"，系统自己判断你的人味够不够。**

### 核心启示

> 验证码军备竞赛的终极出口不是"更难的问题"——是**不再问问题。** 通过连续的行为特征（鼠标、键盘、设备）在后台持续评分——你不需要证明你是人，因为你的一举一动都在暴露你的人味。

## Q57 · 侧信道攻击——不看算法，看CPU功耗、时间、电磁辐射

### 你有没有思考过这么一个问题？

加密算法本身数学上无懈可击。但算法执行时——CPU 的功耗波动、某个 if 分支耗时不同、电磁辐射模式——**这些"边角料"泄露了密钥。**

### 底层发生了什么？

**时序攻击：** RSA 的平方-乘模幂运算——如果当前 bit 为 1 则多做一次乘法 → 乘法比平方慢 → 测量每次运算的耗时能逐 bit 还原私钥。防御：**恒定时间算法**——无论 bit 是 0 还是 1，执行完全相同的操作序列。

**Spectre/Meltdown（2018）：** 利用 CPU 推测执行——推测执行的指令虽然结果被丢弃，但**在缓存中留下痕迹**。攻击者通过测量内存访问延迟推断被推测执行的数据——包括内核内存！

### 大白话解释

你练开保险箱——每次转动密码盘听锁芯的"咔嗒"声。不同数字声音微妙不同——转对的更清脆。你不需要撬锁，只需要"听"。侧信道攻击 = 听功耗的"咔嗒声"、看时间的"微小偏差"、感受电磁的"泄露信号"。

### 核心启示

> 算法可以完美，但**物理实现永远不完美。** CPU 有功耗、有电磁辐射、有缓存命中差异——这些物理特性不受算法控制，却可以被攻击者利用。**安全系统的最薄弱环节往往不是数学——是物理。**

## Q58 · 设计一个接口防刷系统——多层防线

### 你有没有思考过这么一个问题？

一个秒杀接口——同一 IP 一秒 100 次请求、同一手机号注册 100 个账号、同一设备（Canvas 指纹）换了 100 个 IP。防线怎么布？

### 底层发生了什么？

**网络层：** IP 频率限制（Redis 滑动窗口计数器）+ 验证码/滑块人机验证。

**业务层：** 设备指纹（Canvas/WebGL 生成唯一设备 ID → 关联多账号）+ 行为分析（正常用户浏览→下单有路径，脚本直接跳到最后一步）。

**数据层：** 下单频率异常检测 + Token 防重放 + 签名防篡改。

一层拦不住的，下一层接。**纵深防御。**

### 大白话解释

接口防刷 = 银行柜台的安保。门口保安查身份证（IP 频率限制），柜台前摄像头做人脸比对（设备指纹），转账前柜员多问一句"确定是给这个人转吗？"（行为分析 + Token 防重放）。

### 核心启示

> 防刷系统的本质是**让攻击者的成本 > 收益。** 不是"100% 拦住"——是"让攻击者刷 1 单的成本比刷到的利润更高"。**安全不是绝对的——是经济学的。**

## Q59 · "记住密码"和"自动登录"——为什么是安全的大忌？

### 你有没有思考过这么一个问题？

用户体验极好，安全极差。Token 存 `localStorage` → XSS 一偷就走。Refresh Token 长期有效 → 被偷了攻击者可以一直用。**怎么平衡？**

### 底层发生了什么？

- `localStorage`：JS 可读写 → XSS 攻击可窃取
- `httpOnly Cookie`：JS 不可读（防 XSS 窃取）+ SameSite（防 CSRF）+ Secure（仅 HTTPS 传输）
- Refresh Token：长期有效，但必须配合**短时效 Access Token** + **Refresh Token Rotation**（每次刷新换新的 Refresh Token，旧 Token 立即失效）

### 大白话解释

"记住密码" = 你把家门钥匙藏在门口脚垫下——方便自己，也方便小偷。好的设计 = 给客人一个临时密码（Access Token），每天换一次，主人有永久密码（Refresh Token），但如果主人的密码被偷——立刻作废换新的。

### 核心启示

> 安全和体验是永恒的拉扯。**解决方案不是"选一个"——是让安全机制对用户透明。** 用户不需要知道 Token 在哪儿、怎么刷新——他只感受到"我一直是登录状态"。

## Q60 · SIM Swap攻击——为什么你的手机号比密码更值钱？

### 你有没有思考过这么一个问题？

攻击者不需要知道你的密码。他打电话给你的运营商："我手机丢了，帮我把号码转到这张新 SIM 卡上。"**如果客服验证不严格——你的手机没信号了，他收到了你的所有短信验证码。**

然后：忘记密码 → 短信验证 → 重置密码 → 登录成功。你的所有账号全没了。

### 底层发生了什么？

手机号劫持两条路径：SIM Swap（冒充你联系运营商转号）和社工诈骗（"你中奖了，把验证码告诉我"）。

**防御：** TOTP（本地生成验证码，不依赖短信通道）+ FIDO2/Passkey（公私钥体系，私钥不出设备）+ 运营商设置 SIM 锁/号码转移 PIN。

### 大白话解释

你的手机号 = 你的万能钥匙。"找回密码"只需要手机号接收验证码。攻击者不需要撬锁——他只需要让锁匠（运营商）相信他是你，把钥匙复制给他。锁匠有时太好说话——验证流程宽松。

### 核心启示

> 基于短信的 2FA 比没有强，但**SMS 是 2FA 中最弱的一环。** TOTP/FIDO2 把"第二因素"从"能被转移的号"变成"不能被偷的设备"。**安全永远应该假设传输通道可能被劫持。**

## Q61 · 撞库——A站被脱裤，为什么你全网账号都没了？

### 你有没有思考过这么一个问题？

A 论坛被黑客攻破，泄露了你的 `email + 密码 MD5`。攻击者拿这个组合去试 B（你的邮箱）、C（你的网银）、D（你的微博）。**如果你所有网站用同一个密码——全没了。**

### 底层发生了什么？

**Credential Stuffing：** 利用用户密码复用习惯，用泄露凭证去撞别的网站。

防御不在用户——在服务端：登录接口加 MFA（密码对了还需第二因素）+ 异常检测（异地登录、新设备、非常用时段触发风控）。

### 大白话解释

你家大门、办公室门、保险柜门——全是同一把锁。有人捡到了你家大门的钥匙——现在他能开所有门。密码管理器 = 给每扇门配独立钥匙，你只需要记住一把"总钥匙"。

### 核心启示

> 密码复用是个人安全漏洞里**危害最大的单点。** 不是"你的密码太简单"——是"你只有一个密码"。**密码管理器不是更安全的密码——是防止你只有一个密码。**

## Q62 · 零信任——永不信任，始终验证

### 你有没有思考过这么一个问题？

传统安全：进到内网就信任。零信任：**永不信任，始终验证。** 不因为你连上了公司 Wi-Fi 就信任你的设备——每次访问任何资源都要重新验证身份和设备健康状态。

### 底层发生了什么？

零信任三原则：永不信任（不因"在内网"就默认可信）、最小权限（每个服务/用户只获得完成任务的必要权限，动态有时效）、假设已入侵（微隔离限制爆炸半径——一个服务沦陷不能跳板攻击其他服务）。

Google BeyondCorp 实现：所有访问经过统一代理，基于**设备健康状态 + 用户身份**动态授权——不再区分"内网"和"外网"。

### 大白话解释

传统安全 = 小区门口一个门卫——进门后所有住户互相信任。零信任 = 没有小区大门——每家每户门口有自己的保安。每次进任何一扇门前都重新查证。

### 核心启示

> 零信任不是一种"产品"——是**一种假设：你的内网已经被攻破了。** 在这个假设下设计安全策略：把每个服务当外网暴露一样保护。**边界的消失意味着信任的消失。**

## Q63 · 设计防暴力破解登录接口

### 你有没有思考过这么一个问题？

攻击者拿字典在试密码。你需要：同一 IP 试了 10 次 → 限 IP。同一账户试了 5 次 → 临时锁定。试错间隔越来越长（指数退避）。但怎么防攻击者用 IP 池绕开限制？

### 底层发生了什么？

**多层防线：** IP 级别速率限制 + 账户级别锁定 + 密码错误响应时间恒定（防止枚举有效用户名）+ 强制 MFA（即使密码对还需第二因素）。

**防御 IP 池：** 仅限 IP 不够——攻击者用代理池轻松绕过。加上设备指纹 + 行为分析——同一个设备换了 N 个 IP 也是同一攻击者。

### 大白话解释

小偷拿一大串钥匙试你家门。防御：试错 5 次锁门 30 分钟（账户锁定）+ 每次试错等越来越久（指数退避）+ 锁对了钥匙还要对暗号（MFA）。

### 核心启示

> 防暴力破解的每层防线都在做同一件事：**提高攻击者的时间成本。** 你不需要让暴力破解"不可能"——让它从"几小时"变成"几百年"就够了。

## Q64 · 对称加密 vs 非对称加密——TLS怎么把两者组合起来？

### 你有没有思考过这么一个问题？

你我知道同一把钥匙（AES）→ 对称，快。你用我的公钥加密，我用自己的私钥解密 → 非对称，慢 1000 倍。TLS 怎么把两者组合？

### 底层发生了什么？

TLS 握手：先用非对称（ECDHE）安全协商出一个**仅本次会话使用的对称密钥（Session Key）**——这一步安全但只做一次。之后所有数据用对称密钥加密——快。

**前向安全性（Forward Secrecy）：** ECDHE 的临时密钥——即使服务器私钥未来泄露，过去的通信也无法解密。因为每次会话用的密钥都是临时的，不是服务器私钥派生出来的。

### 大白话解释

非对称加密 = 保险箱——安全但每次开关都重，只适合传递"小纸条"（对称密钥）。对称加密 = 密码锁——约定好密码后开关极快，适合搬运大批货物。TLS = 先用保险箱传递密码锁的密码，之后用密码锁搬运。

### 核心启示

> 对称和非对称不是"二选一"——是**让各自做各自擅长的事。** 非对称解决密钥分发（安全但慢），对称解决数据加密（快但需要双方有同一密钥）。**好的系统设计 = 用慢而安全的方法做完必要的基础设施（密钥交换），然后切换到快的方法做大规模操作（加密传输）。**

## Q65 · 被脱裤的数据库泄露之后——攻击者手里有几亿条email+密码，他做什么？

### 你有没有思考过这么一个问题？

A 站泄露了你的 email + MD5。到今天——**这个组合还在被攻击者尝试在你的微信、支付宝、邮箱上登录。** 你能做什么？

### 底层发生了什么？

攻击者的标准流程：破解 MD5 → 拿到 email + 明文密码 → 用这个组合尝试登录全网所有主流服务 → 成功的账号：卖黑产、发垃圾、窃取更多信息 → 信息进一步泄露。

**你能做的：** 检查 Have I Been Pwned → 立刻改所有用到这个密码的网站 → 启用密码管理器为每个站点生成独立随机密码 → 启用 MFA。

### 大白话解释

你家大门钥匙丢了。不只是换大门锁——所有用同一把钥匙的门都要换锁。密码管理器 = 每扇门自动配独立钥匙——丢一把不影响其他门。

### 核心启示

> 数据泄露后你只有一件事要做：**假设你的密码已经被所有攻击者知道。** 改密码、开启 MFA、不再复用密码。**安全不是"不被攻击"——是"被攻击后损失可控"。**
