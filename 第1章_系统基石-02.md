## Q6 · 一个数组访问越界——Java抛异常、Go panic、C可能静默破坏内存。同样是越界，为什么不同语言差这么多？

### 你有没有思考过这么一个问题？

你写 Java 时数组越界了——JVM 抛出一个清晰干净的 `ArrayIndexOutOfBoundsException`，附带完整的堆栈跟踪，告诉你哪个文件、哪一行、试图访问 index 30 但数组只有 10 个元素。你写 Python 时越界了——`IndexError: list index out of range`。Go 直接 `panic: runtime error: index out of range`，程序退出。

然后你听到老前辈说："C 语言里，数组越界不一定会报错。有时它崩、有时它不崩还'正常'跑、有时它默默改了别人变量的值然后你排查一整天。"

这不是"C 语言不好"。这是**不同语言对内存暴露的层次不同**——越界时到底发生了什么，取决于语言的数组在内存中到底"长什么样"。

### 底层发生了什么？

数组在内存中的布局，影响了越界的后果——这取决于你使用的语言选择了三种"数组实现方案"：

**Java 的数组**在 JVM 层面是有元数据的——每个数组对象在堆内存中的布局是："对象头（Mark Word + Klass 指针）+ 数组长度（4字节 int）+ 数组元素数据"。当 JVM 执行 `arr[30]` 时，它先从对象头拿到数组长度 10，然后判断 30 >= 10 → 抛 `ArrayIndexOutOfBoundsException`。这个越界检查是在 JVM 内部做的——它是字节码指令 `aaload` / `iastore` 的内置逻辑。关键在于：这个越界检查**是强制的**，JVM 标准规定必须做，你不能绕过。

**Go 的数组**在编译后的二进制里没有元数据——`[10]int` 在运行时就是 10 个连续的 8 字节整数，共 80 字节，数组长度 10 只在编译时存在，运行时不存在。但 Go 的编译器在每次数组访问前**插入了边界检查指令**：`if index >= len(arr) { panic() }`。这个检查是编译器塞的，不是运行时长的。所以在 Go 汇编中你看到 `CMP`（比较指令）+ `JAE`（条件跳转指令）——它确实是每一次数组访问都在做越界判断。某些场景下（比如你写了一个循环，编译器能证明 index < len），Go 编译器会做边界检查消除优化（BCE），把这个 CMP 去掉。

**C 语言的数组**在运行时**什么都没有**。`int arr[10]` 在栈上分配 10 个连续的 4 字节 int，共 40 字节。数组名 `arr` 只是一个指针——指向首元素地址，没有任何地方存储"这个数组有多长"。`arr[30]` 被编译器翻译成：取 `arr` 的地址 + 30 × sizeof(int) = `arr` 地址 + 120 字节，然后读那个地址的内容。至于那个地址里是什么——它可能是同一栈帧里另一个变量的值、可能是返回地址、可能是未映射的内存（一读就段错误）、也可能是内核空间（一读就被 kill）。没有报错不是因为"它被允许了"——是因为**没有人检查**。

为什么 C 这样设计？因为你真的只想对一块内存按偏移量操作，而 C 提供了做这件事的最直接的途径。嵌入式开发者可能直接对一个绝对地址写寄存器，这时候把数组下标映射为指针偏移而不是做安全检查是你想要的行为——虽然这同时是你需要极其小心的行为。

### 大白话解释

这三种语言对待数组越界，就像三个国家对待边界的安检不一样：

- **Java**就像一个有严密的安检和海关登记系统——过海关时你必须出示护照，系统查到你的通行范围是多少你就只能待在那。超了一定被拦。如果护照被拒，你会收到详细的拒绝通知（堆栈跟踪）。代价是每次出入境都要过海关——虽然慢，但安全。

- **Go**就像有一个简化的电子登记系统——海关把规则写在了所有人都要扫的卡里（编译器插边界检查指令）。你扫码时系统自动检查，越界就被请出去。系统还有一群聪明的工作人员（编译器优化），如果有些场景确定你不会越界（如 `for i:=0; i<len(arr); i++`），就不让你扫这些卡（BCE），让你更快通过。

- **C**就像一个完全开放的边境——没有任何检查站。你踩着油门冲向铁丝网——如果你穿过铁丝网后是一片田野，你会安全穿过去（可能破坏了一片农田——改写了相邻变量的值）。如果铁丝网后面是地雷（未映射内存），你挂了。C 的哲学不是"我们相信你不会越界"，而是"我们没有海关——你开多快、去哪里，完全由你说了算"。

### 核心启示

> 数组不是一个数据容器——它是一个"被包装了不同层级的协议缓冲区的内存块"。**包裹层级越高，越安全但越慢；包裹层级越低，越快但越危险。** Java 的抽象层级最高——牺牲性能换取完整安全；Go 在中间——有检测但有优化；C 在底层——给你原始的指针和偏移，代价是你必须自己负责所有的边界检查。选择哪种语言，本质上是在问：**"我愿意花多少运行时开销，来换取多大程度的'不可能犯这类错'的保证？"**

## Q7 · 0.1+0.2≠0.3——设计金融系统时，浮点数该不该碰？

### 你有没有思考过这么一个问题？

打开任何编程语言的 REPL，敲 `0.1 + 0.2`。Java 返回 `0.30000000000000004`，Python 返回 `0.30000000000000004`，JavaScript 也一样。这不是某个语言有 bug。这是 **IEEE 754 浮点数标准**的特性——而且是"不是 bug 的特性"。但如果你在写一个金融交易系统，`0.1 + 0.2` 的不精确会让你损失真金白银。你不能用 `float` 和 `double` 来存钱。那该用什么？Java 的 `BigDecimal`？存"分"而不是"元"的整数方案？还是用定点数库？每个都有坑。

### 底层发生了什么？

浮点数的真相——二进制无法精确表示大多数十进制小数。10 进制中，1/3 = 0.333333... 是无限循环——同理，在 2 进制中，1/10 = 0.0001100110011... 是无限循环。当我们用 `double`（遵循 IEEE 754 双精度——1 位符号 + 11 位指数 + 52 位尾数）存储 0.1 时，它被"裁剪"了——不是存储真正的 0.1，而是存储它能表示的最近似值：0.1000000000000000055511151231257827021181583404541015625。

0.2 同理——0.2 ≈ 0.2000000000000000111022302462515654042...（实际存储的 52 位尾数精度下的最近近似）。

0.1 + 0.2 是两个近似值相加——结果因 IEEE 754 取近似而溢出了一些二进制位，最终结果被舍入为它最近能表示的值：0.3000000000000000444...。这就是为什么 `0.1 + 0.2 == 0.3` 在几乎所有语言里都返回 `false`。

金融系统的三个方案——各有利弊：

**方案 A：`BigDecimal`（Java）。** 把数字表示为"非缩放值（整数） + 标度（小数位数）"。比如 0.1 被存为 (1, 1)——整数 1，标度 1（即 1 × 10⁻¹）。BigDecimal 的加法和乘法是精确的——1.10 + 2.20 = 3.30 不会产生截断误差。代价是：BigDecimal 操作比 `double` 慢一个数量级以上——一个高吞吐的交易系统如果每笔交易都用 BigDecimal 做复杂计算，性能会吃紧。而且 BigDecimal 本身有"标度管理"的坑：123.40 和 123.4 是两个不同的 BigDecimal（整数分别是 12340 和 1234，标度分别是 2 和 1）。比较之前必须用 `compareTo` 而不是 `equals`。

**方案 B：整数"分"方案。** 所有金额以最小单位（分、厘）存储为 64 位整数。10.99 元 = 1099 分——用 `long` 存 1099。加法、减法全用整数运算——快、无误差。但乘除法非常难：10.99 × 8% 的税 = 1099 × 8 / 100 = 87（整数截断）。你损失了 0.92 分——在数百万笔交易中这累积起来是严重的长款/短款。这种方案需要非常仔细的舍入策略。

**方案 C：定点数库。** 最优雅的工业级解法——内部用整数存"最小单位值"，而在 API 层面提供精确的十进制运算语义，包括可控的舍入模式（HALF_UP、HALF_EVEN）。通常用 128 位整数或专门库实现。性能在 BigDecimal 和 `long`-分方案之间，精确度完全可控。

现实中的选择：绝大多数交易所核心系统采用方案 B 的变体——用 64 位整数存最小单位，配合严格的舍入规则文档和风控系统捕捉舍入误差。方案 C 在需要极高精度（如加密货币交易所）或复杂建模（如利率衍生品）时被选用。

### 大白话解释

你用尺子量东西。用米尺（二进制）量一个恰好 1/10 米的东西——米尺的刻度是按 1/2、1/4、1/8、1/16 刻的——永远找不到 1/10 米的精确刻度，你只能找最接近的那一格（近似）。两段这样的近似拼在一起，误差叠起来。

`BigDecimal` 就是专门准备了一把十进制刻度的尺子。`long`-分方案等于你干脆不说"1.99 元"，说"199 分"——全是整数尺子，为什么需要小数呢？乘除法时的截断差异就像你收到零头——需要在某个环节四舍五入补足。

### 核心启示

> 浮点数的精度问题不是数学错误——**是"精度-速度"的系统级权衡**。IEEE 754 被设计成能在 CPU 上用单一硬件指令完成 `double` 加法（1 个时钟周期级别），代价是"不是精确表示所有实数"。`BigDecimal` 和定点数用更多的 CPU 周期和内存换回精确性——你的选择取决于**在你的场景里，"速度快 10 倍" vs "永远不错一分钱"，哪个更不可妥协"**。金融系统选精确性，物理模拟和图形渲染选速度——**所有工程决策的本质，都是"在约束下，选择你的非妥协项"**。
