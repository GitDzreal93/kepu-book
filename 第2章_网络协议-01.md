## Q16 · 如果让你从零设计一个可靠传输协议——丢包、乱序、拥塞怎么办？

### 你有没有思考过这么一个问题？

你每天用微信发消息、刷抖音看视频。你从未担心过"这条消息会不会只到一半就丢了"。

但如果告诉你：**互联网底层是一张极其不可靠的网络。** 数据包随时可能被丢弃、篡改、乱序到达。可你为什么完全感受不到？

因为 TCP 在底层帮你兜着。理解 TCP 最好的方式不是读它的 RFC，而是问自己：**如果让你来设计，你会怎么做？**

### 底层发生了什么？

你打开一份名为"可靠传输"的空白设计文档，面对四个噩梦。

---

**噩梦一：丢包。** 你发的包可能在路由器队列里被丢弃、光纤断了永远到不了、校验出错被接收方扔了。

你怎么知道丢了？让接收方每收到一个包就回 **ACK（确认）**。你等一段时间没收到 ACK，就**重发**。这就是 **超时重传**。

但等多久？太短——疯狂重发，网络更堵。太长——丢了半天才重传。TCP 的做法：**动态计算 RTT（往返时间）**，每次收到 ACK 时更新超时计时器。不是固定值，是随网络实时变化的。

---

**噩梦二：乱序。** 你发了包 1、2、3。网络路径不同，接收方可能先收到 3、再收到 1。

TCP 的解法：**每个包带一个序列号（Sequence Number）**。接收方按序列号重排，不管到达顺序如何，交给上层的数据永远正确。就像给每页文件标页码——快递员可能先送第 3 页，你按页码排好再读。

---

**噩梦三：发送方太快。** 接收方处理不过来，缓冲区满了，新包来了直接丢。然后又要重传……恶性循环。

TCP 解：**流量控制。** 接收方在 ACK 里夹带**窗口大小**——"我还能收多少字节"。发送方永远不超过这个窗口。

---

**噩梦四：网络本身拥塞。** 不是接收方慢，是中间的路由器队列满了——全网流量太大。

TCP 解：**拥塞控制。** 发送方维护**拥塞窗口（cwnd）**。初始极小（慢启动），逐渐增大。一旦检测到丢包，cwnd 立刻减半。这就是 TCP 的哲学：**不知道路况，先开慢点，路好加速，撞车减速。**

### 大白话解释

TCP 像一个极其负责任的快递公司。每页纸单独装信封（**分片**），写页码（**序列号**）。朋友收到发短信确认（**ACK**）。没确认的重新寄（**超时重传**）。收到第 9 页还没第 8 页？先放一边等（**乱序重排**）。朋友说信箱满了先别寄（**流量控制**）。邮局爆仓了？公司自己判断——少寄点（**拥塞控制**）。

### 核心启示

> TCP 不是"一次设计了完美协议"——它是遇到一个坑填一个坑，在不可靠的 IP 之上，通过 ACK + 序列号 + 窗口 + 拥塞控制四件套，硬生生"模拟"出可靠通道。**任何可靠系统都是在不可靠基础上层层加保证。**

## Q17 · 浏览器输入baidu.com回车——网络层面经历了什么？

### 你有没有思考过这么一个问题？

每天敲几十次 URL，页面在几百毫秒加载。但如果面试官让你**"从输入 URL 到页面加载，说出完整网络链路"**——你能从 DNS 讲到 TCP 握手、TLS 讲到 HTTP 吗？

### 底层发生了什么？

**第一步：DNS 解析。** `baidu.com` → IP 地址。

浏览器 DNS 缓存 → OS hosts 文件 → 系统缓存 → 本地 DNS 服务器 → 递归查询：根 DNS → `.com` 顶级域 → baidu.com 权威 DNS → 拿到 IP。

**第二步：TCP 三次握手。** 拿到 IP，建立连接。

```
客户端 SYN  →  服务器
客户端     ←  服务器 SYN-ACK
客户端 ACK →  服务器
```

三次不是巧合——两次不够（旧 SYN 延迟到达会混淆），四次多余。**三次就是确认双方都能收发的最小次数。**

**第三步：TLS 加密握手。** `https://` 不是 `http://`。

ClientHello（加密套件 + 随机数）→ ServerHello + 证书 → 密钥交换（非对称加密传对称密钥）→ 后续全部用对称加密（AES）通信。非对称加密比对称慢几千倍，所以**只用来传密钥，数据用对称加密。**

**第四步：HTTP 请求/响应。** 安全通道建立，浏览器发 `GET / HTTP/1.1`，服务器回 HTML。

**第五步：浏览器渲染。** HTML → DOM 树 → CSS → 渲染树 → Layout → Paint → 页面呈现。

### 大白话解释

去高级俱乐部找"张三"。前台查登记簿（**DNS**）→ 告诉"301"。走到门口确认彼此能听到（**TCP 握手**）。对暗号证明身份（**TLS 握手**）。"张先生，资料呢？"（**HTTP 请求**）。递给你文件夹（**HTTP 响应**）。打开阅读（**浏览器渲染**）。

### 核心启示

> 一个 URL 的生命周期展示了网络核心哲学：**加一层，解决一个问题。** DNS 解决名字到地址、TCP 解决可靠传输、TLS 解决加密、HTTP 解决超文本、浏览器解决渲染。**永远不要在一个层里解决所有问题。**

## Q18 · TCP vs UDP——不是"可靠vs不可靠"那么简单

### 你有没有思考过这么一个问题？

"TCP 可靠，UDP 不可靠"——这句话每个程序员都会背。但**实时音视频为什么选 UDP？游戏为什么有时 TCP 有时 UDP？QUIC（HTTP/3）基于 UDP 做了什么事让它"可靠"了？**

选择不是"可靠就好"——是**你的场景能容忍什么代价。**

### 底层发生了什么？

TCP 的可靠性是有代价的：**队头阻塞。** TCP 是字节流——数据必须按序交付。如果包 3 丢了，包 4、5、6 已经到了，但**必须等包 3 重传成功后才能一起交给上层。**

实时音视频通话：你宁愿偶尔丢几个音频帧（听不清 0.1 秒），也绝不能等——**等你重传完，通话已经过去了。**

UDP 不保证顺序、不保证送达、不保证不重复。**它几乎什么都不保证**——但正因此它没有队头阻塞。应用层可以自己决定：哪些帧可以丢、哪些必须重传、允许乱序到什么程度。

**QUIC（HTTP/3）选择基于 UDP 重新实现可靠性。** 它把 TCP 的可靠传输搬到用户态，但做了关键改进：**每个 Stream 独立的可靠传输**——Stream A 丢包不影响 Stream B。而 TCP 的丢包阻塞同一连接上的所有 Stream。这就是为什么 HTTP/3 比 HTTP/2 快。

### 大白话解释

TCP = 寄挂号信。每封要签收，丢了补寄，严格按顺序。安全但慢。UDP = 寄明信片。扔进邮筒不管了，可能丢、可能乱。快但你自己负责。QUIC = 你雇了个专业快递员——他跑得和明信片一样快，但每件单独追踪，A 件丢了 B 件照送。

### 核心启示

> "可靠性"不是免费的。TCP 为它付出的代价是队头阻塞和延迟。**选 TCP 还是 UDP，不是在"可靠"和"不可靠"之间选——是在"让协议栈替你兜底"和"你自己在应用层精准控制"之间选。** QUIC 走了一条中间路：用 UDP 的速度 + 在用户态重建 TCP 的可靠性，但修掉了队头阻塞。

## Q19 · DNS是一个全球分布式目录——一个域名解析请求怎么在几百毫秒内完成？

### 你有没有思考过这么一个问题？

你在浏览器输入 `github.com`，几百毫秒后页面出来了。但在这几百毫秒里，你的电脑跟**全球至少 3 台 DNS 服务器**进行了对话。这些服务器分布在全球各地，每台只知道自己该知道的那一小部分信息。它们是怎么协作的？

### 底层发生了什么？

DNS 是一个**分层、分布式的树形目录**。

```
根 DNS（全球 13 组）→ 顶级域 DNS（.com/.org/.cn）→ 权威 DNS（github.com）
```

你查询 `www.github.com` 的 IP 时：

1. 你的递归 DNS（运营商分配或 8.8.8.8）收到查询
2. 递归 DNS 问根 DNS："`.com` 的 DNS 在哪？"
3. 根 DNS 答："去问这些服务器"（`.com` 顶级域的 NS 记录）
4. 递归 DNS 问 `.com` 顶级域："`github.com` 的 DNS 在哪？"
5. 顶级域答："去问这些服务器"（github 的权威 DNS）
6. 递归 DNS 问 github 权威 DNS："`www.github.com` 的 IP？"
7. 拿到答案，缓存起来，返回给你

**为什么这么快？** 缓存无处不在。你的浏览器缓存、操作系统缓存、递归 DNS 缓存——绝大多数查询在第一步就命中了，根本不需要走完整递归链。只有"冷"域名才走全程。

### 大白话解释

DNS = 一个全球电话簿。你问本地接线员"张三电话？"接线员不知道，打电话给省局、省局不知道打给总局、总局说"张三在 B 市分局"——逐级问下来，最后打通张三本人。本地的接线员记性好（**缓存**），张三的电话第二次打来秒答。

### 核心启示

> DNS 的分层设计是分布式系统的经典范本：**每层只知道自己该知道的信息，上层不需要知道下层细节。** 根 DNS 不知道 github 的 IP，也不需要知道——它只需要知道 `.com` 的 DNS 在哪。**分层 = 限制知识的传播范围 = 降低每个节点的认知负担 = 系统可以无限扩展。**

## Q20 · HTTP/1.1 → HTTP/2 → HTTP/3——每一代在解决什么？

### 你有没有思考过这么一个问题？

"HTTP/2 比 1.1 快，HTTP/3 比 2 快"——为什么？不是"代码优化了"，而是**每一代都在修上一代的队头阻塞问题**。这是一个"搬来搬去"的故事。

### 底层发生了什么？

**HTTP/1.1 的队头阻塞：** 一个 TCP 连接同时只能处理一个请求-响应。浏览器开了 6 个并行连接来缓解，但每个连接内部仍有队头阻塞——第一个请求慢，后面的全等。

**HTTP/2 的解法：多路复用。** 一个 TCP 连接上同时跑多个 Stream，不再需要 6 个连接。队头阻塞从"应用层（请求排队）"被消除了——**但转移到了 TCP 层。** 如果 TCP 丢了一个包，该连接上**所有 Stream** 都阻塞，等这个包重传成功。

**HTTP/3 的解法：换传输层。** 不再用 TCP，改用 QUIC（基于 UDP）。QUIC 每个 Stream 独立传输——Stream A 丢包，Stream B、C 照常跑。**队头阻塞被彻底消灭。**

代价：UDP 穿透 NAT 更难，QUIC 加密头不允许中间件做优化。

### 大白话解释

HTTP/1.1 = 一条单车道。HTTP/2 = 同一条路上画了多条虚拟车道（多路复用），但路基底层（TCP）一堵全堵。HTTP/3 = 每条虚拟车道是独立公路（QUIC Stream），一条堵别的照走。

### 核心启示

> 队头阻塞是网络协议设计中最顽固的难题之一。HTTP 三代演进证明：**复杂度不会消失，只会转移。** HTTP/2 消除了应用层队头阻塞，转移到了 TCP 层。HTTP/3 用 UDP 重建传输层，把问题在更底层解决。**选协议 = 选你把复杂度放在哪一层。**

## Q21 · HTTPS的锁图标——浏览器怎么验证这个网站是真的？

### 你有没有思考过这么一个问题？

地址栏那个小锁图标——点击显示"连接是安全的"。但"安全"到底指什么？如果有人伪造了一个和百度一模一样的网站，你的浏览器怎么识破？

### 底层发生了什么？

HTTPS 的信任基于**证书链（Certificate Chain）**。

你的浏览器内置了一个"信任列表"——**根 CA 证书**。全球只有几十个根 CA，它们的公钥预装在操作系统和浏览器里。

当你访问 `baidu.com` 时，服务器发来**三个证书**：

```
根 CA 证书（浏览器已有）→ 中间 CA 证书 → baidu.com 证书
```

浏览器验证链：中间 CA 的签名用根 CA 公钥验证 → baidu.com 的签名用中间 CA 公钥验证 → 链完整 → 域名匹配 → 证书未过期 → 信任。

**如果中间 CA 被攻破（如 2011 年 DigiNotar 事件），它签发的所有证书全部不可信。** 浏览器厂商直接把它从信任链里踢出去。Certificate Transparency（CT）通过公开日志防止 CA 偷偷签发恶意证书。

### 大白话解释

HTTPS 证书链 = 身份证体系。你的身份证由公安局签发——你不信我，但信公安局。公安局由国务院授权——你不信"授权书"本身，但国务院公信力是全社会默认的。**信任从根向下逐级"签名"传递。**

### 核心启示

> HTTPS 的安全不基于"加密算法不可破解"，而基于**信任链的起点——根 CA 的公钥——是你选择信任的。** 如果根 CA 被攻破或被污染，整个信任链崩塌。**信任永远是一个社会性选择，不是纯技术问题。**

## Q22 · WebSocket vs HTTP长轮询 vs SSE——实时消息推送怎么选？

### 你有没有思考过这么一个问题？

你要做一个实时聊天系统。服务端有新消息了，需要"推"给客户端。怎么推？

方案一：客户端每秒问一次"有新消息吗？"（短轮询）——90% 的请求返回空。方案二：客户端发请求，服务端 hold 住不返回，等到有消息才返回（长轮询）。方案三：建立 WebSocket 连接，服务端随时主动推。方案四：用 SSE（Server-Sent Events）——服务端能推，但客户端不能通过同一通道发。

**四个方案，性能差异巨大。什么时候选哪个？**

### 底层发生了什么？

- **短轮询**：HTTP 请求-响应，客户端定时发起。浪费带宽和连接——但最简单，兼容所有代理。
- **长轮询**：HTTP 请求，服务端不立即返回，直到有数据或超时。比短轮询好，但仍有轮询间隙。
- **WebSocket**：先 HTTP Upgrade，之后 TCP 连接升级为**全双工**——服务端随时推，客户端随时发。帧开销极小（2 字节起）。适合双向高频场景（聊天、游戏、行情）。
- **SSE**：服务端单向推送。客户端不能通过同一通道发数据。适合单向通知（新闻推送、状态更新）。

**选型关键：** 双向高频 → WebSocket。单向推送 → SSE。低频简单 → 长轮询。WebSocket 在企业代理/防火墙下容易被阻断——SSE（基于 HTTP）穿透性更好。

### 大白话解释

- 短轮询 = 你每隔 5 秒问"饭好了吗？"
- 长轮询 = 你跟老板说"饭好了叫我"，老板 hold 住，好了才回答
- WebSocket = 老板给你对讲机——随时能喊你，你也能随时喊他
- SSE = 老板有广播喇叭——只能他喊你

### 核心启示

> "实时"不是一种技术，是**一种延迟容忍度。** 延迟 < 100ms 且双向 → WebSocket。延迟 < 1 秒且单向 → SSE。延迟 > 1 秒 → 长轮询。**选型永远取决于场景的延迟预算，而不是技术的"高级程度"。**
